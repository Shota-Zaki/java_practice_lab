エラーメッセージの読み方

コンパイルエラー・例外メッセージの見る場所を整理します。

記事本文上部
学習記事一覧型・配列文字列演算・制御クラス設計継承例外メソッド表

最初に固定する読み方

結論を急がない。 まずコンパイルできるか、次に実行時に止まるか、最後に出力を追います。出力だけを先に追うと、型・アクセス・例外で簡単に崩れます。
  1. 宣言を見る。 型、修飾子、継承関係、メソッドシグネチャを先に確定します。
  2. 代入できるかを見る。 右辺の型・値・参照先が左辺に入るかを判断します。
  3. 呼び出せるかを見る。 変数の宣言型から、その場で見えるメンバだけを呼べます。
  4. 実体で動くものを分ける。 オーバーライドされたインスタンスメソッドは実体、フィールドやstaticは別規則です。
  5. 副作用を表にする。 代入、インクリメント、可変オブジェクト、例外で状態が変わる箇所を書き出します。

最初に固定する分類

分類発生タイミング選択肢での扱い判断の軸
コンパイルエラー実行前出力も例外も発生しない文法、型、アクセス、override、未処理チェック例外
実行時例外実行中発生位置までの出力は残るnull、範囲外、キャスト、数値変換、0除算
正常終了最後まで実行出力結果を追う評価順、参照、メソッド呼び出し、例外処理
出力を追う前にコンパイル可否。
コンパイルエラーのコードに対して出力を追っても無意味です。試験本番では、まず「宣言・型・アクセス・例外処理・override」を高速に確認してください。

コンパイルエラー：型・代入・初期化

原因典型コードなぜエラーか問題での見抜き方
ローカル変数未初期化int a; System.out.println(a);ローカル変数にはデフォルト値がない。フィールドなら初期値あり、ローカルなら確定代入が必要。
縮小変換byte b = 128;byte の範囲を超える。byte は -128〜127。
小数リテラルfloat f = 1.0;1.0 は double。float にするなら 1.0F。
longからintint x = 10L;long を int に暗黙代入できない。L 接尾辞を見落とさない。
byte/short演算byte c = a + b;byte 同士でも演算結果は int。変数同士の演算は定数式ではない。
booleanと数値if (1) {}Javaでは条件式は boolean が必要。C言語系の感覚で読まない。
nullを基本型へ代入int n = null;null は参照型用。ラッパー型なら代入自体は可能。
不正な識別子int 1a = 10;識別子は数字で始められない。_ 単体や予約語にも注意。

コンパイルエラー：var

原因典型コード理由判断
初期化なしvar a;推論材料がない。var は宣言時初期化が必須。
null単独var a = null;型を決められない。キャストがあれば推論可能な場合あり。
フィールドprivate var value = 10;var はローカル変数用。フィールド・引数・戻り値には使えない。
メソッド引数void test(var x) {}引数型には使えない。ラムダ引数の特殊ケースと混同しない。
配列初期化子var a = {1, 2};配列型を直接推論できない。var a = new int[]{1,2}; は可能。
複数宣言var a = 1, b = 2;var では複数変数宣言不可。1文1変数で見る。

コンパイルエラー：クラス・メソッド・コンストラクタ

原因典型コード理由見抜き方
publicクラス名とファイル名不一致public class A を B.java に保存publicトップレベルクラス名とファイル名は一致が必要。ファイル名が設問に書かれていたら必ず確認。
戻り値ありコンストラクタ風メソッドvoid Sample() {}これはコンストラクタではなくメソッド。クラス名と同じでも戻り値型があればメソッド。
暗黙super()失敗親に無引数コンストラクタがない子のコンストラクタ先頭で暗黙に super() が呼ばれる。親コンストラクタを確認。
this()/super()の位置不正処理後に this()コンストラクタ呼び出しは先頭文のみ。1行目以外ならエラー。
staticからインスタンスメンバ参照static void m(){ System.out.println(value); }インスタンスなしでインスタンスメンバは使えない。main 内から直接フィールドを触っていないか見る。
戻り値不足int m(){ if(x) return 1; }全経路で戻り値が必要。条件分岐の片側に return がない。
到達不能文return; System.out.println();絶対に実行されない文はエラー。return、throw、無限ループ後を見る。

コンパイルエラー：継承・override・interface

原因典型コード理由判断
overrideでアクセスを狭くするpublic → protectedオーバーライドでは同じか広いアクセスが必要。親子メソッドのアクセス修飾子を比較。
戻り値型が不正親より広い戻り値型戻り値は同じかサブクラス型のみ。共変戻り値かどうか見る。
checked例外を広げる親が IOException、子が Exceptionoverrideでチェック例外を広げられない。RuntimeException系は別扱い。
finalメソッドをoverridefinal void m() を子で再定義final メソッドはoverride不可。親メソッドの final を確認。
privateメソッドをoverride扱い親 private void m()、子 @Override void m()private は継承されないためoverrideではない。@Override があるとエラー。
abstractメソッド未実装具象クラスがinterfaceメソッド未実装抽象メソッドの実装義務が残る。クラスが abstract なら未実装でもよい場合あり。
defaultメソッド衝突複数interfaceが同じdefaultを持つクラス側で解決しないと衝突。同名同シグネチャのdefaultを探す。
Objectメソッドをdefault定義interfaceで default String toString()Object のpublicメソッド相当をdefaultで提供できない。toString、equals、hashCode に注意。

コンパイルエラー：アクセス制御・修飾子

原因典型理由判断
privateメンバへの外部アクセスobj.valueprivate は同じクラス内のみ。継承していても直接アクセス不可。
defaultアクセスの別パッケージ利用無修飾クラスを別パッケージから使うdefaultは同一パッケージのみ。パッケージ宣言を見る。
protectedの誤用別パッケージで親型変数からprotectedアクセス別パッケージではサブクラス経由の制約がある。「同じパッケージか」「サブクラス内か」を分ける。
トップレベルprivate/protectedprivate class Aトップレベルクラスに使えない。トップレベルか内部クラスかを確認。
sealedのpermits不整合許可されていないクラスが継承permits に列挙されていない。継承先の名前を照合。
sealed子クラスの修飾漏れclass B extends Asealedを継承した直接サブクラスは final / sealed / non-sealed が必要。直接の子クラスを確認。

コンパイルエラー：例外処理

原因典型コード理由見抜き方
チェック例外未処理throw new IOException();catchするかthrows宣言が必要。Exception 直下や IOException を見る。
catch順序不正catch(Exception) の後に catch(IOException)後続catchが到達不能。親例外を先にcatchしていないか。
multi-catchの親子関係catch(Exception | IOException e)同じmulti-catch内に親子関係は不可。| の左右の継承関係を見る。
try構文不正try {} だけcatch または finally が必要。tryの後続ブロックを確認。
try-with-resources対象外AutoCloseableでない型をresourceに書くリソースは AutoCloseable が必要。close() だけでなく型が実装しているか。

実行時例外：null・範囲外・キャスト

例外典型コード原因問題での注意
NullPointerExceptions.length() where s == nullnull参照でメソッド/フィールドアクセス。System.out.println(s) だけなら null 表示で例外ではない。
ArrayIndexOutOfBoundsExceptionarray[array.length]配列範囲外。最後の有効添字は length - 1。
StringIndexOutOfBoundsExceptionstr.charAt(str.length())文字列範囲外。substring(length) は空文字で可能、charAt(length) は不可。
IndexOutOfBoundsExceptionlist.get(list.size())List範囲外。配列とは例外クラス名が違う。
ClassCastExceptionString s = (String) obj;実体が変換先型ではない。コンパイル可能でも実体が違えば落ちる。
ArrayStoreExceptionObject[] a = new String[1]; a[0] = new Object();実体配列の要素型に合わない。配列の共変性で出る。
ArithmeticException10 / 0整数の0除算。小数の0除算とは挙動が違う。
NumberFormatExceptionInteger.parseInt("A")数値へ変換できない。起動引数の変換問題で頻出。
UnsupportedOperationExceptionList.of("A").add("B")変更不可リストへの変更操作。List.of や固定サイズリストに注意。

実行時例外：catchされる場合・されない場合

例外が発生しても、対応する catch があれば処理は継続します。対応する catch がなければ、そこで異常終了します。finally がある場合は、異常終了前にも実行されます。

try {
    System.out.println("A");
    String s = null;
    s.length();
    System.out.println("B");
} catch (NullPointerException e) {
    System.out.println("C");
} finally {
    System.out.println("D");
}
System.out.println("E");

この場合、出力は A C D E です。B は例外発生後なので実行されません。catchで処理できたため、finally後に後続処理へ進みます。

チェック例外と非チェック例外の見分け

分類代表例コンパイラの処理要求覚え方
チェック例外Exception, IOExceptioncatch または throws が必要。RuntimeException 系ではない Exception。
非チェック例外RuntimeException, NullPointerException, ClassCastException, NumberFormatException処理しなくてもコンパイル可能。実行時のプログラムミス系。
ErrorOutOfMemoryError など通常処理対象にしない。問題では例外階層として区別。
RuntimeExceptionかどうかを見る。
すべてを丸暗記するより、頻出の非チェック例外を先に押さえる方が効率的です。NullPointerException、ArrayIndexOutOfBoundsException、ClassCastException、NumberFormatException、ArithmeticException は非チェック例外として即判定できるようにしてください。

エラー判定の実戦フロー

  1. 宣言を見る。 ファイル名、package/import、クラス修飾子、mainメソッド、変数宣言を確認する。
  2. 型を見る。 代入、戻り値、引数、演算結果、キャストの可否を確認する。
  3. アクセスを見る。 private/default/protected/public、staticからの参照、継承関係を見る。
  4. override/interfaceを見る。 アクセス、戻り値、throws、abstract未実装、default衝突を確認する。
  5. チェック例外を見る。 IOExceptionなどがcatch/throwsされているか確認する。
  6. ここまで通ったら実行時へ進む。 null、範囲外、キャスト、0除算、数値変換を左から追う。
  7. 例外処理を見る。 catchされるか、finallyが実行されるか、後続処理へ進むかを判断する。

ミニ演習

演習1：コンパイルか実行時か

String s = null;
System.out.println(s.length());

答え: コンパイルは可能。実行時に NullPointerException。

演習2：ローカル変数

int x;
if (args.length > 0) {
    x = 10;
}
System.out.println(x);

答え: コンパイルエラー。if に入らない経路では x が未初期化。

演習3：catch順序

try {
    throw new java.io.IOException();
} catch (Exception e) {
} catch (java.io.IOException e) {
}

答え: コンパイルエラー。先に Exception で捕まえるため、後続の IOException は到達不能。

最後に見るチェックリスト

  • 出力を追う前にコンパイルエラー候補を潰したか。
  • ローカル変数とフィールドの初期値ルールを混同していないか。
  • var をフィールド・引数・戻り値に使っていないか。
  • 小さい整数型の演算結果を int として見ているか。
  • overrideでアクセス・戻り値・throwsを確認したか。
  • チェック例外をcatchまたはthrowsしているか。
  • nullのメソッド呼び出し、範囲外、キャスト失敗を左から追ったか。
  • 例外発生後、tryブロックの残りを実行していないか。
  • catchされた後にfinallyと後続処理へ進むかを見たか。

最後に自分で確認すること

この記事を読み終えたら、次の3点を口頭で説明できるか確認します。説明できない場合は、まだ「読んだだけ」で、コードに適用できる状態ではありません。

  • この単元で最初に見るべき場所はどこか。
  • コンパイルエラー、実行時例外、正常出力をどの順番で分けるか。
  • 自分が間違えやすい引っかけは、型・参照・戻り値・アクセス・例外のどれか。
