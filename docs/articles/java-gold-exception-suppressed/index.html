try-with-resourcesと抑制された例外

close時の例外と本体例外の関係を整理します。

本体とcloseの両方で例外が起こる場合

tryブロック内で例外が発生し、さらにcloseでも例外が発生することがあります。

本体の例外が主になる

通常、tryブロック内の例外が主例外となり、close時の例外はsuppressedとして保持されます。

Goldでは詳細理解が必要

Silverでは閉じる順序が重要ですが、Goldでは抑制例外も理解しておくと強いです。

例

catch (Exception e) {
    for (Throwable t : e.getSuppressed()) {
        System.out.println(t);
    }
}

試験・学習での注意

  • 実行結果を見る前に、まずコンパイルできる形か確認します。
  • 似た構文でも、型・戻り値・アクセス修飾子で結果が変わります。
  • 暗記だけでなく、なぜその結果になるかを言語化してください。

演習前チェック

この記事を演習で使える知識に変える

対象レベル：Gold。Goldでは、構文暗記よりもAPIの戻り値型、遅延実行、型推論、例外、リソース管理まで含めて判定します。Silverレベルの文法確認で止まると、ここでは足りません。

本文で必ず押さえること

  • まずコンパイル時点で必要な処理が書かれているか見る。checked例外は特にここで落ちる。
  • try内で例外が出た行以降は実行されない。catchへ制御が移る。
  • finallyとtry-with-resourcesの実行順序を分ける。

この後の演習で見るポイント

  • catchの順番が親子関係として正しいか確認する。
  • 例外が発生する行を特定し、その後の行が実行されるかを判断する。
  • 非チェック例外でも、catchされるかされないかで終了の仕方が変わる。

ここで落ちる典型ミス

  • Exceptionを先にcatchしてから、その下に細かい例外を書く。これは到達不能になる。
  • finallyがあるから例外が消えると誤解する。finallyは原則として実行されるだけ。
  • コンパイルエラーと実行時例外を同じ「エラー」として処理する。

演習前の小課題

try内で3行目に例外が出るコードを想定し、4行目が実行されるか、catch後にどこへ進むかを言葉で説明してください。
