並行処理 / Executor / Future完全整理

Goldの並行処理は、スレッドを直接作る話ではなく「タスクをどこに投入し、いつ結果を取り、どう終了させるか」を読む問題です。ExecutorService、Callable、Future、shutdownを分けて判定します。

最初に押さえる判定順

並行処理の問題は、出力結果だけを追うと崩れます。先に以下の順で事実を確定します。

  1. タスクはRunnableかCallableか。
  2. 投入方法はexecuteかsubmitか。
  3. 戻り値としてFutureを受け取っているか。
  4. get()で待機しているか。
  5. shutdown()後に新しいタスクを投入していないか。
  6. 出力順を保証する同期・待機・単一スレッド実行があるか。

Threadを直接扱うのではなく、タスクをExecutorに渡す

Threadは実行単位そのものです。一方、ExecutorServiceはタスクを受け取り、内部のスレッドで実行する仕組みです。Goldでは、スレッド生成よりもExecutorにタスクを投げた後の戻り値・例外・終了処理が狙われます。

import java.util.concurrent.*;

public class Main {
  public static void main(String[] args) throws Exception {
    ExecutorService ex = Executors.newSingleThreadExecutor();
    Future<Integer> f = ex.submit(() -> 10 + 20);
    System.out.println(f.get()); // 30。get()は結果が出るまで待つ
    ex.shutdown();
  }
}
試験では「submitした」だけでは結果を取得していません。結果を使うにはFuture.get()が必要です。

RunnableとCallableの違い

Runnableは戻り値なし、Callable<V>は戻り値ありです。さらに、Callableのcall()はchecked例外を投げられます。

項目RunnableCallable<V>
抽象メソッドvoid run()V call() throws Exception
戻り値なしあり
checked例外そのまま投げられない投げられる
主な投入先execute / submitsubmit / invokeAll / invokeAny
Runnable r = () -> System.out.println("run");
Callable<String> c = () -> "call";

// NG: Runnableは値を返せない
// Runnable bad = () -> "x";

// OK: Callableは値を返せる
Callable<String> ok = () -> "x";

executeとsubmitの違い

executeはRunnableだけを受け取り、戻り値はありません。submitはRunnableもCallableも受け取れ、Futureを返します。

操作受け取れるタスク戻り値試験での注意
execute(runnable)Runnableなし結果取得できない。
submit(callable)Callable<V>Future<V>get()で戻り値を取得。
submit(runnable)RunnableFuture<?>get()の結果は通常null。
submit(runnable, result)RunnableFuture<T>完了時に指定したresultを返す。
ExecutorService ex = Executors.newSingleThreadExecutor();

Future<String> a = ex.submit(() -> "A");
System.out.println(a.get()); // A

Future<?> b = ex.submit(() -> System.out.print("B"));
System.out.println(b.get()); // null

Future<String> c = ex.submit(() -> System.out.print("C"), "done");
System.out.println(c.get()); // done

ex.shutdown();

Futureは「完了するかもしれない結果」

Futureは、タスクの実行結果を後から受け取るための入れ物です。タスクがまだ終わっていなければ、get()は待機します。

メソッド意味注意点
get()完了まで待って結果を取得InterruptedException、ExecutionExceptionを処理する必要がある。
get(timeout, unit)時間制限つきで待つ時間切れはTimeoutException。
isDone()完了済みか確認結果を取り出すわけではない。
cancel(boolean)キャンセルを試みる必ず停止できる保証はない。
isCancelled()キャンセル済みか確認正常完了とは別。
ExecutorService ex = Executors.newSingleThreadExecutor();
Future<String> f = ex.submit(() -> {
  Thread.sleep(100);
  return "done";
});

System.out.println(f.isDone()); // falseの可能性がある
System.out.println(f.get());    // done。ここで完了まで待つ
System.out.println(f.isDone()); // true
ex.shutdown();

getはchecked例外を要求する

Future.get()はchecked例外を投げる可能性があります。したがって、throws Exceptionなどで逃がすか、try-catchで処理します。

// NG: get() の checked例外を処理していない
// public static void main(String[] args) {
//   Future<Integer> f = ex.submit(() -> 1);
//   System.out.println(f.get());
// }

// OK
public static void main(String[] args) throws Exception {
  ExecutorService ex = Executors.newSingleThreadExecutor();
  Future<Integer> f = ex.submit(() -> 1);
  System.out.println(f.get());
  ex.shutdown();
}

タスク内の例外はExecutionExceptionに包まれる

CallableやsubmitされたRunnableの中で例外が発生した場合、Future.get()時にExecutionExceptionとして受け取ります。元の例外はgetCause()で確認します。

ExecutorService ex = Executors.newSingleThreadExecutor();
Future<Integer> f = ex.submit(() -> 10 / 0);

try {
  System.out.println(f.get());
} catch (ExecutionException e) {
  System.out.println(e.getCause().getClass().getSimpleName());
  // ArithmeticException
}
ex.shutdown();
ここを「ArithmeticExceptionが直接catchされる」と読むと誤りです。get()が投げるのはExecutionExceptionです。

ExecutorServiceの作り方

Executorsには複数のファクトリメソッドがあります。試験では、スレッド数が出力順に影響するかを見るのが重要です。

生成メソッド特徴出力順の注意
newSingleThreadExecutor()1本のワーカースレッドで順に実行投入順に処理されやすい。ただしmainスレッドの出力とは別。
newFixedThreadPool(n)固定数のスレッドで実行複数タスクは並行実行され、順序は固定しない。
newCachedThreadPool()必要に応じてスレッドを再利用・作成順序を決めつけない。
newScheduledThreadPool(n)遅延実行・周期実行時間指定と終了処理を見る。
ExecutorService ex = Executors.newFixedThreadPool(2);
ex.submit(() -> System.out.print("A"));
ex.submit(() -> System.out.print("B"));
ex.shutdown();
// ABとは限らない。BAもあり得る。

出力順が保証されるケース・されないケース

並行処理の出力問題では、「たまたまそうなりそう」ではなく、保証があるかを見ます。

コード上の条件出力順理由
同一スレッド内の連続文保証される同じスレッド内では順に実行される。
Future.get()で待ってから次に進む待機前後の順序は保証されるget()が完了までブロックする。
newSingleThreadExecutor()に同順で投入タスク同士は順に実行される単一ワーカーがキューを処理する。
newFixedThreadPool(2)以上原則保証されない複数スレッドで実行される。
shutdown()だけ呼ぶmainが待つ保証はない新規受付停止であり、待機処理にならない。
ExecutorService ex = Executors.newFixedThreadPool(2);
Future<String> f = ex.submit(() -> "A");
System.out.print(f.get()); // ここでAの完了を待つ
System.out.print("B");    // 必ずAの後
ex.shutdown();

shutdownは「新規受付停止」

shutdown()はExecutorServiceに「もう新しいタスクを受け付けない」と伝える操作です。投入済みタスクを即座に止める処理ではありません。

メソッド意味注意
shutdown()新規タスク受付を停止投入済みタスクは基本的に実行される。
shutdownNow()実行中タスクの停止を試み、未開始タスクを返す即停止は保証されない。
isShutdown()shutdown開始済みか全タスク完了を意味しない。
isTerminated()全タスク完了済みかshutdown後、全タスク終了でtrue。
awaitTermination終了まで一定時間待つ戻り値は完了したかどうかのboolean。
ExecutorService ex = Executors.newSingleThreadExecutor();
ex.submit(() -> System.out.println("A"));
ex.shutdown();

// NG: shutdown後に新規タスクを投入
// ex.submit(() -> System.out.println("B")); // RejectedExecutionException
shutdown()を忘れると、非デーモンスレッドが残り、プログラムが終了しないことがあります。試験では終了処理の有無も見る必要があります。

awaitTerminationで終了を待つ

Executorの終了を待ちたい場合は、shutdown()の後にawaitTerminationを使います。戻り値は、指定時間内に終了したかどうかです。

ExecutorService ex = Executors.newFixedThreadPool(2);
ex.submit(() -> System.out.println("task"));
ex.shutdown();

if (!ex.awaitTermination(1, TimeUnit.SECONDS)) {
  ex.shutdownNow();
}

cancelは必ず止める命令ではない

Future.cancel(true)は、タスクをキャンセルしようとする操作です。trueは、実行中なら割り込みを試みるという意味です。タスクが割り込みに反応しなければ、即停止しない可能性があります。

Future<?> f = ex.submit(() -> {
  while (!Thread.currentThread().isInterrupted()) {
    // 作業
  }
});

boolean requested = f.cancel(true);
System.out.println(requested);
状態cancelの結果注意
まだ開始前キャンセルされやすい実行されない。
実行中割り込みを試みるタスクが割り込みを無視すれば止まらない可能性。
完了済み通常キャンセルできないfalseになる。

invokeAllとinvokeAny

invokeAllは複数のCallableをまとめて実行し、すべて完了するまで待ちます。invokeAnyは、正常に完了したどれか1つの結果を返します。

メソッド意味戻り値
invokeAll(tasks)全部の完了を待つList<Future<T>>
invokeAny(tasks)どれか1つの正常完了を待つT
ExecutorService ex = Executors.newFixedThreadPool(2);
List<Callable<String>> tasks = List.of(
  () -> "A",
  () -> "B"
);

List<Future<String>> results = ex.invokeAll(tasks);
for (Future<String> f : results) {
  System.out.println(f.get());
}

String one = ex.invokeAny(tasks);
System.out.println(one); // AかBのどちらか
ex.shutdown();

ScheduledExecutorService

ScheduledExecutorServiceは、遅延実行や周期実行を扱います。戻り値はScheduledFutureです。

メソッド意味注意
schedule指定時間後に1回実行戻り値をget()すると完了まで待つ。
scheduleAtFixedRate一定周期で実行開始前回の開始時刻基準。
scheduleWithFixedDelay前回終了後、一定遅延して次回実行終了時刻基準。
ScheduledExecutorService ex = Executors.newSingleThreadScheduledExecutor();
ScheduledFuture<String> f = ex.schedule(() -> "done", 1, TimeUnit.SECONDS);
System.out.println(f.get());
ex.shutdown();

共有データの更新は競合する

複数スレッドで同じ変数を更新すると、競合が起こります。count++は1命令ではなく、読み取り、加算、書き戻しに分かれます。

class Counter {
  private int count;
  void increment() { count++; } // 複数スレッドでは安全ではない
  int get() { return count; }
}

対策として、synchronized、Lock、AtomicInteger、並行コレクションなどを使います。

AtomicInteger count = new AtomicInteger();
count.incrementAndGet();

synchronizedの基本

synchronizedは、同じモニタを使うスレッドを同時に入れないようにします。インスタンスメソッドのsynchronizedは、そのインスタンスをロックします。staticメソッドの場合はClassオブジェクトをロックします。

書き方ロック対象注意
synchronized void m()そのインスタンス別インスタンスなら別ロック。
static synchronized void m()Classオブジェクト全インスタンスで共有される。
synchronized(obj) { ... }objobjがnullならNullPointerException。
class SafeCounter {
  private int count;
  synchronized void increment() { count++; }
  synchronized int get() { return count; }
}

ConcurrentHashMapと同期化コレクション

通常のHashMapやArrayListは、複数スレッドからの同時更新に安全ではありません。並行処理用にはConcurrentHashMapなどを使います。

ConcurrentMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("A", 1);
map.compute("A", (k, v) -> v == null ? 1 : v + 1);

Collections.synchronizedList(new ArrayList<>())のような同期化ラッパーもあります。ただし、反復処理まで常に安全になると雑に覚えるのは危険です。

parallelStreamは順序と副作用に注意

parallelStream()は、内部で並列実行される可能性があります。出力順や共有変数の更新を前提にしたコードは危険です。

List.of("A", "B", "C").parallelStream()
    .forEach(System.out::print); // ABCとは限らない

List.of("A", "B", "C").parallelStream()
    .forEachOrdered(System.out::print); // encounter orderを維持

不可能・誤りパターン

コード・判断結果理由
Callable<String> c = () -> { System.out.println("x"); };コンパイルエラーCallable<String>はStringを返す必要がある。
Runnable r = () -> "x";コンパイルエラーRunnable.run()はvoid。
Future<String> f = ex.submit(() -> System.out.println("x"));コンパイルエラーsubmit(Runnable)の戻り値はFuture<?>。
f.get()の例外処理なしコンパイルエラーchecked例外の処理が必要。
shutdown()後にsubmit実行時例外RejectedExecutionException。
shutdown()を「全タスク完了まで待つ」と読む誤り新規受付停止であり、待機ではない。
複数スレッドのprintln順を固定する誤り同期・待機・単一スレッドなどの保証がない。
synchronized(null) {}実行時例外NullPointerException。

読解ミニ演習

演習1: RunnableをsubmitしたFuture

ExecutorService ex = Executors.newSingleThreadExecutor();
Future<?> f = ex.submit(() -> System.out.print("A"));
System.out.print(f.get());
ex.shutdown();

判定: Anullと表示されます。Runnableは値を返さないため、Future.get()の結果はnullです。

演習2: getがある場合の順序

ExecutorService ex = Executors.newFixedThreadPool(2);
Future<String> f = ex.submit(() -> "A");
System.out.print(f.get());
System.out.print("B");
ex.shutdown();

判定: ABです。get()がAの完了を待つため、Bはその後です。

演習3: getがない場合の順序

ExecutorService ex = Executors.newFixedThreadPool(2);
ex.submit(() -> System.out.print("A"));
System.out.print("B");
ex.shutdown();

判定: ABまたはBAなど、順序を固定できません。mainスレッドとワーカースレッドの実行順が保証されていないためです。

演習4: submit後の例外

ExecutorService ex = Executors.newSingleThreadExecutor();
Future<Integer> f = ex.submit(() -> 10 / 0);
try {
  System.out.println(f.get());
} catch (ExecutionException e) {
  System.out.println("E");
}
ex.shutdown();

判定: Eです。タスク内のArithmeticExceptionは、get()時にExecutionExceptionとして受け取ります。

試験での判断基準

  • submitしただけで結果が表示されるとは限らない。
  • Future.get()は待機し、checked例外の処理が必要。
  • RunnableのFuture.get()は通常null。
  • タスク内の例外はExecutionExceptionに包まれる。
  • shutdown()は待機ではなく新規受付停止。
  • 複数スレッドの出力順は、保証がない限り固定しない。
  • 共有データ更新は、synchronizedやAtomicなどの対策がないと危険。
並行処理で落ちる人は、出力順を勘で決めています。保証がある順序と、保証がない順序を分けるのが最優先です。
