javac・javaコマンドで詰まる原因

Javaのコンパイル・実行コマンドで起こりやすいミスをまとめます。

Java実行・宣言ルールの全体像

Javaの実行ルールは、main メソッドだけを見ると断片的になります。まず、ソースコードが実行されるまでの流れを固定します。

Sample.java  --javac-->  Sample.class  --java-->  JVMがmainを呼び出す
用語役割判定ポイント
ソースファイル.java のファイル。人間が書くコード。public トップレベルクラス名とファイル名が一致するか。
コンパイルjavac がソースを検査し、classファイルを作る処理。型、構文、未初期化、アクセス制御、例外処理漏れを検査する。
classファイルJVMが実行できるバイトコード。コンパイルエラーなら作られない。したがって実行も出力もない。
JVMclassファイルを読み込み、実行する仕組み。実行時例外は、JVMが実行中に問題へ到達したときに発生する。
JDK開発に必要な道具一式。javac や java を含む。javac と java の役割を混同しない。
重要: 「コンパイルできること」と「javaコマンドで起動できること」は別です。構文上正しいクラスでも、正しい main がなければ起動用クラスにはなりません。

ファイル名・クラス名・mainメソッドを分ける

観点問われること判定
ファイル名public トップレベルクラスと一致しているか。一致しなければコンパイルエラー。
クラス宣言トップレベルクラスを複数書けるか。書ける。ただし public は1つまで。
mainメソッドjavaコマンドの入口として使える形か。public static void main(String[] args) または String... args。
起動引数args に何が入るか。渡された文字列の配列。引数なしなら長さ0で、null ではない。
 
// A.java
public class A { }

class B {
    public static void main(String[] args) {
        System.out.println("B");
    }
}

この例では、ファイル名は A.java で正しいためコンパイルできます。B は public ではありませんが、正しい main を持っていれば実行対象になれます。つまり、publicクラス名の規則 と 実行入口の規則 は別です。

クラス名とファイル名

public class Sampleなら、基本的にファイル名はSample.javaです。大文字小文字も一致させます。

// Sample.java
public class Sample {
    public static void main(String[] args) {}
}

実行時は拡張子を付けない

javacはファイル名を指定しますが、javaはクラス名を指定します。

javac Sample.java
java Sample
// java Sample.class は誤り

パッケージがある場合

package ex; のクラスは、ルートディレクトリから ex.Sample のように完全修飾名で実行します。

javac -d build src/ex/Sample.java
java -cp build ex.Sample

問題での注意

問題文に「classファイルはない」「カレントディレクトリ」「ファイル名」が書かれている場合は、コマンド問題のヒントです。

コンパイルエラーと実行時例外を混ぜない

分類例判断
コンパイルエラー型不一致、未初期化ローカル変数、アクセス不可実行前に止まる。出力はない。
実行時例外null参照、配列範囲外、キャスト失敗そこまでの出力は残る可能性がある。
正常出力型・アクセス・例外がすべて通る最後に評価順を追う。

出力より先に落ちる箇所を探す

コンパイルエラーの問題は、実行結果を追う前に終わります。未初期化、型不一致、アクセス不可、到達不能コード、catch順序を先に確認します。

分類例判断
型不一致int x = 10L;代入時点で不可。
未初期化ローカル変数を初期化前に使用デフォルト値はない。
到達不能return後の文構文上許されない。

よくある誤解と修正

誤解正しい見方確認方法
見た目が似ていれば同じ動きになる型、戻り値、参照、修飾子が1つ違うだけで結果は変わる。行ごとに「型」「値」「参照先」をメモする。
実行結果だけ追えばよいコンパイルできなければ実行されない。例外で止まれば後続行は動かない。コンパイル可否 → 実行時例外 → 出力の順に判定する。
暗記した用語で判断できる用語は入口にすぎない。コード上のどの行に効いているかまで見る。本文の例を1行変更し、結果が変わる理由を説明する。

ミニドリル

次の形式で、自分の答えを文章にしてください。正解だけを出しても、弱点は潰れません。

  1. このコードはコンパイルできるか。
  2. 実行時に例外が起きるなら、どの行で起きるか。
  3. 正常に動くなら、出力は何か。
  4. 自分が間違えた場合、原因は「型」「参照」「戻り値」「アクセス」「例外」「評価順」のどれか。
学習時は、選択肢を見る前に自分の判定を作ってください。選択肢を先に見ると、もっともらしい誤答に引っ張られます。
