例外処理完全整理

例外処理は、コンパイル可否、発生する例外の型、catchの一致、finallyとtry-with-resourcesの順序を分けて読む単元です。出力だけを追う読み方は捨ててください。

例外処理問題の判定順

Silverの例外問題は、出力を追う前にコンパイル可否を確定させます。コンパイルできないコードは、実行時例外も出力も発生しません。

  1. try / catch / finally の形が正しいかを見る。
  2. checked例外の処理漏れがないかを見る。
  3. catchの順序がサブクラスからスーパークラスになっているかを見る。
  4. コンパイルできる場合だけ、try内のどこで例外が発生するかを追う。
  5. 対応するcatch、finally、try-with-resourcesのclose()順を追う。
ここを雑に読むな: 「例外っぽいからcatchされる」ではありません。実際に発生した例外オブジェクトの型が、catchの型へ代入できるかで決まります。

例外とは

例外は、プログラム実行中に通常の流れを続けられなくなったことを表すオブジェクトです。Javaでは、例外もクラスから作られるオブジェクトとして扱います。

try {
    throw new IllegalArgumentException("bad");
} catch (RuntimeException e) {
    System.out.println(e.getClass().getSimpleName());
}
// IllegalArgumentException
用語意味試験での見方
例外オブジェクトnew IOException() のように作られるトラブル情報。型を見て、どのcatchに入るか判断する。
throw例外オブジェクトを実際に投げる文。この行で通常の流れは中断される。
throwsメソッドが例外を呼び出し元へ投げる可能性を宣言する句。checked例外の処理漏れ対策。
伝播catchされなかった例外が呼び出し元へ戻ること。呼び出し元にcatch/throwsが必要か見る。

checked例外とunchecked例外

Silverでは、例外の名前を見た瞬間に、コンパイラが処理を要求するかを判断できる必要があります。

Throwable
├─ Error                         // 通常はアプリ側で処理しない
└─ Exception
   ├─ RuntimeException            // unchecked例外
   │  ├─ NullPointerException
   │  ├─ ArithmeticException
   │  ├─ ClassCastException
   │  ├─ ArrayIndexOutOfBoundsException
   │  └─ NumberFormatException
   ├─ IOException                  // checked例外
   └─ SQLException など            // checked例外
分類代表例コンパイラの要求覚え方
checked例外IOException, Exceptioncatchするかthrows宣言が必要。RuntimeException 系ではない Exception はまず疑う。
unchecked例外RuntimeException とそのサブクラスcatch/throwsは任意。NullPointer、配列範囲外、数値変換失敗など。
ErrorStackOverflowError などcatch/throwsは通常不要。基本問題では積極的に処理対象にしない。
import java.io.IOException;

class Sample {
    static void read() throws IOException {
        throw new IOException();
    }

    public static void main(String[] args) {
        read();       // コンパイルエラー: checked例外の処理漏れ
    }
}

throwとthrowsの違い

書く場所意味例
throwメソッド本体の中で、例外オブジェクトを投げる。throw new RuntimeException();
throwsメソッド宣言部で、呼び出し元へ例外を投げる可能性を示す。void read() throws IOException
void a() throws Exception {  // 宣言
    throw new Exception();   // 実際に投げる
}
不可能パターン: throw IOException; は不可。投げるのはクラス名ではなく例外オブジェクトです。throw new IOException(); の形にします。
throw null;   // コンパイルは可能
              // 実行すると NullPointerException

try-catch-finallyの実行順

try内で例外が発生すると、その行以降のtry内処理はスキップされ、対応するcatchへ移ります。その後、finallyがあれば実行されます。

try {
    System.out.print("A");
    int x = 10 / 0;
    System.out.print("B");
} catch (ArithmeticException e) {
    System.out.print("C");
} finally {
    System.out.print("D");
}
// ACD
状況流れ
例外なしtry → finally
catchされる例外ありtry途中 → catch → finally
catchされない例外ありtry途中 → finally → 呼び出し元へ伝播

catchの順序

catchは上から順に判定されます。先にスーパークラス型で受け取ると、後ろのサブクラス型catchには絶対に到達できません。

try {
    throw new NumberFormatException();
} catch (RuntimeException e) {
    System.out.println("A");
} catch (NumberFormatException e) { // コンパイルエラー
    System.out.println("B");
}

正しい順序は、より具体的な例外を先に書く形です。

try {
    throw new NumberFormatException();
} catch (NumberFormatException e) {
    System.out.println("B");
} catch (RuntimeException e) {
    System.out.println("A");
}

multi-catchの制約

multi-catchでは、継承関係にある例外型を同時に並べられません。片方がもう片方を含むため、意味が重複するからです。

try {
    risky();
} catch (IOException | RuntimeException e) {
    System.out.println("handled");
}
try {
    risky();
} catch (Exception | IOException e) { // コンパイルエラー
    System.out.println("bad");
}

finallyで結果が変わるパターン

finally は例外の有無にかかわらず実行されます。ただし、finally の中で return や新しい例外を発生させると、try/catch側の結果を上書きします。

static int test() {
    try {
        return 1;
    } finally {
        return 2;
    }
}
// 戻り値は2
try {
    throw new IllegalArgumentException();
} finally {
    throw new NullPointerException();
}
// 最終的に外へ出るのは NullPointerException
注意: 実務ではfinallyでreturnする設計は避けるべきです。試験では「結果を上書きする」ひっかけとして出ます。

try-with-resources

try-with-resourcesは、AutoCloseable を実装したリソースを自動で閉じる構文です。closeは、tryブロックを抜けるときに宣言と逆順で呼ばれます。

class R implements AutoCloseable {
    private final String name;
    R(String name) { this.name = name; }
    public void close() { System.out.print(name); }
}

try (R a = new R("A"); R b = new R("B")) {
    System.out.print("T");
} finally {
    System.out.print("F");
}
// TBAF
  1. try本体が実行される。
  2. リソースは宣言と逆順にcloseされる。
  3. 最後にfinallyが実行される。

Java 9以降は、effectively finalな変数をtry-with-resourcesに指定できます。

R r = new R("R");
try (r) {
    System.out.print("T");
}

実行時例外の典型

例外発生条件例
NullPointerExceptionnull参照でインスタンスメンバを呼ぶ。s.length() で s == null
ArrayIndexOutOfBoundsException配列の範囲外添字。a[a.length]
StringIndexOutOfBoundsException文字列の範囲外位置。"abc".charAt(3)
NumberFormatException数値に変換できない文字列。Integer.parseInt("A")
ArithmeticException整数の0除算。10 / 0
ClassCastException実体に合わない参照型へキャスト。(String) new Object()

整数0除算と浮動小数点0除算

0除算は全部同じではありません。整数演算では ArithmeticException、浮動小数点演算では Infinity や NaN になります。

System.out.println(10 / 0);    // ArithmeticException
System.out.println(10.0 / 0);  // Infinity
System.out.println(0.0 / 0);   // NaN

読解ミニ演習

public class Main {
    public static void main(String[] args) {
        try {
            System.out.print("A");
            String s = null;
            System.out.print(s.length());
            System.out.print("B");
        } catch (ArithmeticException e) {
            System.out.print("C");
        } catch (RuntimeException e) {
            System.out.print("D");
        } finally {
            System.out.print("E");
        }
    }
}
答えを見る

ADE です。s.length() で NullPointerException が発生し、ArithmeticException では受け取れず、次の RuntimeException で受け取られます。try内の B は実行されません。
