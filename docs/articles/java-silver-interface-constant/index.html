interfaceの変数が暗黙にpublic static finalになる理由

interface内のフィールドと再代入不可の罠を整理します。

記事本文上部

この詳細記事で扱う範囲

ここでは、継承・interfaceの中でも問題でひっかけにされやすい細部を扱います。特に次の論点は、表面理解だと落ちます。

  • interfaceの暗黙修飾子
  • defaultメソッドの衝突と優先順位
  • interfaceのstatic/privateメソッド
  • abstractクラスとinterfaceの実装義務
  • sealed / non-sealed / permits
  • protectedと別パッケージのアクセス
  • フィールド解決とメソッド解決の違い

interfaceの暗黙修飾子

interfaceに書いたメンバは、クラスと同じ感覚で読むと間違えます。抽象メソッドとフィールドには暗黙の修飾子があります。

書いた形実質的な意味注意点
void test();public abstract void test();実装側は public 必須
int MAX = 10;public static final int MAX = 10;代入し直せない
default void test(){}実装付きインスタンスメソッドpublic 扱い。private/protected不可
static void test(){}interfaceに属するstaticメソッド実装クラスへ継承されない
private void helper(){}interface内部の補助メソッド外部から呼べない。実装クラスからも呼べない
 
interface A {
    int VALUE = 10;
    void run();
}

class B implements A {
    public void run() {          // publicが必要
        // VALUE = 20;           // NG: static final
        System.out.println(VALUE);
    }
}

defaultメソッドの基本

defaultメソッドは、interfaceに実装を持たせる仕組みです。実装クラスがoverrideしなければ、そのまま使えます。

interface A {
    default void test() {
        System.out.println("A");
    }
}

class B implements A { }

new B().test(); // A

ただし、defaultメソッドは「interfaceの通常メソッド」ではありません。default を付けずに本体を書くとコンパイルエラーです。

interface A {
    // void test() { } // NG: default, static, privateのいずれでもない実装付きメソッド
}

defaultメソッドの衝突

複数のinterfaceから同じシグネチャのdefaultメソッドを受け継ぐと、クラス側でどちらを使うか決めなければいけません。

interface A {
    default void test() { System.out.println("A"); }
}
interface B {
    default void test() { System.out.println("B"); }
}

class C implements A, B {
    public void test() {
        A.super.test();
    }
}

A.super.test() は、interfaceのdefault実装を明示的に呼ぶ構文です。super.test() ではありません。

クラスの具象メソッドがdefaultより優先される

スーパークラスに具象メソッドがあり、interfaceにも同じシグネチャのdefaultメソッドがある場合、クラス側が優先されます。

class A {
    public void test() {
        System.out.println("class");
    }
}

interface B {
    default void test() {
        System.out.println("interface");
    }
}

class C extends A implements B { }

new C().test(); // class

優先順位は、ざっくり「クラスが勝つ」です。defaultメソッドは、クラス階層に実装がないときの補助と考えると判断しやすくなります。

interfaceのstaticメソッドは継承されない

interfaceのstaticメソッドは、そのinterface名で呼び出します。実装クラス名やインスタンスから呼ぶものではありません。

interface A {
    static void test() {
        System.out.println("A");
    }
}

class B implements A { }

A.test();
// B.test();       // NG
// new B().test(); // NG

クラスのstaticメソッドとは感覚が違います。interface staticは「継承されない」と覚えてください。

interfaceのprivateメソッド

privateメソッドは、defaultメソッドやstaticメソッド内の共通処理を切り出すためのものです。実装クラスから呼ぶためのものではありません。

interface A {
    default void test() {
        log();
    }

    private void log() {
        System.out.println("log");
    }
}

class B implements A {
    void sample() {
        // log(); // NG: interfaceのprivateメソッドは呼べない
    }
}

interface同士の継承

interfaceはinterfaceを extends で継承します。クラスのように implements は使いません。また、interfaceは複数のinterfaceを継承できます。

interface A {
    void a();
}

interface B {
    void b();
}

interface C extends A, B {
    void c();
}

class D implements C {
    public void a() {}
    public void b() {}
    public void c() {}
}

D は C だけでなく、A と B の抽象メソッドも実装する必要があります。

抽象クラスはinterfaceメソッドを未実装のままにできる

interfaceを実装したクラスが抽象クラスなら、すべての抽象メソッドを実装しなくても構いません。ただし、最終的な具象クラスでは実装が必要です。

interface A {
    void test();
}

abstract class B implements A {
    // testを実装しなくてもOK。B自身がabstractだから
}

class C extends B {
    public void test() {
        System.out.println("C");
    }
}

protectedの別パッケージアクセス

protected は「同じパッケージならアクセス可」「別パッケージでも継承関係があれば条件付きでアクセス可」です。別パッケージのサブクラスから、親型の別インスタンス経由で自由に触れるわけではありません。

// package p1;
public class A {
    protected void test() {}
}

// package p2;
class B extends A {
    void ok() {
        test();       // OK: 自分自身の継承メンバとして呼ぶ
        this.test();  // OK
    }

    void ng(A a) {
        // a.test();  // NGになり得る: 別パッケージで親型参照経由
    }
}

protectedは「サブクラスなら何でもOK」ではありません。この誤解はかなり危険です。

sealedクラス・sealed interface

sealed は、継承・実装できるクラスを制限する仕組みです。許可する相手は permits で指定します。

sealed interface Shape permits Circle, Rectangle { }

final class Circle implements Shape { }
non-sealed class Rectangle implements Shape { }

sealedを継承・実装した側は、次のいずれかを明示する必要があります。

修飾子意味使う場面
finalここで継承を止めるこれ以上派生させない
sealedさらに許可先を制限する階層を引き続き管理する
non-sealed制限を解除するここから先は通常の継承に戻す
ひっかけ：permitsに書かれていないクラス

許可されていないクラスがsealedクラスを継承・実装するとコンパイルエラーです。

sealedの典型コンパイルエラー

sealed class A permits B { }

// class C extends A { } // NG: permitsにCがない

// class B extends A { } // NG: final/sealed/non-sealedのいずれかが必要

final class B extends A { } // OK

問題では、許可リストだけでなく、許可されたサブクラス側の修飾子も見ます。片方だけ見ても足りません。

メソッドとフィールドの解決を分ける

継承/interfaceの詳細問題では、出力追跡と組み合わされます。特に、同名フィールドとoverrideメソッドの混在に注意します。

class A {
    String value = "A";
    String getValue() { return value; }
}

class B extends A {
    String value = "B";
    String getValue() { return value; }
}

A a = new B();
System.out.println(a.value);      // A
System.out.println(a.getValue()); // B

フィールドは宣言型、インスタンスメソッドは実体。ここは機械的に分けてください。

defaultメソッドとObjectメソッド

interfaceのdefaultメソッドでは、Object の equals、hashCode、toString などをdefault実装として提供することはできません。すべてのクラスがObjectを継承するため、クラス側のルールと衝突するからです。

interface A {
    // default String toString() { return "A"; } // NG
}

これは細かいですが、interface問題で出されると落としやすい論点です。

総合判断フロー

  1. 宣言を読む：class / abstract class / interface / sealed interface を分類する。
  2. 継承関係を読む：extends / implements / permits を線でつなぐ。
  3. 未実装メソッドを見る：具象クラスが抽象メソッドを残していないか確認する。
  4. メソッド修飾子を見る：public不足、static/instance不一致、戻り値型、例外を確認する。
  5. 衝突を見る：複数default、クラス具象メソッド、同名フィールドを確認する。
  6. 実行を追う：コンパイルできる場合だけ、宣言型と実体を分けて出力を追う。

ミニ演習

Q1. interfaceの void test(); を実装するとき、void test(){} でよい？

ダメです。interfaceの抽象メソッドは暗黙にpublicなので、実装側は public void test(){} にします。

Q2. 2つのinterfaceに同じdefaultメソッドがある場合、実装クラスは空でよい？

ダメです。どちらを使うか曖昧なので、実装クラス側でoverrideして解決します。

Q3. interfaceのstaticメソッドは実装クラス名で呼べる？

呼べません。InterfaceName.method() の形で呼びます。

Q4. sealedクラスを継承したクラスは、何も修飾子を付けなくてよい？

ダメです。final、sealed、non-sealed のいずれかが必要です。
