Java Silverのコード読解で落ちない読み方

Java Silver系の問題は、コードを上から漫然と読むと落ちます。ここでは、問題文の種類ごとに「どの情報を信用し、どこを疑うか」を整理します。固定の判定手順ではなく、実際の問題で使える読み方にしています。

このページで扱うこと
  1. 出力問題に見えてコンパイルエラーになる例
  2. 実行時例外とコンパイルエラーの切り分け
  3. 参照型と実体型の読み分け
  4. 選択肢を消す順番

出力を追う前に、宣言と代入を疑う

public class Main {
    public static void main(String[] args) {
        int a;
        int b = 10;
        if (b > 0) {
            a = b;
        }
        System.out.println(a);
    }
}

このコードはコンパイルできます。b > 0 がコンパイル時定数ではないので迷いやすいですが、Javaの到達可能性・初期化判定は単純な実行結果の推測とは一致しません。対して、次はエラーです。

int a;
if (args.length > 0) {
    a = 10;
}
System.out.println(a); // 未初期化の可能性

ポイントは「実行したら代入されそう」ではなく、コンパイラが全経路で代入済みと判断できるかです。

参照型変数は「変数の型」と「実体の型」を分ける

class A { int x = 10; void print(){ System.out.println("A"); } }
class B extends A { int x = 20; void print(){ System.out.println("B"); } }
A obj = new B();
System.out.println(obj.x); // 10
obj.print();               // B

フィールドは変数の型で決まり、メソッドは実体の型で動的ディスパッチされます。ここを混ぜると、継承問題の半分以上を落とします。

選択肢の消し方

選択肢に「コンパイルエラー」「実行時例外」「出力結果」が並んでいる場合、出力候補を計算する前に、型・アクセス・例外・到達可能性を確認します。出力を最後まで追ってからコンパイルエラーに気付くのは時間の浪費です。

よくある誤答の原因

  • String の不変性を忘れる
  • == を値比較だと思う
  • static からインスタンスメンバーを呼ぶ
  • ローカル変数にデフォルト値が入ると思う
  • 親型変数から子クラス固有メソッドを呼べると思う
