例外処理とfinally

catch順序、checked例外、finallyの実行をまとめる。

記事本文上部

例外は「文字」ではなくオブジェクト

例外処理は、エラーメッセージを表示するだけの仕組みではありません。Javaでは、問題が起きたときに例外クラスのインスタンス、つまり 例外オブジェクト が投げられます。

用語意味コード上の見え方
例外オブジェクト発生した問題を表すオブジェクト。new IOException(), new NullPointerException()
throw例外オブジェクトを実際に発生させる。throw new Exception();
throwsそのメソッドが例外を呼び出し元へ投げる可能性を宣言する。void read() throws IOException
catch投げられた例外オブジェクトを受け取る。catch (IOException e)
伝播処理されなかった例外が呼び出し元へ戻っていくこと。呼び出し元の catch を探しに行く。
 
try {
    throw new IllegalArgumentException();
} catch (RuntimeException e) {
    System.out.println("caught");
}

この例では、投げられた実体は IllegalArgumentException ですが、RuntimeException 型の catch で受け取れます。例外も通常のクラスと同じく、継承関係によるポリモーフィズムで扱われます。

throw と throws を混同しない

項目throwthrows
書く場所メソッド本体の中に書く文。メソッド宣言に書く句。
役割実際に例外を投げる。例外を投げる可能性を宣言する。
後ろに書くもの例外オブジェクト。例外クラス名。
例throw new IOException();throws IOException
 
void sample() throws java.io.IOException {
    throw new java.io.IOException();
}
誤解を潰す: チェック例外は「重大な例外」という意味ではありません。コンパイラが処理または宣言を要求する例外 という意味です。RuntimeException とそのサブクラスは、原則としてその要求を受けません。

この単元の結論

例外処理の問題は、次の4点を順番に見ればかなり安定します。

  1. コンパイルできるか。チェック例外の処理漏れ、catch順序、try/catch/finallyの構文を確認する。
  2. tryのどこで例外が起きるか。例外が起きた行より後のtry内処理は実行されません。
  3. どのcatchが受け取るか。catchは上から順に評価されます。
  4. finallyとcloseの順序。finallyは原則最後。try-with-resourcesではcloseが先、finallyが後です。

例外問題でいきなり出力を追うのは悪手です。先にコンパイル可否を切ってください。特にチェック例外とcatch順序で落とされます。

例外クラスの大枠

Javaの例外・エラーは Throwable を根にした階層です。Javaでは、細かい暗記よりも「チェック例外か、非チェック例外か」を即判断できることが重要です。

Throwable
├─ Error                 // 通常はアプリ側で処理しない
└─ Exception
   ├─ RuntimeException   // 非チェック例外
   │  ├─ NullPointerException
   │  ├─ ArithmeticException
   │  ├─ ClassCastException
   │  ├─ ArrayIndexOutOfBoundsException
   │  └─ NumberFormatException
   └─ IOException など    // チェック例外
分類代表例コンパイラが処理を要求するか
チェック例外Exception、IOException、FileNotFoundException要求する
非チェック例外RuntimeException とそのサブクラス要求しない
ErrorOutOfMemoryError など通常は処理対象にしない

チェック例外と非チェック例外

チェック例外は、メソッド内で発生させる可能性があるなら、try-catch で処理するか、throws で呼び出し元に宣言しなければいけません。

static void checked() throws java.io.IOException {
    throw new java.io.IOException();
}

static void unchecked() {
    throw new RuntimeException();
}

IOException はチェック例外なので throws が必要です。RuntimeException は非チェック例外なので、throws を書かなくてもコンパイルできます。

Exceptionはチェック例外

Exception そのものはチェック例外です。ただし RuntimeException とそのサブクラスは非チェック例外です。「Exception系は全部チェック」と雑に覚えると危険です。

try-catch-finallyの実行順

try内で例外が発生したら、その行以降のtry内処理はスキップされます。その後、対応するcatchへ移動し、finallyがあれば最後に実行されます。

try {
    System.out.print("A");
    String s = null;
    s.length();
    System.out.print("B");
} catch (NullPointerException e) {
    System.out.print("C");
} finally {
    System.out.print("D");
}
// ACD
処理実行されるか理由
print("A")される例外発生前
s.length()ここで例外s が null
print("B")されない例外発生後のtry内処理
catchされるNullPointerException を受け取れる
finallyされる例外の有無にかかわらず実行

try構文の正しい形

try は単独では書けません。catch か finally の少なくとも一方が必要です。

// OK
try {
    test();
} catch (Exception e) {
}

// OK
try {
    test();
} finally {
}

// OK
try {
    test();
} catch (Exception e) {
} finally {
}

一方、次のように finally の後に catch を書くことはできません。

// NG
try {
    test();
} finally {
} catch (Exception e) {
}

catchの順序

catchは上から順に評価されます。スーパークラス例外を先に書くと、サブクラス例外のcatchに到達できなくなります。

// OK: サブクラスを先に書く
try {
    throw new java.io.FileNotFoundException();
} catch (java.io.FileNotFoundException e) {
    System.out.println("file");
} catch (java.io.IOException e) {
    System.out.println("io");
} catch (Exception e) {
    System.out.println("exception");
}
// NG: IOExceptionが先にあるため、FileNotFoundExceptionのcatchは到達不能
try {
    throw new java.io.FileNotFoundException();
} catch (java.io.IOException e) {
} catch (java.io.FileNotFoundException e) {
}

Javaでは、このcatch順序だけでコンパイルエラーを選ばせる問題があります。

catchされない例外

対応するcatchがなければ、例外は呼び出し元へ伝播します。mainメソッドまで処理されなければ、プログラムは異常終了します。

try {
    int[] nums = {};
    nums[0] = 10;
} catch (NullPointerException e) {
    System.out.println("NPE");
}
// ArrayIndexOutOfBoundsException はcatchされない

この例では、発生するのは ArrayIndexOutOfBoundsException です。NullPointerException のcatchでは受け取れません。

finallyの役割

finally は、例外が発生しても発生しなくても実行したい処理を書く場所です。

try {
    System.out.print("A");
} catch (Exception e) {
    System.out.print("B");
} finally {
    System.out.print("C");
}
// AC

例外が発生していないのでcatchは実行されません。しかしfinallyは実行されます。

finallyが実行されない例外的ケース

System.exit やJVM自体の停止など、通常の制御フローを外れる場合はfinallyが実行されないことがあります。ただしJavaでは、基本は「finallyは通る」と判断すれば十分です。

finally内のreturn・例外

finally内で return したり、新しい例外を投げたりすると、try/catch側の結果を上書きすることがあります。問題で出る場合は、最後に実行されたfinallyの制御を優先して見ます。

static int test() {
    try {
        return 1;
    } finally {
        return 2;
    }
}

System.out.println(test()); // 2

実務ではこのような書き方は避けるべきですが、問題では「finallyが最後に制御を奪う」ことを問われる場合があります。

throwとthrows

throw は例外インスタンスを実際に投げる文です。throws は、メソッド宣言で「このメソッドはこの例外を投げる可能性がある」と表すものです。

static void read() throws java.io.IOException {
    throw new java.io.IOException();
}
キーワード位置意味
throwメソッド本体例外インスタンスを投げる
throwsメソッド宣言呼び出し元へ例外の可能性を宣言する

throws宣言と呼び出し元

チェック例外をthrowsするメソッドを呼ぶ側は、その例外を処理するか、さらにthrowsで宣言する必要があります。

static void read() throws java.io.IOException {
    throw new java.io.IOException();
}

static void caller1() throws java.io.IOException {
    read(); // 呼び出し元へ投げる
}

static void caller2() {
    try {
        read();
    } catch (java.io.IOException e) {
        System.out.println("handled");
    }
}

override時の例外ルール

オーバーライドするメソッドでは、チェック例外を親より広くできません。親と同じ、またはより狭いチェック例外ならOKです。非チェック例外はこの制限の対象外です。

class A {
    void test() throws java.io.IOException {}
}

class B extends A {
    @Override
    void test() throws java.io.FileNotFoundException {} // OK
}

class C extends A {
    // @Override
    // void test() throws Exception {} // NG: IOExceptionより広い
}

親メソッドを呼ぶ側は「IOExceptionまで処理すればよい」と期待しています。子クラスで突然 Exception まで広げると、その期待を壊すため禁止されます。

try-with-resources

try-with-resources は、AutoCloseable を実装したリソースを自動でcloseする構文です。

class R implements AutoCloseable {
    private final String name;
    R(String name) {
        this.name = name;
    }
    public void close() {
        System.out.print(name);
    }
}

try (R r = new R("R")) {
    System.out.print("T");
}
// TR

tryブロックが終わった後に close() が呼ばれます。例外が発生してもcloseは呼ばれます。

複数リソースのclose順

try-with-resourcesで複数リソースを宣言した場合、close() は宣言と逆順に呼ばれます。

try (R a = new R("A"); R b = new R("B")) {
    System.out.print("T");
}
// TBA
  1. tryブロックで T を出力
  2. 後に宣言した b.close() で B
  3. 先に宣言した a.close() で A

try-with-resourcesとfinally

try-with-resourcesにfinallyが付く場合、close() が呼ばれた後にfinallyが実行されます。

try (R a = new R("A"); R b = new R("B")) {
    System.out.print("T");
} finally {
    System.out.print("F");
}
// TBAF

順序は、try本体 → close逆順 → finally です。

try-with-resourcesのリソース条件

try-with-resourcesで宣言できるのは、AutoCloseable または Closeable を実装した型です。

class R implements AutoCloseable {
    public void close() {}
}

try (R r = new R()) {
    System.out.println("use");
}

close() メソッドを持っているだけでは不十分です。型として AutoCloseable を実装している必要があります。

multi-catch

複数の例外に同じ処理をしたい場合、multi-catchを使えます。

try {
    test();
} catch (java.io.IOException | NumberFormatException e) {
    System.out.println("error");
}

ただし、継承関係のある例外を同じmulti-catchに並べることはできません。

// NG: FileNotFoundExceptionはIOExceptionのサブクラス
// catch (java.io.IOException | java.io.FileNotFoundException e) {}

カスタム例外

独自例外を作るとき、Exception を継承すればチェック例外、RuntimeException を継承すれば非チェック例外になります。

class MyCheckedException extends Exception {}
class MyUncheckedException extends RuntimeException {}

static void a() throws MyCheckedException {
    throw new MyCheckedException();
}

static void b() {
    throw new MyUncheckedException();
}

「自作例外だからチェック例外」とは限りません。何を継承しているかで決まります。

起動引数と例外

main(String[] args) の args は、起動引数がなくても null ではありません。長さ0の配列です。

public static void main(String[] args) {
    System.out.println(args.length); // 起動引数なしなら0
    System.out.println(args[0]);     // ArrayIndexOutOfBoundsException
}

「引数なしだからargsがnull」と考えるのは誤りです。要素数0の配列なので、args[0] にアクセスした時点で配列範囲外です。

典型ひっかけ

チェック例外の処理漏れ

IOException を投げるのにcatchもthrowsもないならコンパイルエラーです。

catch順序

スーパークラス例外を先にcatchすると、サブクラス例外のcatchが到達不能になります。

finallyの位置

try-finally-catch は不可。catchを書くならfinallyより前です。

try内の残りを実行してしまう

例外発生後、try内の残りはスキップされます。

close順序

try-with-resourcesのcloseは宣言と逆順です。

argsをnull扱いする

起動引数なしでも args は長さ0の配列です。

ミニ演習

Q1. IOException を throw するメソッドで、catchもthrowsもない場合は？

コンパイルエラーです。IOException はチェック例外なので、処理または宣言が必要です。

Q2. catch (Exception e) の後に catch (IOException e) を書ける？

書けません。IOException は Exception のサブクラスなので、先の Exception catchで必ず捕まってしまい、後続catchが到達不能です。

Q3. try-with-resourcesで A、B の順にリソースを宣言した場合、close順は？

B → A です。リソースは宣言と逆順にcloseされます。

Q4. 起動引数なしの args.length は？

0 です。args 自体は null ではありません。

Q5. RuntimeException をthrows宣言しないとコンパイルエラー？

なりません。RuntimeException は非チェック例外です。throws宣言は任意です。
