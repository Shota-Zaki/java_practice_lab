キャストとinstanceofの読み方

参照型キャスト、ClassCastException、instanceofパターンを整理します。

宣言型・実体・参照を最初に分ける

ポリモーフィズムは、「変数の型」と「実際に作られたオブジェクトの型」を分けて扱う仕組みです。まず次の3語を固定してください。

用語意味例何を決めるか
宣言型変数を宣言したときの型。A aその変数で呼べるメソッドを決める。
実体new で実際に作られたオブジェクト。new B()overrideメソッドの実行内容を決める。
参照変数が保持する、オブジェクトへのリンク情報。a = new B()どの実体を見ているかを示す。
 
class A { void test() { System.out.println("A"); } }
class B extends A { void test() { System.out.println("B"); } void onlyB() {} }

A a = new B();
a.test();   // 実行されるのはBのtest
// a.onlyB();  // コンパイルエラー。宣言型AにはonlyBがない

キャストは実体を変える処理ではない

キャストは、オブジェクトの中身を別クラスに作り替える処理ではありません。参照を「その型として扱ってよいか」をコンパイラやJVMに示すだけです。

種類例意味失敗条件
アップキャストA a = new B();サブクラスをスーパークラス型で扱う。基本的に安全。
ダウンキャストB b = (B) a;スーパークラス型の参照をサブクラス型として扱う。実体が合わなければ ClassCastException。
無関係な型へのキャスト(Car) dog変換可能性がなければコンパイル不可。コンパイルエラー、または実体不一致で実行時例外。
 
A a = new B();
((B) a).onlyB(); // キャストでBとして見せてからonlyBを呼ぶ

// (B) a.onlyB(); // NG: 先にa.onlyB()を解釈しようとする
判定の順番: 先に宣言型で「呼べるか」を見る。キャストがあるなら括弧の範囲を見る。最後に実体がキャスト先として扱えるかを見る。

この単元の結論

ポリモーフィズムとキャストの問題は、必ず 宣言型 と 実体 を分けて見ます。ここを混ぜると、ほぼ確実に間違えます。

見るもの決まること例
宣言型その場で呼べるメソッド・参照できるフィールドA a なら、基本的に A に見えるメンバだけ
実体overrideされたインスタンスメソッドの実行内容new B() なら、overrideメソッドは B 側が動く
キャストコンパイル時の見え方を一時的に変える((B) a).onlyB()
実行時の型ダウンキャストが安全かどうか実体が B でなければ ClassCastException

雑に「中身はBだからBのメソッドを呼べる」と考えるのは誤りです。呼べるかどうかはまず宣言型で決まります。

判定フロー

  1. クラス・interfaceの関係を確認する。親子関係があるか、interface実装関係があるか。
  2. 左辺の宣言型を見る。その変数型でメソッド呼び出しがコンパイルできるか。
  3. キャスト式の有無を見る。キャスト後の型でメソッドを探しているか、括弧の位置は正しいか。
  4. 実体を見る。ダウンキャストなら、実体がその型として扱えるか。
  5. メソッドかフィールドかを見る。インスタンスメソッドは実体側、フィールドとstaticメソッドは宣言型寄りに解決されます。
  6. 例外発生を見る。型関連はあるが実体が合わなければ ClassCastException です。

宣言型と実体を分ける

次のコードでは、変数 a の宣言型は A、実体は B です。

class A {
    void test() {
        System.out.println("A");
    }
}

class B extends A {
    @Override
    void test() {
        System.out.println("B");
    }

    void onlyB() {
        System.out.println("onlyB");
    }
}

A a = new B();
a.test();      // B
// a.onlyB();  // コンパイルエラー

a.test() はコンパイルできます。A に test() があるからです。実行されるのは B のoverrideメソッドです。

一方、a.onlyB() はコンパイルエラーです。実体が B でも、宣言型 A には onlyB() が見えません。

アップキャスト

サブクラスのインスタンスは、スーパークラス型の変数に代入できます。これをアップキャストと呼びます。明示キャストは不要です。

class Animal {}
class Dog extends Animal {}

Dog dog = new Dog();
Animal animal = dog;       // OK: DogはAnimalとして扱える
Animal animal2 = new Dog(); // OK

「DogはAnimalである」と言えるので自然に代入できます。ただし、Animal 型として扱っている間は、Animal に見えるメンバしか呼べません。

ダウンキャスト

スーパークラス型の参照をサブクラス型として扱うには、明示キャストが必要です。ただし、キャストを書けることと実行時に安全なことは別です。

class Animal {}
class Dog extends Animal {
    void bark() {
        System.out.println("bark");
    }
}

Animal animal = new Dog();
Dog dog = (Dog) animal; // OK: 実体がDog
dog.bark();

この場合、変数 animal の宣言型は Animal ですが、実体は Dog です。そのため (Dog) animal は実行時にも成功します。

ClassCastExceptionになる形

型に継承関係があるためキャスト式自体はコンパイルできても、実体が目的の型でなければ実行時に ClassCastException です。

class Animal {}
class Dog extends Animal {}
class Cat extends Animal {}

Animal animal = new Cat();
Dog dog = (Dog) animal; // 実行時にClassCastException

変数 animal は Animal 型なので、コンパイラは「実体がDogの可能性もある」と見ます。そのためキャスト式は通ります。しかし実体は Cat なので、実行時に失敗します。

コンパイルエラーになるキャスト

そもそも型同士に変換可能性がない場合、キャスト式自体がコンパイルエラーになります。

class Dog {}
class Car {}

Dog dog = new Dog();
// Car car = (Car) dog; // コンパイルエラー

Dog と Car に継承関係もinterface実装関係もなければ、キャストできる可能性がありません。この場合は実行時例外ではなく、コンパイルエラーです。

interfaceが絡む場合は慎重に見る

interface型へのキャストは「実装している可能性」があるため、クラス同士よりコンパイルが通る場面があります。ただし、実体がそのinterfaceを実装していなければ実行時に ClassCastException

nullのキャスト

null はどの参照型にもキャストできます。実体がないため、キャスト自体で ClassCastException にはなりません。

String s = (String) null; // OK
System.out.println(s);    // null
// s.length();            // NullPointerException

キャストは成功します。ただし、その後にメソッドを呼ぶと、参照先がないため NullPointerException です。

メソッドは実体で動く

overrideされたインスタンスメソッドは、実行時の実体で決まります。

class A {
    String name() {
        return "A";
    }
}

class B extends A {
    @Override
    String name() {
        return "B";
    }
}

A a = new B();
System.out.println(a.name()); // B

呼び出し可能かどうかは宣言型 A で確認します。A に name() があるのでコンパイルできます。実行時は実体 B のoverrideメソッドが選ばれます。

フィールドはoverrideされない

フィールドはメソッドと違い、overrideされません。同じ名前のフィールドをサブクラスに定義すると、隠蔽に近い状態になります。参照するフィールドは宣言型で決まります。

class A {
    String value = "A";
}

class B extends A {
    String value = "B";
}

A a = new B();
System.out.println(a.value);       // A
System.out.println(((B) a).value); // B

ここは問題でよく落とされます。メソッドは実体、フィールドは宣言型 とまず覚えてください。

staticメソッドもoverrideされない

staticメソッドはインスタンスに属さず、クラスに属します。サブクラスで同じシグネチャのstaticメソッドを書いてもoverrideではなく、隠蔽です。

class A {
    static void test() {
        System.out.println("A");
    }
}

class B extends A {
    static void test() {
        System.out.println("B");
    }
}

A a = new B();
a.test(); // A。コンパイル時の型Aで解決される

実務ではインスタンス参照からstaticメソッドを呼ぶべきではありません。ただし問題では出ます。overrideの動的ディスパッチとは別物です。

overloadはコンパイル時に決まる

overloadは引数の型で呼び分ける仕組みです。どのoverloadを呼ぶかは、基本的にコンパイル時の引数型で決まります。

class A {}
class B extends A {}

class Sample {
    void test(A a) {
        System.out.println("A");
    }
    void test(B b) {
        System.out.println("B");
    }
}

Sample s = new Sample();
A x = new B();
s.test(x); // A

実体は B ですが、変数 x の宣言型は A です。そのため test(A) が選ばれます。

overload選択後にoverrideが効く

やや難しいですが、まずoverloadがコンパイル時に選ばれ、その後、そのメソッドがoverrideされていれば実体側が実行されます。

class A {}
class B extends A {}

class Parent {
    void test(A a) {
        System.out.println("Parent A");
    }
    void test(B b) {
        System.out.println("Parent B");
    }
}

class Child extends Parent {
    @Override
    void test(A a) {
        System.out.println("Child A");
    }
}

Parent p = new Child();
A arg = new B();
p.test(arg); // Child A
  1. arg の宣言型は A なので、overloadでは test(A) が選ばれる。
  2. p の実体は Child なので、test(A) のoverride先である Child A が実行される。

キャストとメソッド呼び出しの括弧

キャストしてからメソッドを呼ぶには、キャスト式全体を括弧で囲みます。括弧がずれると、先にメソッドを探してしまいコンパイルエラーになります。

Object obj = "abc";

// String s = (String) obj.substring(1); // NG: Objectにsubstringはない
String s = ((String) obj).substring(1);  // OK

(String) obj.substring(1) のように見える式は、obj.substring(1) を先に解釈しようとします。Object に substring はないので失敗します。

instanceofの基本

instanceof は、左辺の参照先インスタンスが右辺の型として扱えるかを判定します。

Object obj = "abc";

System.out.println(obj instanceof String);        // true
System.out.println(obj instanceof StringBuilder); // false

変数の宣言型だけではなく、実体を見ます。obj の宣言型は Object ですが、実体が String なので String 判定はtrueです。

instanceofとnull

null instanceof 型 は常にfalseです。例外にはなりません。

String s = null;
System.out.println(s instanceof String); // false

ここで NullPointerException になると考えるのは誤りです。instanceof はメソッド呼び出しではありません。

instanceofのパターンマッチング

Java 17では、instanceof と同時に変数を宣言できます。trueと判定された範囲内で、その変数を使えます。

Object obj = "abc";

if (obj instanceof String s) {
    System.out.println(s.length()); // sはString型として使える
}

パターン変数 s は、条件がtrueになる範囲でだけ有効です。if文の外で無条件に使えるわけではありません。

Object obj = "abc";

if (!(obj instanceof String s)) {
    return;
}
System.out.println(s.length()); // ここではStringであることが確定している

interface型で受けるポリモーフィズム

interface型の変数は、そのinterfaceを実装したクラスのインスタンスを受けられます。呼べるのはinterfaceに定義されたメソッドです。

interface Printable {
    void print();
}

class Report implements Printable {
    public void print() {
        System.out.println("report");
    }
    void export() {
        System.out.println("export");
    }
}

Printable p = new Report();
p.print();   // OK
// p.export(); // NG: Printable型にはexportがない

interface型で受ける意味は、実装クラスを差し替えられることです。ただし、宣言型に見えないメソッドは呼べません。

配列のポリモーフィズムとArrayStoreException

配列は参照型であり、配列にもポリモーフィズムがあります。ただし、配列は実行時に要素型を覚えているため、代入時に ArrayStoreException が起きることがあります。

Object[] values = new String[2];
values[0] = "A";          // OK
values[1] = new Object(); // 実行時にArrayStoreException

宣言型は Object[] なので Object を入れられそうに見えます。しかし実体は String[] なので、String以外を格納できません。

総合例：何が表示されるか

class A {
    String value = "A-field";
    String name() {
        return "A-method";
    }
}

class B extends A {
    String value = "B-field";
    @Override
    String name() {
        return "B-method";
    }
}

public class Main {
    public static void main(String[] args) {
        A a = new B();
        System.out.println(a.value);
        System.out.println(a.name());
        System.out.println(((B) a).value);
    }
}
式判断結果
a.valueフィールドは宣言型 AA-field
a.name()メソッドは実体 BB-method
((B) a).valueキャスト後の宣言型は BB-field

ミニ演習

Q1. A a = new B(); で、B だけにある onlyB() は呼べる？

そのままでは呼べません。宣言型 A に onlyB() がないためコンパイルエラーです。実体が B であることは、メソッド探索の前提にはなりません

Q2. A a = new C(); B b = (B) a; は必ずコンパイルエラー？ ただし B と C はどちらも A のサブクラス。

コンパイルは通る可能性があります。 a の宣言型は A であり、 B に変換できる可能性があるためです。ただし実体が C なので、実行時に ClassCastException

Q3. null instanceof String の結果は？

false です。 NullPointerException ではありません

Q4. A a = new B(); System.out.println(a.value); で A と B の両方に value がある場合は？

宣言型 A のフィールドが参照されます。フィールドはoverrideされません
