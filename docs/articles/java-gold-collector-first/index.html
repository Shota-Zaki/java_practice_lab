Collectors完全整理

Collectorsは、Streamの終端操作collectに渡す「集め方」です。Goldでは、結果の型、重複キー、順序、変更可能性、下流Collectorを読めないと落とされます。

最初に読む場所

collect(Collectors.xxx(...))の問題は、Streamの中身を追う前に、まず「どのCollectorに集めているか」を確認します。toListならList、toMapならMap、groupingByならMapの値側がさらにListやSetなどになります。

Collector主な戻り値試験で見る点
toList()List<T>変更可能性を決めつけない。Java 16以降のStream.toList()とは別物。
toSet()Set<T>重複は消える。順序を決めつけない。
toCollection(Supplier)指定したCollectionArrayList::new、LinkedHashSet::newなどで実装型を指定できる。
joining()StringStream要素はCharSequence系。区切り文字、前後文字も指定可能。
toMap()Map<K,V>キー重複、値null、マージ関数、Map実装型。
groupingBy()Map<K,List<T>>など分類キーと下流Collectorの型を分けて読む。
partitioningBy()Map<Boolean,List<T>>などtrue側/false側の2分類。キーはBoolean。
counting() / summingInt()Long / Integerなど下流Collectorで使われることが多い。

collectは終端操作

collectを呼んだ時点でStreamは消費されます。collect後に同じStreamをもう一度使うことはできません。

var stream = Stream.of("A", "B", "C");
var list = stream.collect(Collectors.toList());
// stream.count(); // 実行時例外: stream has already been operated upon or closed

Collectors.toList() と Stream.toList() は別物

Gold SE 17では特に重要です。stream.collect(Collectors.toList())とstream.toList()は似ていますが、同じものとして扱うと危険です。

書き方導入変更可能性試験での注意
collect(Collectors.toList())古くから使用可能仕様上、変更可能とは保証されない実装がArrayListだと決めつけない。
Stream.toList()Java 16以降変更不可addするとUnsupportedOperationException。
Collectors.toUnmodifiableList()Java 10以降変更不可null要素を許さない。
 
var a = Stream.of("A", "B").collect(Collectors.toList());
var b = Stream.of("A", "B").toList();

// a.add("C"); // 実装依存で動くことはあるが、仕様として保証しない
// b.add("C"); // UnsupportedOperationException
雑に「toListはArrayList」と覚えるのは危険です。試験では、仕様上保証されているかどうかで判断します。

toSet と toCollection

toSet()は重複を除去します。ただし、戻るSetの具体的な実装型や順序は保証されません。順序や実装型が必要ならtoCollectionで指定します。

var set = Stream.of("B", "A", "A")
    .collect(Collectors.toSet());
// 重複は消える。ただし表示順は決めつけない。

var linked = Stream.of("B", "A", "A")
    .collect(Collectors.toCollection(LinkedHashSet::new));
// LinkedHashSetを指定しているので、挿入順を保ちやすい。

joining

joiningは文字列を連結するCollectorです。要素型はStringなどのCharSequence系である必要があります。数値Streamにそのままjoiningは使えません。

String s1 = Stream.of("A", "B", "C")
    .collect(Collectors.joining());
System.out.println(s1); // ABC

String s2 = Stream.of("A", "B", "C")
    .collect(Collectors.joining(",", "[", "]"));
System.out.println(s2); // [A,B,C]

String nums = Stream.of(1, 2, 3)
    .map(String::valueOf)
    .collect(Collectors.joining("-"));
System.out.println(nums); // 1-2-3

toMapは重複キーで落ちる

toMap(keyMapper, valueMapper)で同じキーが生成されると、マージ関数を指定していない場合は実行時例外です。コンパイルエラーではありません。

Map<Integer, String> map = Stream.of("aa", "bb")
    .collect(Collectors.toMap(String::length, s -> s));
// どちらもキー2になるため、実行時例外: IllegalStateException

重複する可能性があるなら、第三引数でマージ関数を指定します。

Map<Integer, String> map = Stream.of("aa", "bb", "ccc")
    .collect(Collectors.toMap(
        String::length,
        s -> s,
        (oldValue, newValue) -> oldValue
    ));

System.out.println(map); // {2=aa, 3=ccc} など
toMapの形意味注意点
toMap(key, value)キーと値を作るキー重複でIllegalStateException。
toMap(key, value, merge)重複時の処理を指定oldValueとnewValueのどちらを残すか読む。
toMap(key, value, merge, supplier)Map実装型も指定LinkedHashMap::newなど。
 
Map<Character, String> map = Stream.of("apple", "ant", "banana")
    .collect(Collectors.toMap(
        s -> s.charAt(0),
        s -> s,
        (a, b) -> a + "/" + b,
        LinkedHashMap::new
    ));

System.out.println(map); // {a=apple/ant, b=banana}

toMapのnull

toMapでは、値側がnullになるとNullPointerExceptionです。HashMap自体はnull値を扱えることがありますが、Collectors.toMapの処理中ではnull値が問題になります。キー側のnullは実装上通る場合がありますが、試験ではnullを絡めたCollectorは危険箇所として扱うべきです。

Stream.of("A")
    .collect(Collectors.toMap(s -> s, s -> null));
// NullPointerException

groupingByは「Mapのキー」と「値側」を分けて読む

groupingByの基本形は、分類キーごとに要素をListへ集めます。

Map<Integer, List<String>> byLength = Stream.of("a", "bb", "cc", "ddd")
    .collect(Collectors.groupingBy(String::length));

System.out.println(byLength); // {1=[a], 2=[bb, cc], 3=[ddd]} など

第二引数に下流Collectorを渡すと、値側をList以外にできます。

Map<Integer, Long> countByLength = Stream.of("a", "bb", "cc", "ddd")
    .collect(Collectors.groupingBy(
        String::length,
        Collectors.counting()
    ));

System.out.println(countByLength); // {1=1, 2=2, 3=1} など
groupingBy戻り値の例読み方
groupingBy(String::length)Map<Integer,List<String>>長さごとに元要素をList化。
groupingBy(String::length, counting())Map<Integer,Long>長さごとに件数。
groupingBy(String::length, mapping(...))下流次第分類後に値側を変換。
groupingBy(key, LinkedHashMap::new, downstream)指定MapMap実装型も指定。

下流Collector: mapping / filtering / flatMapping

Goldで差がつくのは下流Collectorです。groupingByの後、値側をどう集めるかをさらに指定できます。

record User(String dept, String name, int score) {}

var users = List.of(
    new User("dev", "A", 80),
    new User("dev", "B", 60),
    new User("sales", "C", 90)
);

Map<String, List<String>> namesByDept = users.stream()
    .collect(Collectors.groupingBy(
        User::dept,
        Collectors.mapping(User::name, Collectors.toList())
    ));

System.out.println(namesByDept); // {dev=[A, B], sales=[C]} など
Map<String, List<String>> highScorers = users.stream()
    .collect(Collectors.groupingBy(
        User::dept,
        Collectors.filtering(
            u -> u.score() >= 80,
            Collectors.mapping(User::name, Collectors.toList())
        )
    ));

System.out.println(highScorers); // {dev=[A], sales=[C]} など

filteringを下流で使う場合、分類キー自体は残ることがあります。Streamの前段でfilterする場合とは結果のMap構造が変わる点に注意します。

partitioningByはBooleanキーの2分類

partitioningByは、条件に合うものをtrue、合わないものをfalseに分けます。キーは任意の値ではなくBooleanです。

Map<Boolean, List<Integer>> result = Stream.of(1, 2, 3, 4)
    .collect(Collectors.partitioningBy(n -> n % 2 == 0));

System.out.println(result); // {false=[1, 3], true=[2, 4]} など
Map<Boolean, Long> count = Stream.of(1, 2, 3, 4)
    .collect(Collectors.partitioningBy(
        n -> n % 2 == 0,
        Collectors.counting()
    ));

System.out.println(count); // {false=2, true=2} など

数値集約: counting / summingInt / averagingInt / summarizingInt

数値系Collectorは、単体でも使えますが、groupingByの下流でよく出ます。

record Item(String type, int price) {}

var items = List.of(
    new Item("book", 1000),
    new Item("book", 1500),
    new Item("food", 500)
);

Map<String, Integer> sum = items.stream()
    .collect(Collectors.groupingBy(
        Item::type,
        Collectors.summingInt(Item::price)
    ));

System.out.println(sum); // {book=2500, food=500} など
IntSummaryStatistics stats = items.stream()
    .collect(Collectors.summarizingInt(Item::price));

System.out.println(stats.getCount()); // 3
System.out.println(stats.getSum());   // 3000
System.out.println(stats.getMin());   // 500
System.out.println(stats.getMax());   // 1500

collectingAndThen

collectingAndThenは、集めた後に最後の変換をかけます。たとえば、Listに集めた後で変更不可Listにする、最大値Optionalから値を取り出す、などです。

List<String> list = Stream.of("A", "B")
    .collect(Collectors.collectingAndThen(
        Collectors.toList(),
        Collections::unmodifiableList
    ));

// list.add("C"); // UnsupportedOperationException

maxBy / minBy はOptionalを返す

maxByやminByは、要素がない可能性があるためOptional<T>を返します。groupingByの下流で使うと、値側がOptionalになります。

Optional<String> max = Stream.of("a", "bbb", "cc")
    .collect(Collectors.maxBy(Comparator.comparingInt(String::length)));

System.out.println(max.orElse("none")); // bbb
Map<Integer, Optional<String>> maxByLength = Stream.of("a", "bb", "cc")
    .collect(Collectors.groupingBy(
        String::length,
        Collectors.maxBy(Comparator.naturalOrder())
    ));

System.out.println(maxByLength); // {1=Optional[a], 2=Optional[cc]} など

reducing は型が読みにくい

Collectors.reducingは、Streamのreduceに似ていますが、Collectorとして使います。下流Collectorで登場すると型が崩れやすいので、戻り値を必ず書いて確認します。

reducingの形戻り値用途
reducing(BinaryOperator)Optional<T>空の可能性がある。
reducing(identity, BinaryOperator)T初期値があるのでOptionalではない。
reducing(identity, mapper, op)U変換してから畳み込む。
 
Integer total = Stream.of("A", "BB", "CCC")
    .collect(Collectors.reducing(
        0,
        String::length,
        Integer::sum
    ));

System.out.println(total); // 6

コンパイルエラーになりやすいパターン

コード結果理由
Stream.of(1,2).collect(Collectors.joining(","))コンパイルエラーjoiningはCharSequence系Stream用。
Collectors.groupingBy(s -> s.length(), String::toUpperCase)コンパイルエラー第二引数はCollector。通常のFunctionではない。
Collectors.toMap(String::length, s -> s)実行時例外の可能性同じ長さの文字列があるとキー重複。
collect(Collectors.toMap(s -> s, s -> null))実行時例外値nullでNullPointerException。
Stream.of("A").toList().add("B")実行時例外Stream.toList()の戻り値は変更不可。

読解ミニ演習

次のコードの結果を考えます。

var result = Stream.of("ant", "apple", "bee", "boat")
    .collect(Collectors.groupingBy(
        s -> s.charAt(0),
        Collectors.mapping(String::length, Collectors.toList())
    ));

System.out.println(result);

分類キーは先頭文字です。値側は元の文字列ではなく、mapping(String::length, toList())によって長さのListになります。

{a=[3, 5], b=[3, 4]}  // 表示順はMap実装に依存
悪い判断

groupingByだから値側は必ずList<String>だと決めつける。

良い判断

groupingByの第二引数以降を読み、下流Collectorで値側の型がどう変わるかを確認する。

Collectors問題は、Streamの流れを追う前に戻り値の型を固定してください。型が見えていない状態で値だけ追うと、Optional、Map<K,List<V>>、Map<K,Long>を取り違えます。
