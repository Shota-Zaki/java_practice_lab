Stream API完全整理

GoldでStreamを落とす原因は、メソッド名の暗記不足ではありません。型の流れ、評価タイミング、終端操作の戻り値、Collectorの戻り値を追えていないことです。ここではコードを読む順番で整理します。

最初に見るべき4点

Stream問題は、次の順番で読むと崩れにくくなります。

  1. Streamの要素型は何か。例: Stream<String>、IntStream。
  2. 中間操作で要素型が変わるか。特に map、flatMap、mapToInt。
  3. 終端操作の戻り値は何か。例: count は long、findFirst は Optional<T>。
  4. 実際に何個の要素が評価されるか。遅延評価と短絡操作を確認する。

Streamはデータそのものではなく、処理の流れ

Streamはコレクションではありません。要素を保持する箱ではなく、要素を流して処理するためのパイプラインです。そのため、基本的に同じStreamを2回使うことはできません。

var stream = List.of("A", "B", "C").stream();
System.out.println(stream.count());
System.out.println(stream.count()); // IllegalStateException
一度終端操作を実行したStreamは消費済みです。2回目の終端操作は実行時例外です。これはコンパイルエラーではありません。

Streamの作り方

作り方例要素型注意点
Collection.stream()list.stream()コレクションの要素型最も基本。元のListを直接変更するわけではない。
Stream.ofStream.of("A", "B")Stream<String>引数を要素として流す。
Arrays.streamArrays.stream(array)配列の要素型int[]ならIntStreamになる。
Stream.emptyStream.<String>empty()指定した型型推論が弱い場面では型引数を明示する。
Stream.generateStream.generate(() -> "A")Supplierの戻り値型無限Stream。通常はlimitが必要。
Stream.iterateStream.iterate(1, n -> n + 1)初期値の型無限Stream。Java 9以降は条件付きiterateもある。

中間操作と終端操作

中間操作はStreamを返してパイプラインを続けます。終端操作はStream以外の結果を返すか、副作用を実行してパイプラインを終了します。

分類代表例戻り値試験での見方
中間操作filterStream<T>条件に合う要素だけ残す。
中間操作mapStream<R>要素型が変わる。
中間操作flatMapStream<R>Streamの入れ子を平らにする。
中間操作distinct / sortedStream<T>状態を持つ中間操作。順序や比較に注意。
中間操作peekStream<T>主にデバッグ用。終端操作がなければ実行されない。
終端操作countlongintではない。
終端操作forEachvoid戻り値なし。代入できない。
終端操作findFirst / findAnyOptional<T>値がない可能性がある。
終端操作anyMatch / allMatch / noneMatchboolean短絡する。
終端操作collectCollector次第List、Set、Map、Stringなどに集約。

遅延評価: 中間操作だけでは実行されない

中間操作は「予約」されるだけです。終端操作が呼ばれて初めて、必要な要素が上流から流れます。

Stream<String> s = Stream.of("A", "B")
    .peek(System.out::print);

// ここでは何も表示されない
s.count(); // 終端操作

ただし、countはサイズが既知のソースでは最適化され、peekが実行されないことがあります。試験ではpeekの副作用に頼る読み方は危険です。peekは「必ず表示確認に使える操作」と雑に覚えるべきではありません。

filter・map・peekの実行順

Streamは「全要素にfilterを全部かけてからmap」ではなく、基本的には要素ごとにパイプラインを流れます。

Stream.of("a", "bb", "ccc")
    .filter(s -> {
        System.out.print("F" + s + " ");
        return s.length() > 1;
    })
    .map(s -> {
        System.out.print("M" + s + " ");
        return s.toUpperCase();
    })
    .forEach(s -> System.out.print("E" + s + " "));

// Fa Fbb Mbb EBB Fccc Mccc ECCC

短絡操作: 全要素を評価しないことがある

anyMatch、allMatch、noneMatch、findFirst、limitは、結果が確定した時点で処理を止めることがあります。

boolean result = Stream.of(1, 2, 3, 4)
    .peek(System.out::print)
    .anyMatch(n -> n > 2);

// 123 と表示され、result は true
操作空Streamでの結果短絡の例
anyMatchfalsetrueが出た時点で終了。
allMatchtruefalseが出た時点で終了。
noneMatchtruetrueになる要素が見つかった時点で終了。
findFirstOptional.empty()最初の要素が見つかれば終了。

mapとflatMapの違い

mapは1要素を1つの結果に変換します。flatMapは1要素をStreamに変換し、それらを1本のStreamに平らにします。

var list = List.of("A,B", "C,D");

var a = list.stream()
    .map(s -> Arrays.stream(s.split(",")));
// Stream<Stream<String>>

var b = list.stream()
    .flatMap(s -> Arrays.stream(s.split(",")));
// Stream<String>
flatMapのラムダは、値そのものではなくStreamを返します。.flatMap(s -> s.toUpperCase()) のようなコードは戻り値型が合わずコンパイルエラーです。

プリミティブStream: Stream<Integer>とは別物

IntStream、LongStream、DoubleStreamは、Stream<Integer>などとは別の型です。使えるメソッドと戻り値が変わります。

操作戻り値注意点
mapToIntIntStream以後はsum、averageなどが使える。
boxedStream<Integer>プリミティブStreamをラッパー型Streamへ戻す。
sumint / long / doubleStream<Integer>には直接ない。
averageOptionalDouble空の可能性がある。
max / minOptionalIntなど空Streamでは値がない。
int total = Stream.of("10", "20")
    .mapToInt(Integer::parseInt)
    .sum(); // 30

OptionalDouble avg = IntStream.of(10, 20, 30).average();

reduceの3パターン

reduceは要素を1つの値へ畳み込みます。オーバーロードごとに戻り値が違います。

形戻り値空Stream
reduce(BinaryOperator<T>)Optional<T>Optional.empty()
reduce(identity, BinaryOperator<T>)Tidentity
reduce(identity, accumulator, combiner)Uidentity
Optional<Integer> a = Stream.of(1, 2, 3)
    .reduce((x, y) -> x + y); // Optional[6]

int b = Stream.of(1, 2, 3)
    .reduce(10, (x, y) -> x + y); // 16

int c = Stream.of("A", "BB")
    .reduce(0,
        (sum, s) -> sum + s.length(),
        (x, y) -> x + y); // 3
3引数reduceは、逐次Streamではcombinerが目立たないことがあります。しかしparallelではcombinerが重要です。試験では「accumulatorとcombinerの型が合うか」を優先して見ます。

collectとCollectors

collectは、Streamの要素をList、Set、Map、Stringなどへ集約します。戻り値は指定したCollectorで決まります。

Collector戻り値の典型注意点
Collectors.toList()List<T>具体クラスや変更可能性に依存しない。
Collectors.toSet()Set<T>順序は保証されないことがある。
Collectors.joining()String要素はCharSequence系。
Collectors.counting()Longlongではなくラッパー型として扱われる場面がある。
Collectors.mapping()下流Collector次第集約前に要素を変換する。
Collectors.groupingBy()Map<K, List<T>>など下流Collectorで値側が変わる。
Collectors.partitioningBy()Map<Boolean, ...>キーはtrueとfalse。

toList()とCollectors.toList()

Java 16以降はStreamにtoList()があります。ただし、試験やコード読解ではcollect(Collectors.toList())との違いを雑に扱わないでください。

List<String> a = Stream.of("A", "B").toList();
List<String> b = Stream.of("A", "B").collect(Collectors.toList());

Java Gold SE 11だけを見る場合、Stream.toList()はJava 16以降のAPIなので対象外です。SE 11では基本的にcollect(Collectors.toList())で読みます。SE 17範囲まで見るなら、Stream.toList()との違いも押さえます。

Stream.toList()の戻り値は変更不可です。a.add("C")は実行時にUnsupportedOperationExceptionです。Collectors.toList()は変更可能なListを返すことが多いですが、仕様として具体的な型や変更可能性に依存すべきではありません。

toMapは重複キーで落ちる

Collectors.toMapはキーが重複すると、マージ関数を指定していない限り実行時例外です。

var map = Stream.of("apple", "ant", "banana")
    .collect(Collectors.toMap(
        s -> s.charAt(0),
        s -> s.length()
    ));
// 'a' が重複するため IllegalStateException
var map = Stream.of("apple", "ant", "banana")
    .collect(Collectors.toMap(
        s -> s.charAt(0),
        s -> s.length(),
        (oldValue, newValue) -> oldValue + newValue
    ));
// {a=8, b=6} のように集約

groupingByは値側を見る

groupingByは、分類関数だけなら Map<K, List<T>> です。下流Collectorを指定すると値側の型が変わります。

record Item(String category, int price) {}

var result = Stream.of(
        new Item("book", 1000),
        new Item("book", 1500),
        new Item("food", 500))
    .collect(Collectors.groupingBy(
        Item::category,
        Collectors.summingInt(Item::price)
    ));

// Map<String, Integer>
// {book=2500, food=500}

joiningはString専用ではなくCharSequence系

Collectors.joiningは、Streamの要素がCharSequence系のときに使います。数値Streamにそのまま使うのは不可能です。

String s = Stream.of("A", "B", "C")
    .collect(Collectors.joining(",", "[", "]"));
// [A,B,C]

String nums = IntStream.of(1, 2, 3)
    .mapToObj(String::valueOf)
    .collect(Collectors.joining("-"));
// 1-2-3

sortedとComparator

sorted()は自然順序で並べます。要素がComparableでない場合はコンパイルは通っても実行時にClassCastExceptionになることがあります。比較基準が必要ならComparatorを渡します。

record User(String name, int age) {}

var names = Stream.of(
        new User("A", 20),
        new User("B", 18))
    .sorted(Comparator.comparingInt(User::age).reversed())
    .map(User::name)
    .toList();
// [A, B]

forEachとforEachOrdered

逐次Streamでは同じに見えることが多いですが、parallel StreamではforEachの実行順は保証されません。順序が必要ならforEachOrderedです。

List.of(1, 2, 3, 4).parallelStream()
    .forEach(System.out::print); // 順序は保証されない

List.of(1, 2, 3, 4).parallelStream()
    .forEachOrdered(System.out::print); // encounter orderを維持

無限Streamはlimitなしだと終わらない

generateやiterateは無限Streamを作ります。limitや短絡終端操作がないと、処理が終わりません。

Stream.iterate(1, n -> n + 1)
    .limit(5)
    .forEach(System.out::print); // 12345

boolean found = Stream.iterate(1, n -> n + 1)
    .anyMatch(n -> n == 5); // trueで停止
無限Streamにsorted()のように全要素を必要とする操作を組み合わせると、結果が出ません。短絡するかどうかを必ず見ます。

Optionalを返す終端操作

findFirst、findAny、max、min、初期値なしreduceは、値が存在しない可能性があるためOptionalを返します。

Optional<String> first = Stream.<String>empty().findFirst();
System.out.println(first.orElse("none")); // none

String value = Stream.<String>empty()
    .findFirst()
    .get(); // NoSuchElementException

nullとStream

Streamはnull要素を絶対に持てないわけではありません。ただし、null要素に対してメソッド参照やインスタンスメソッドを呼ぶと実行時例外になります。

Stream.of("A", null, "B")
    .map(String::length)
    .forEach(System.out::println);
// null要素で NullPointerException
Stream.of("A", null, "B")
    .filter(Objects::nonNull)
    .map(String::length)
    .forEach(System.out::println);
// 1 1

不可能・誤りパターン

コード結果理由
stream.filter(s -> s.length())コンパイルエラーfilterはbooleanを返すPredicateが必要。
stream.map(s -> s.length()).sum()コンパイルエラーmap後はStream<Integer>。sumはない。mapToIntを使う。
stream.flatMap(s -> s.toUpperCase())コンパイルエラーflatMapのラムダはStreamを返す必要がある。
stream.forEach(...).count()コンパイルエラーforEachはvoidを返す終端操作。
stream.count(); stream.findFirst();実行時例外同じStreamを2回消費している。
Collectors.toMap(k, v)でキー重複実行時例外マージ関数なしで重複キーを処理できない。
Stream.generate(...).count()終わらない無限Streamを制限していない。
Optional.empty().get()実行時例外値がないOptionalから取得している。

読解ミニ演習

問題1

long n = Stream.of("a", "bb", "ccc")
    .filter(s -> s.length() > 1)
    .map(String::toUpperCase)
    .count();
System.out.println(n);

答え: 2。bbとcccだけが残ります。countの戻り値はlongです。

問題2

var r = Stream.of(1, 2, 3, 4)
    .filter(n -> n % 2 == 0)
    .findFirst();
System.out.println(r.get());

答え: 2。findFirstはOptional<Integer>を返します。この例では値が存在するためgetできます。

問題3

var result = Stream.of("A", "BB")
    .map(String::length)
    .reduce(10, Integer::sum);
System.out.println(result);

答え: 13。初期値10に、長さ1と2を加えます。

問題4

var result = Stream.of("A", "B", "AA")
    .collect(Collectors.groupingBy(String::length));
System.out.println(result.get(1).size());

答え: 2。長さ1の要素はAとBです。分類関数だけのgroupingByは、値側がListになります。

試験での最短判定手順

  1. まずStreamの要素型を書く。
  2. map、flatMap、mapToIntで型が変わる箇所に印を付ける。
  3. 終端操作を探す。終端操作がなければ基本的に処理は流れない。
  4. 終端操作の戻り値型を決める。
  5. anyMatch、findFirst、limitがあれば短絡を疑う。
  6. collectならCollectorの戻り値、特にMapのキー重複と値側の型を見る。
Stream APIは「何をしたいか」ではなく「各メソッドの戻り値型が次のメソッドに接続できるか」で読むべきです。雰囲気でチェーンを追うと、Goldの問題では高確率で落ちます。
