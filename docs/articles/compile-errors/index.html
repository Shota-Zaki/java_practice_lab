Java Silver 頻出

コンパイルエラーになる原因まとめ

Java Silverでは、出力を追わせるように見せて、実際にはコンパイルで止まる問題が多いです。ここでは頻出原因を「何がダメか」「どう直すか」に分けて確認します。

記事本文上部
まず疑う場所
  1. ローカル変数の未初期化
  2. static文脈からインスタンスメンバー参照
  3. アクセス修飾子の弱体化
  4. 戻り値型・引数・例外宣言の不一致
  5. 型変換・数値昇格・ジェネリクスの不一致
  6. 到達不能コード、catch順序、コンストラクタ規則

1. ローカル変数を初期化せずに使う

public class Main {
    public static void main(String[] args) {
        int x;
        System.out.println(x); // コンパイルエラー
    }
}

ローカル変数にはデフォルト値が入りません。フィールドの int は 0 になりますが、メソッド内のローカル変数は明示的な初期化が必要です。

int x = 0;
System.out.println(x);

2. staticメソッドからインスタンスフィールドを直接使う

public class Main {
    int count = 10;
    public static void main(String[] args) {
        System.out.println(count); // コンパイルエラー
    }
}

main は static なので、インスタンスなしで実行されます。インスタンスフィールドを使うならオブジェクトを作ります。

Main m = new Main();
System.out.println(m.count);

3. オーバーライドでアクセス修飾子を弱くする

class A {
    protected void test() {}
}
class B extends A {
    void test() {} // コンパイルエラー
}

オーバーライドでは、親より狭いアクセスにできません。protected をパッケージプライベートへ弱めているためエラーです。

class B extends A {
    protected void test() {}
}

4. 戻り値型だけ違うメソッドはオーバーロードにならない

class Sample {
    int calc() { return 1; }
    long calc() { return 2L; } // コンパイルエラー
}

メソッド識別では戻り値型だけでは区別できません。引数リストが同じなら重複定義です。

5. interface実装メソッドをpublicにしていない

interface Task {
    void run();
}
class Job implements Task {
    void run() {} // コンパイルエラー
}

interfaceの抽象メソッドは暗黙に public です。実装側も public が必要です。

class Job implements Task {
    public void run() {}
}

6. checked例外を処理していない

class Main {
    static void risky() throws Exception {}
    public static void main(String[] args) {
        risky(); // コンパイルエラー
    }
}

Exception はchecked例外です。try-catch か throws が必要です。

public static void main(String[] args) throws Exception {
    risky();
}

7. catchの順序が広い型から狭い型になっている

try {
    throw new java.io.IOException();
} catch (Exception e) {
} catch (java.io.IOException e) { // コンパイルエラー
}

先に Exception で捕まえると、後続の IOException は到達不能です。狭い型から広い型へ並べます。

8. 到達不能コード

while (true) {
    break;
    System.out.println("X"); // コンパイルエラー
}

break、return、throw の後で絶対に実行されない文は到達不能コードです。

9. varをフィールド・引数・戻り値に使う

class Sample {
    var value = 10; // コンパイルエラー
}

var はローカル変数の型推論だけで使えます。フィールド、メソッド引数、戻り値型には使えません。

10. varの初期化式がない、またはnullだけ

var a;        // コンパイルエラー
var b = null; // コンパイルエラー

var は初期化式から型を推論するため、型を決められない宣言は不可です。

11. byte / short / char の演算結果をそのまま代入する

byte a = 1;
byte b = 2;
byte c = a + b; // コンパイルエラー

byte や short は演算時に int へ昇格します。必要ならキャストします。

byte c = (byte)(a + b);

12. 通常代入と複合代入の違い

byte b = 1;
b = b + 1; // コンパイルエラー
b += 1;    // OK

b + 1 は int です。一方 += は暗黙のキャストを含みます。この差は頻出です。

13. コンストラクタのthis() / super() が先頭にない

class A {
    A(int x) {}
}
class B extends A {
    B() {
        System.out.println("B");
        super(1); // コンパイルエラー
    }
}

this(...) または super(...) は、コンストラクタ本体の先頭文でなければいけません。

14. 親クラスに引数なしコンストラクタがない

class A {
    A(int x) {}
}
class B extends A {
    B() {} // コンパイルエラー
}

明示しない場合、子コンストラクタの先頭に super() が補われます。しかし親に A() がないのでエラーです。

15. recordのコンポーネントは再代入できない

record Item(String name) {
    public void change() {
        name = "A"; // コンパイルエラー
    }
}

recordのコンポーネントは実質 private final フィールドです。アクセサは name() で、getName() は自動生成されません。

16. defaultメソッドの競合

interface A { default void test() {} }
interface B { default void test() {} }
class C implements A, B {} // コンパイルエラー

同じシグネチャのdefaultメソッドを複数継承すると、実装クラス側で明示的に解決する必要があります。

class C implements A, B {
    public void test() { A.super.test(); }
}

17. sealedの継承先が final / sealed / non-sealed でない

sealed class A permits B {}
class B extends A {} // コンパイルエラー

sealedクラスを直接継承するクラスは、final、sealed、non-sealed のいずれかで宣言する必要があります。

18. 配列の宣言・生成構文ミス

int array = new int[2]; // コンパイルエラー
int[] a = new int[];    // コンパイルエラー

配列変数は int[] 型です。また、new int[] だけでは要素数か初期化子がないため不可です。

試験での見抜き方

  • 出力候補を追う前に、変数宣言・初期化・アクセス修飾子を見る。
  • static、var、throws、catch、super() はコンパイルエラーの頻出箇所。
  • 問題文が「コンパイル、実行したとき」と書いてあっても、実行まで到達するとは限らない。
  • コードが長い問題ほど、出力追跡より先に構文・型・アクセス権を疑う。
