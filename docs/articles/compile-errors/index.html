Java Silver 頻出

Java Silverでコンパイルエラーになる原因まとめ

Java Silverの出力問題は、出力を追う前にコンパイルで止まることが多いです。このページでは、原因を「何がエラーか」「なぜエラーか」「どう直すか」「試験でどう紛れるか」まで分けて整理します。

1. ローカル変数の未初期化

public class Main {
    public static void main(String[] args) {
        int x;
        System.out.println(x); // コンパイルエラー
    }
}

ローカル変数にはデフォルト値が入りません。フィールドなら int は 0 ですが、メソッド内の変数は使う前に必ず代入が必要です。

int x = 0;
System.out.println(x);

試験での罠：同じ int x; でも、フィールドかローカル変数かで結果が変わります。

2. static文脈からインスタンスメンバーを直接参照

class Sample {
    int count = 10;
    static void print() {
        System.out.println(count); // コンパイルエラー
    }
}

static メソッドはインスタンスなしで呼べます。そのため、どのオブジェクトの count なのか決められません。

static void print() {
    Sample s = new Sample();
    System.out.println(s.count);
}

誤解：count が同じクラス内にあるから使える、は誤りです。同じクラスかどうかではなく、static文脈かどうかを見ます。

3. オーバーライドでアクセス修飾子を弱くする

class Parent {
    protected void run() {}
}
class Child extends Parent {
    void run() {} // コンパイルエラー
}

オーバーライドでは、親より狭いアクセス権にできません。protected をパッケージプライベートに弱めています。

class Child extends Parent {
    protected void run() {}
}

よくある選択肢：「privateにすれば隠蔽できる」は、この例では不可です。親のメソッドをオーバーライドしているためです。

4. 戻り値型だけ違うメソッド

class Sample {
    int calc() { return 1; }
    long calc() { return 2L; } // コンパイルエラー
}

オーバーロードは引数リストで区別します。戻り値型だけでは別メソッドになりません。

修正：引数を変える、または片方のメソッド名を変えます。

5. interface実装メソッドのpublic不足

interface Task {
    void execute();
}
class Job implements Task {
    void execute() {} // コンパイルエラー
}

interfaceの抽象メソッドは暗黙に public abstract です。実装側も public が必要です。

class Job implements Task {
    public void execute() {}
}

6. checked例外の未処理

class Sample {
    static void read() throws java.io.IOException {}
    public static void main(String[] args) {
        read(); // コンパイルエラー
    }
}

IOException はchecked例外です。呼び出し側で try-catch するか、throws で伝播させます。

public static void main(String[] args) throws java.io.IOException {
    read();
}

注意：RuntimeException 系は同じように書いてもコンパイルエラーにはなりません。

7. catch順序が広い型から狭い型

try {
    throw new java.io.IOException();
} catch (Exception e) {
} catch (java.io.IOException e) { // 到達不能
}

先に Exception で捕まえると、後ろの IOException には絶対に到達しません。狭い例外から広い例外へ並べます。

8. 到達不能コード

void test() {
    return;
    System.out.println("A"); // コンパイルエラー
}

return、throw、無条件の break の後にある文は到達不能になり得ます。

試験での罠：if (true) や while(false) も到達可能性の判定に絡みます。

9. varの使用不可箇所

class Sample {
    var value = 10; // フィールドでは不可
    void test(var x) {} // 引数でも不可
}

var はローカル変数の型推論専用です。フィールド、引数、戻り値型には使えません。

10. varの型を推論できない

var a;        // 初期化式がない
var b = null; // nullだけでは型が決まらない

var は右辺から型を決めます。右辺がない、または null だけでは型を決められません。

11. byte / short / char の数値昇格

byte a = 1;
byte b = 2;
byte c = a + b; // intになるので不可

byte 同士の足し算でも結果は int です。代入先が byte ならキャストが必要です。

byte c = (byte)(a + b);

12. 通常代入と複合代入

byte b = 1;
b = b + 1; // コンパイルエラー
b += 1;    // OK

b + 1 は int です。一方、+= は暗黙のキャストを含むため通ります。

注意：通るから安全という意味ではありません。値が範囲外なら丸められます。

13. this() / super() の位置

class A { A(int x) {} }
class B extends A {
    B() {
        System.out.println("B");
        super(1); // コンパイルエラー
    }
}

this(...) と super(...) はコンストラクタ本体の先頭文でなければいけません。

14. 親に引数なしコンストラクタがない

class A {
    A(int x) {}
}
class B extends A {
    B() {} // 暗黙のsuper()が入るが、A()がない
}

子コンストラクタで明示しないと super() が自動挿入されます。親に引数なしコンストラクタがない場合は、明示的に super(値) を書きます。

15. recordのフィールド再代入

record User(String name) {
    void change() {
        name = "A"; // コンパイルエラー
    }
}

recordコンポーネントは実質的にfinalです。値を変える設計ではなく、新しいrecordを作る設計です。

16. defaultメソッドの衝突

interface A { default void test() {} }
interface B { default void test() {} }
class C implements A, B {} // コンパイルエラー

同じシグネチャのdefaultメソッドを複数継承した場合、実装クラス側でどちらを使うか明示します。

class C implements A, B {
    public void test() { A.super.test(); }
}

17. sealedの直接サブクラス

sealed class A permits B {}
class B extends A {} // コンパイルエラー

sealedクラスの直接サブクラスは、final、sealed、non-sealed のいずれかで宣言します。

18. 配列生成の構文

int array = new int[2]; // 型が違う
int[] a = new int[];    // 長さも初期化子もない

配列変数は int[] 型です。また、生成時は長さか初期化子が必要です。

19. importでサブパッケージまで読めると思う

import com.example.*; // com.example.sub.Sample までは読めない

* はそのパッケージ直下だけです。サブパッケージは別パッケージです。

20. publicクラスとファイル名

// File: Main.java
public class Sample {} // コンパイルエラー

publicトップレベルクラスの名前とファイル名は一致する必要があります。

最後の確認

  • 出力候補を見る前に、変数・型・static・例外を確認する。
  • var、super()、catch、sealed はエラー判定の頻出箇所。
  • 長いコードほど、出力追跡より先に「そもそも通るか」を疑う。
