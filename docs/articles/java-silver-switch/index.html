switch文とswitch式

switchは、文と式の違い、フォールスルー、yield、caseラベルの型が頻出です。

1. switch文はフォールスルーする

int n = 1;
switch (n) {
    case 1: System.out.print("A");
    case 2: System.out.print("B"); break;
    default: System.out.print("C");
}

break がないため case 2 に流れ、出力は AB です。

2. switch式は値を返す

String result = switch (n) {
    case 1 -> "A";
    case 2 -> "B";
    default -> "C";
};

switch式は式なので、結果を変数へ代入できます。末尾のセミコロンも必要です。

3. yieldが必要なケース

String result = switch (n) {
    case 1: yield "A";
    default: yield "Z";
};

コロン形式のswitch式では、値を返すために yield を使います。

4. caseラベルは定数

int x = 1;
switch (n) {
    case x: break; // コンパイルエラー
}

case にはコンパイル時定数が必要です。final int x = 1; なら使える場合があります。

5. nullの扱い

String s = null;
switch (s) {
    case "A": System.out.println("A");
}

範囲では、null をswitch対象にすると実行時例外になる問題として出やすいです。

確認ポイント

  • break があるか。
  • 文なのか式なのか。
  • -> なのか : なのか。
  • 全ケースで値を返しているか。
  • case が定数か。

追加構文は「何が自動生成され、何が禁止されるか」で読む

構文確認すること落とし穴
recordコンポーネント、アクセサ、equals/hashCode/toStringアクセサはgetXではなくx()。
sealedpermitsとサブクラス側の修飾子継承許可された側もfinal/sealed/non-sealedが必要。
switch式値を返すか、全分岐が網羅されるかyieldが必要なブロック形式を見落とす。
instanceofパターン変数の有効範囲条件がfalseの分岐では使えない。

追加構文を読むときの共通視点

追加構文は、見た目の新しさではなく「何が自動生成されるか」「どこまで書けるか」「どのスコープで有効か」で判断します。

構文まず見ること典型ミス
recordコンポーネントとアクセサ名getX()だと思い込む。
sealedpermitsとサブクラス側の修飾子許可された側のfinal/sealed/non-sealedを忘れる。
switch式値を返すか、全分岐が値を出すかyieldが必要なブロックを見落とす。
pattern変数有効範囲false側や条件外で使おうとする。

よくある誤解と修正

誤解正しい見方確認方法
見た目が似ていれば同じ動きになる型、戻り値、参照、修飾子が1つ違うだけで結果は変わる。行ごとに「型」「値」「参照先」をメモする。
実行結果だけ追えばよいコンパイルできなければ実行されない。例外で止まれば後続行は動かない。コンパイル可否 → 実行時例外 → 出力の順に判定する。
暗記した用語で判断できる用語は入口にすぎない。コード上のどの行に効いているかまで見る。本文の例を1行変更し、結果が変わる理由を説明する。

ミニドリル

次の形式で、自分の答えを文章にしてください。正解だけを出しても、弱点は潰れません。

  1. このコードはコンパイルできるか。
  2. 実行時に例外が起きるなら、どの行で起きるか。
  3. 正常に動くなら、出力は何か。
  4. 自分が間違えた場合、原因は「型」「参照」「戻り値」「アクセス」「例外」「評価順」のどれか。
学習時は、選択肢を見る前に自分の判定を作ってください。選択肢を先に見ると、もっともらしい誤答に引っ張られます。
