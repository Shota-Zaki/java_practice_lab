module-info.java / Javaモジュール完全整理

Goldのモジュール問題は、module-info.javaの暗記だけでは足りません。classpathとmodule-pathの違い、requiresとexportsの関係、opensとリフレクション、サービス提供までを、コンパイル可否・実行可否の順で判定します。

最初に押さえる判定順

モジュール問題は、先に構成を確認します。コード本文だけを見て判断すると外します。

  1. 対象は通常のclasspath実行か、名前付きモジュールか。
  2. module-info.javaが存在するか。
  3. 依存先モジュールをrequiresしているか。
  4. 参照されるパッケージが依存先でexportsされているか。
  5. 通常アクセスの話か、リフレクションの話か。リフレクションならopensを見る。
  6. サービス利用なら、利用側のusesと提供側のprovides ... with ...を見る。
  7. コマンドが--class-pathではなく--module-pathになっているか。

モジュールは「パッケージの公開」と「依存関係」を宣言する単位

Javaのモジュールは、複数のパッケージをまとめた単位です。モジュールは、外部に公開するパッケージと、必要とする他モジュールをmodule-info.javaで宣言します。

module com.example.app {
    requires java.sql;
    exports com.example.app.api;
}
要素意味試験で見る点
module com.example.appモジュール名Java識別子をドットでつなぐ。ハイフンは不可。
requires java.sql依存モジュール使う側が依存を宣言する。
exports com.example.app.api外部公開パッケージ公開はクラス単位ではなくパッケージ単位。
public classであっても、そのパッケージがexportsされていなければ、他モジュールから通常アクセスできません。

module-info.javaの置き場所

module-info.javaは、そのモジュールのルートに置きます。通常のクラスと違い、package宣言を書きません。

src/
  com.example.app/
    module-info.java
    com/
      example/
        app/
          Main.java
// src/com.example.app/module-info.java
module com.example.app {
    exports com.example.app;
}

// src/com.example.app/com/example/app/Main.java
package com.example.app;

public class Main {
    public static void main(String[] args) {
        System.out.println("app");
    }
}
判断結果理由
module-info.javaにpackage宣言を書くコンパイルエラーモジュール宣言はパッケージ内の宣言ではない。
module-info.javaを通常パッケージの中に置く構成として不正モジュールルートに置く。
モジュール名とパッケージ名が同じ可能別概念。ただし混同しやすい。
モジュール名とパッケージ名が違う可能一致必須ではない。

モジュールのコンパイルと実行コマンド

名前付きモジュールは、通常のjavac Sample.javaやjava Sampleだけで判断しません。--module-source-path、--module-path、-mを読む必要があります。

# モジュール形式のソースをコンパイル
javac -d out --module-source-path src -m com.example.app

# モジュールとして実行
java --module-path out -m com.example.app/com.example.app.Main
オプション意味注意
-d outclassファイルの出力先モジュールごとに出力される。
--module-source-path srcモジュール形式のソース置き場src/モジュール名/...構成を前提にする。
-m com.example.appコンパイル対象モジュール--moduleの短縮形。
--module-path outモジュール探索先-p outとも書ける。
-m module/package.Main実行するモジュールとmainクラス通常のjava Mainではない。
悪い判断

java com.example.app.Mainで実行できると考える。

良い判断

名前付きモジュールとして実行するなら、java --module-path out -m com.example.app/com.example.app.Mainと読む。

複数モジュールの最小例

モジュールの基本は「使う側はrequires」「使われる側はexports」です。片方だけでは足りません。

src/
  com.example.lib/
    module-info.java
    com/example/lib/api/Message.java
    com/example/lib/internal/Secret.java
  com.example.app/
    module-info.java
    com/example/app/Main.java
// com.example.lib/module-info.java
module com.example.lib {
    exports com.example.lib.api;
}

// com.example.lib/com/example/lib/api/Message.java
package com.example.lib.api;
public class Message {
    public static String text() { return "hello"; }
}

// com.example.app/module-info.java
module com.example.app {
    requires com.example.lib;
}

// com.example.app/com/example/app/Main.java
package com.example.app;
import com.example.lib.api.Message;

public class Main {
    public static void main(String[] args) {
        System.out.println(Message.text());
    }
}
javac -d out --module-source-path src -m com.example.app
java --module-path out -m com.example.app/com.example.app.Main
この例では、com.example.appをコンパイル対象にすると、必要なcom.example.libも合わせて解決されます。

requiresは「読む権限」

requiresは、他のモジュールを読むための宣言です。モジュール内のコードが別モジュールの型を使うなら、基本的にrequiresが必要です。

module com.example.app {
    requires com.example.lib;
}
書き方意味注意
requires com.example.lib;実行時にも必要な依存標準形。
requires transitive com.example.api;依存を利用側へ伝播APIとして外部に見せる型を再公開する場合に使う。
requires static com.example.tool;コンパイル時だけ必要実行時には必須依存にしない。
requires java.base;基礎モジュール暗黙に依存するため、通常は書かない。

requires transitive

requires transitiveは、依存先を自分の利用者にも読ませる指定です。ライブラリの公開APIが別モジュールの型を露出している場合に必要になります。

// com.example.api
module com.example.api {
    exports com.example.api;
}

// com.example.lib
module com.example.lib {
    requires transitive com.example.api;
    exports com.example.lib;
}

// com.example.app
module com.example.app {
    requires com.example.lib;
    // com.example.api を直接 requires しなくても読める
}
試験では「appはapiを直接requiresしていないから不可」と単純判断すると誤りです。間にrequires transitiveがあるか確認します。

requires static

requires staticは、コンパイル時には必要だが、実行時の必須依存にはしない指定です。アノテーション用モジュールなどで使われます。

module com.example.app {
    requires static com.example.annotations;
}
指定コンパイル時実行時
requires必要必要
requires static必要必須ではない
requires transitive必要必要。さらに利用側へ伝播

exportsは「通常アクセスの公開」

exportsは、外部モジュールから通常のJavaコードとしてアクセスできるパッケージを指定します。クラス単位ではなく、パッケージ単位です。

module com.example.lib {
    exports com.example.lib.api;
    // com.example.lib.internal は公開していない
}
// OK: exported package
import com.example.lib.api.Message;

// NG: not exported package
// import com.example.lib.internal.Secret;
条件他モジュールからアクセス理由
パッケージがexportsされ、クラスがpublic可能公開パッケージ内のpublic型。
パッケージがexportsされていない不可publicクラスでも通常アクセス不可。
パッケージはexportsされているが、クラスがpackage-private不可通常のアクセス修飾子の制限を受ける。
exportsに存在しないパッケージを書くコンパイルエラー空または存在しないパッケージはexportできない。

限定exports

exports パッケージ to モジュール;と書くと、指定したモジュールにだけ公開できます。

module com.example.lib {
    exports com.example.lib.internal to com.example.app;
}
利用側アクセス理由
com.example.app可能toで指定されている。
com.example.other不可限定公開の対象外。

opensは「リフレクション向けに開く」

opensは通常のコンパイル時アクセスを許可するものではありません。リフレクションによる深いアクセスを許可するための指定です。

module com.example.lib {
    opens com.example.lib.model;
}
指定通常importリフレクション用途
exports p;可能publicメンバ中心通常API公開。
opens p;不可可能DI、JSON変換、ORMなど。
exports p; opens p;可能可能通常APIかつ反射利用。
悪い判断

opensされているからimportできると考える。

良い判断

通常コードから使うならexports、リフレクションならopensを見る。

限定opens

opens パッケージ to モジュール;と書くと、指定したモジュールにだけリフレクション用に開けます。

module com.example.lib {
    opens com.example.lib.model to com.fasterxml.jackson.databind;
}

この指定は、通常のimport可否ではなく、反射アクセスの可否に関係します。

open module

open moduleにすると、そのモジュール内の全パッケージをリフレクション向けに開きます。ただし、通常アクセスを公開するには、やはりexportsが必要です。

open module com.example.app {
    exports com.example.app.api;
}
コード結果理由
open module m { exports p; }可能全体を反射用に開きつつ、通常APIを公開。
open module m { opens p; }コンパイルエラーopen module内では個別opensは不要であり、許可されない。
open moduleだけで通常import不可exportsがないと通常アクセスは公開されない。

exportsとopensの違いをコードで読む

次のモジュールでは、com.example.lib.apiは通常アクセス可能ですが、com.example.lib.modelは通常アクセスできません。

module com.example.lib {
    exports com.example.lib.api;
    opens com.example.lib.model;
}
// OK
import com.example.lib.api.PublicApi;

// NG: opensされていてもexportsされていない
// import com.example.lib.model.User;

services: uses / provides ... with ...

サービスローダーを使う場合、利用側はuses、提供側はprovides ... with ...を宣言します。実装クラスを直接newする設計とは違います。

// service API module
module com.example.plugin.api {
    exports com.example.plugin.api;
}

// provider module
module com.example.plugin.impl {
    requires com.example.plugin.api;
    provides com.example.plugin.api.Plugin
        with com.example.plugin.impl.PluginImpl;
}

// app module
module com.example.app {
    requires com.example.plugin.api;
    uses com.example.plugin.api.Plugin;
}
ServiceLoader<Plugin> loader = ServiceLoader.load(Plugin.class);
for (Plugin p : loader) {
    p.run();
}
宣言書く場所意味
uses ServiceType;サービスを使うモジュールServiceLoaderで探すサービス型を宣言。
provides ServiceType with Impl;実装を提供するモジュールサービス型と実装クラスを対応付ける。
exportsAPIモジュールサービス型そのものを外部から見えるようにする。
実装クラスのパッケージは必ずしもexportsする必要はありません。サービス経由で見せるなら、外部に直接公開しない設計が可能です。

classpathとmodule-pathの違い

従来の-cp / --class-pathはクラスを探す場所です。モジュールとして解決するには--module-pathを使います。

指定対象特徴
--class-path / -cp従来のクラスパス名前付きモジュールとしては扱わない。
--module-path / -pモジュール探索先モジュール記述子を使って解決する。
--module / -m実行モジュール指定module/mainClass形式で指定できる。
# classpath実行
java -cp out com.example.app.Main

# module-path実行
java --module-path out -m com.example.app/com.example.app.Main

名前付きモジュール・無名モジュール・自動モジュール

Goldでは、すべてが明示的なmodule-info.javaを持つとは限りません。クラスパス上のコード、モジュールパス上の通常JARも区別します。

種類条件特徴
名前付きモジュールmodule-info.classを持つrequires、exportsなどで制御される。
無名モジュールクラスパス上のコード従来型。名前付きモジュールからrequiresできない。
自動モジュールモジュールパス上にある通常JAR自動的にモジュール名が与えられ、パッケージが公開される。
名前付きモジュールから、クラスパス上の無名モジュールをrequiresすることはできません。ここは実務でも試験でも混乱しやすい点です。

モジュールJARの作成と実行

コンパイルしたモジュールはJAR化できます。モジュールJARは--module-pathに置いて実行します。

javac -d out --module-source-path src -m com.example.app

jar --create --file mods/com.example.app.jar \
    -C out/com.example.app .

java --module-path mods \
    -m com.example.app/com.example.app.Main

jar --describe-module --file mods/com.example.app.jarを使うと、JARのモジュール記述を確認できます。

標準モジュールの確認コマンド

JDKに含まれるモジュールはjava --list-modulesで確認できます。依存関係の調査にはjdepsも使います。

java --list-modules
java --describe-module java.sql
jdeps --module-path mods --module com.example.app
コマンド用途
java --list-modules利用可能なモジュール一覧を見る。
java --describe-module モジュール名モジュールのrequiresやexportsを見る。
jdepsクラスやJARの依存関係を調べる。

コンパイルエラーになりやすいパターン

パターン結果理由
module-info.javaにpackage宣言コンパイルエラーモジュール宣言はパッケージ宣言しない。
module com.example-app {}コンパイルエラーモジュール名にハイフンは使えない。
exports com.example.Memo;コンパイルエラーexportsはクラス名ではなくパッケージ名。
存在しないパッケージをexportsコンパイルエラー空または存在しないパッケージは公開できない。
requiresなしで他モジュールの型をimportコンパイルエラー依存モジュールを読めない。
依存先がexportsしていないパッケージをimportコンパイルエラーモジュール外へ公開されていない。
opensだけのパッケージを通常importコンパイルエラーopensは通常アクセスの公開ではない。
open module m { opens p; }コンパイルエラーopen moduleでは個別opensを指定できない。
同じパッケージを複数の名前付きモジュールに置くエラー原因分割パッケージは避ける。
名前付きモジュールからclasspath上の無名モジュールをrequires不可無名モジュールにはrequiresで指定できる名前がない。

実行時に失敗しやすいパターン

コマンド・状況結果理由
java -cp out -m com.example.app/...失敗名前付きモジュールは--module-pathで探す。
java --module-path out com.example.app.Main失敗モジュール実行なら-mが必要。
java --module-path out -m com.example.app失敗する場合ありメインクラスがモジュールに設定されていなければ起動対象が不明。
必要な依存モジュールをmodule-pathに置かない起動失敗モジュール解決できない。
サービス提供モジュールをmodule-pathに置かないサービスが見つからないprovides側が解決対象にない。

読解ミニ演習

演習1: publicだがexportsされていない

// module lib
module lib {
}

// lib/internal/Tool.java
package internal;
public class Tool {}

判定: 他モジュールからinternal.Toolを通常importすることはできません。publicでも、パッケージがexportsされていないからです。

演習2: opensだけのパッケージ

module lib {
    opens lib.model;
}

判定: lib.modelのクラスを他モジュールから通常importすることはできません。opensはリフレクション向けです。

演習3: requires transitive

module api { exports api; }
module lib { requires transitive api; exports lib; }
module app { requires lib; }

判定: appはapiを直接requiresしていなくても、lib経由でapiを読めます。requires transitiveがあるためです。

演習4: open module内のopens

open module app {
    opens app.model;
}

判定: コンパイルエラーです。open module内では、個別のopens指定はできません。

演習5: モジュール実行コマンド

java --module-path out -m com.example.app/com.example.app.Main

判定: outからcom.example.appモジュールを探し、その中のcom.example.app.Mainを起動します。-mの左側はモジュール名、右側は完全修飾クラス名です。

試験での判断基準

  • 通常アクセスはrequiresとexportsの両方を見る。
  • exportsはパッケージ単位。クラス単位ではない。
  • opensは通常アクセスではなくリフレクション向け。
  • open moduleは全パッケージを反射用に開くが、通常アクセスにはexportsが必要。
  • requires transitiveがあると依存が利用側へ伝播する。
  • module-pathとclasspathを混同しない。
  • 名前付きモジュールから無名モジュールをrequiresできない。
  • サービスはusesとprovides ... with ...を対応させて読む。
モジュール問題で一番危険なのは「publicなら使える」というSilverの感覚をそのまま持ち込むことです。Goldでは、publicに加えて、モジュールがそのパッケージを公開しているかを必ず確認します。
