出力結果問題の読み方

頻出の出力問題を、どう分解して読むかをまとめます。

この単元のゴール

出力追跡は「なんとなく頭で読む」単元ではありません。問題では、1文字違い・1行違い・例外発生位置の違いで選択肢が分かれます。紙やメモに 変数表・参照表・制御フロー表 を作って、実行順に潰してください

  1. まずコンパイル可否: 文法、型、アクセス、例外処理、到達不能文を先に見る。
  2. 次に実行時例外: null、配列/List範囲外、キャスト、parse、算術例外を探す。
  3. 最後に出力: 評価順、参照の共有、メソッド呼び出し、finallyを表にする。
重要: コンパイルエラーのコードは実行されません。例外が発生するコードは、例外発生前の出力だけ残ります。この2つを混ぜると、出力問題でほぼ確実に落とします。

出力追跡の基本手順

順番見るもの切れる選択肢
1クラス・メソッド・変数宣言・型コンパイルエラー
2実行開始位置、起動引数、初期値mainに入れるか、argsで落ちないか
3式の評価順数値結果、文字列連結結果
4参照の向きとオブジェクトの中身同じ実体を見ているか、別物か
5分岐・ループ・例外処理表示される/されない行
6println / print / return / throw改行、途中終了、finally実行

選択肢を見ながら解くと引っ張られます。まず自分で「コンパイル可否」「例外発生位置」「出力文字列」を確定し、その後で選択肢に当てはめる方が安定します。

変数表の作り方

数値・boolean・String結合の問題は、行番号ごとに変数の値を書きます。式の途中で値が変わる ++/-- は、1つの式をさらに分解してください。

int a = 10;
int b = a++ + ++a + a--;
System.out.println(a + ":" + b);
処理aの状態式に使う値説明
開始10-初期値
a++1110後置。先に10を使い、その後11になる
++a1212前置。先に12にしてから使う
a--1112後置。12を使った後、11になる
代入1134b = 10 + 12 + 12

出力は 11:34 です。式を左から読むだけでは不十分です。前置/後置で「値を使うタイミング」が違います。

String結合の追跡

+ は左から評価されます。片方がStringになると、以降は文字列結合として進みます。

int a = 10;
int b = 20;
System.out.println(a + b + "A" + a + b);
System.out.println("A" + a + b + (a + b));
式評価出力
a + b + "A" + a + b30 + "A" + 10 + 2030A1020
"A" + a + b + (a + b)"A10" + 20 + 30A102030
判断: 文字列結合で迷ったら、左から順に「今の結果の型は数値かStringか」を書いてください。括弧の中だけは先に評価します。

参照表の作り方

参照型の問題では、変数の中にオブジェクトそのものが入るわけではありません。変数はオブジェクトへの参照を持ちます。代入はオブジェクトのコピーではなく、参照のコピーです。

class Item { String name; }

Item a = new Item();
Item b = new Item();
a.name = "apple";
b.name = "banana";
a = b;
a.name = "cherry";
System.out.println(b.name);
行abItem#1Item#2
a = new Item()Item#1-name=null-
b = new Item()Item#1Item#2name=nullname=null
a.name = "apple"Item#1Item#2applenull
b.name = "banana"Item#1Item#2applebanana
a = bItem#2Item#2applebanana
a.name = "cherry"Item#2Item#2applecherry

出力は cherry です。a = b 後は、aとbが同じItem#2を見ています。

StringBuilderは「中身が変わる」

String は不変ですが、StringBuilder は可変です。同じインスタンスを複数変数で参照していると、片方からの変更がもう片方にも見えます。

StringBuilder a = new StringBuilder("A");
StringBuilder b = a;
a.append("B");
b.append("C");
System.out.println(a);

a.append("B")SB#1SB#1AB
b.append("C")SB#1SB#1ABC

出力は ABC です。append は新しいStringBuilderを返しているのではなく、基本的に自分自身を変更します。

メソッド呼び出しの追跡

Javaは常に値渡しです。参照型引数でも、渡されるのは「参照値のコピー」です。メソッド内でオブジェクトの中身を変えることと、仮引数へ別の参照を代入することは別です。

static void test(StringBuilder sb) {
    sb.append("B");
    sb = new StringBuilder("C");
    sb.append("D");
}

StringBuilder x = new StringBuilder("A");
test(x);
System.out.println(x);
処理xsb見える中身
呼び出し直後SB#1SB#1SB#1 = A
sb.append("B")SB#1SB#1SB#1 = AB
sb = new StringBuilder("C")SB#1SB#2SB#1 = AB / SB#2 = C
sb.append("D")SB#1SB#2SB#1 = AB / SB#2 = CD

呼び出し元の x はSB#1のままなので、出力は AB です。

フィールドとメソッド呼び出しの違い

継承問題では、フィールドは宣言型、オーバーライドされたインスタンスメソッドは実体 で決まります。ここは混同しやすいので表で分けます。

class A {
    String name = "A";
    String getName() { return name; }
}
class B extends A {
    String name = "B";
    @Override String getName() { return name; }
}

A obj = new B();
System.out.println(obj.name + ":" + obj.getName());
式決まり方結果
obj.name変数の宣言型 A のフィールドA
obj.getName()実体 B のオーバーライドメソッドB

出力は A:B です。メソッドとフィールドを同じルールで見てはいけません。

例外処理の出力表

例外が発生したら、tryブロックの残りは実行されません。対応するcatchへ移り、その後finallyがあれば実行します。

try {
    System.out.print("A");
    String s = null;
    System.out.print(s.length());
    System.out.print("B");
} catch (NullPointerException e) {
    System.out.print("C");
} finally {
    System.out.print("D");
}
System.out.print("E");
処理出力制御
print("A")A続行
s.length()-NullPointerException 発生
print("B")-実行されない
catchCfinallyへ
finallyDtry文の後へ
後続処理E終了

出力は ACDE です。catchで処理された例外は、通常はその後の処理へ進みます。

try-with-resourcesの追跡

リソース付きtryでは、リソースの close() は宣言と逆順に呼ばれます。finallyがある場合、closeの後にfinallyです。

class R implements AutoCloseable {
    private String name;
    R(String name) { this.name = name; }
    public void close() { System.out.print(name); }
}

try (R a = new R("A"); R b = new R("B")) {
    System.out.print("C");
} finally {
    System.out.print("D");
}
順番処理出力
1try本体C
2b.close()B
3a.close()A
4finallyD

出力は CBAD です。宣言順ではなく、逆順です。

ループ追跡の型

ループは「初期化 → 条件 → 本体 → 更新」を表にします。continue がある場合、通常forでは更新式へ進む点に注意します。

int sum = 0;
for (int i = 0; i < 4; i++) {
    if (i == 2) continue;
    sum += i;
}
System.out.println(sum);
i条件本体sum更新
0truesum += 00i=1
1truesum += 11i=2
2truecontinue1i=3
3truesum += 34i=4
4false終了4-

出力は 4 です。

よくある誤答パターン

コンパイルエラーを見落とす

未初期化ローカル変数、戻り値不足、到達不能catch、型不一致があると、出力追跡以前に終了

参照代入をコピーだと思う

a = b はオブジェクト複製ではありません。同じ参照先になるだけ

後置++を先に増やす

a++ は「式には元の値を使い、その後増える」です。値の使用と変数更新を分けます

例外後のtry残りを実行する

例外発生後、同じtryブロックの残りは実行されません。catch/finallyへ移ります

ミニ演習

Q1. String s = null; System.out.println("A" + s); は例外？

例外ではありません。文字列結合では null が文字列 "null" として扱われ、 Anull と表示されます。 s.length() のようにメソッドを呼ぶと例外

Q2. System.out.print("A"); throw new RuntimeException(); の後に例外が出たらAは消える？

消えません。例外発生前の出力は残ります。ただし、その後にcatchされなければプログラムは異常終了します

Q3. A obj = new B(); obj.field と obj.method() は同じ決まり方？

違います。フィールドは宣言型、オーバーライドされたインスタンスメソッドは実体で決まります

この記事の目的

Javaの問題は、知識量よりも「判定順」で落ちることが多いです。特に出力問題では、値を追い始める前にコンパイルエラーを見落とすと、その時点で負けです。このページでは、どの問題でも同じ順番で確認できるように、コンパイル可否、実行時例外、出力結果の3段階に分けて整理します

  • 第1段階: そもそもコンパイルできるか。
  • 第2段階: コンパイルできるなら、実行中に例外で止まるか。
  • 第3段階: 例外で止まらないなら、最終的に何が表示されるか。

この順番を崩すと、難問ではなく普通の問題でも誤答します。逆に、この順番を固定できれば、複雑なコードでも見る場所がかなり限定されます

全体フロー

最初に次の表の順番で確認します。これは毎回使う基本型

順番確認対象見るポイント典型的な落とし穴
1設問と選択肢何を答える問題かを先に見る出力を聞かれていると思い込み、コンパイルエラー選択肢を見落とす
2ソース構造package、import、publicクラス名、トップレベルクラス修飾子packageより前にimportを書く、トップレベルにprivate/protected/staticを付ける
3mainとメソッド宣言mainの形、戻り値、引数、static、呼び出し先の存在mainに見えるが実行エントリポイントではない
4変数宣言とスコープ初期化、varの使用場所、同名変数、ローカル変数の有効範囲ローカル変数未初期化、varをフィールドや引数に使う
5型と代入互換性、暗黙変換、キャスト、数値昇格、戻り値型byte/short演算のint化、longをintへ代入、doubleをfloatへ代入
6クラス・継承・interfaceconstructor、this/super、override、abstract、default methodoverride条件違反、abstract実装漏れ、constructorを継承すると誤解する
7例外処理の構文checked例外、catch順序、try-catch-finallyの並び親例外を先にcatch、checked例外の未処理、finallyとcatchの順序違反
8実行時例外null、配列範囲外、キャスト失敗、数値変換失敗、0除算コンパイルは通るが途中で止まる
9出力追跡評価順、副作用、参照、短絡評価、動的ディスパッチ++/--、+=、String結合、method override、field隠蔽

重要なのは、上から順に「脱落条件」を潰すことです。出力を追うのは最後

ステップ1: 選択肢から問題の型を読む

コードを見る前に、選択肢選択肢はヒントです。何を検証すべきかがほぼ書かれています

選択肢の形最初に疑うもの理由
「コンパイルエラー」がある宣言、型、override、例外処理Javaでは実行前に落ちる問題が多い
「実行時例外」があるnull、配列、キャスト、parse、0除算文法は正しいが実行途中で止まる可能性がある
出力候補だけ副作用、分岐、ループ、参照コンパイル可否の確認後、値を追う
「何も表示されない」がある条件分岐、ループ未実行、例外前停止printlnに到達しているかを見る
複数選択正しい説明と誤り説明を個別判定1つ正しいから終わりではない

ここを雑にやると、問題が求めていない方向に時間を使います。先に「この問題は何を落とし穴にしているか」を推測してください

ステップ2: ソース構造で落とす

最初に見るのは、プログラムの骨格です。ここで落ちるなら、値の追跡は不要

packageとimportの順序

// OK
package sample;
import java.util.*;
public class Main { }

// NG
import java.util.*;
package sample;
public class Main { }

package 宣言は、コメントを除いてソースファイルの先頭に置きます。 import はその後です。順番が逆ならコンパイルエラー

publicクラスとファイル名

// Sample.java に保存されている場合
public class Sample { }
class Helper { }

トップレベルの public クラスがある場合、ソースファイル名はそのクラス名と一致する必要があります。 public ではないトップレベルクラスは、同じファイルに複数書けます

トップレベルクラスの修飾子

修飾子トップレベルで使えるか注意
public使えるファイル名との一致が必要
final使える継承不可
abstract使える直接インスタンス化不可
sealed使える許可するサブクラスの指定が必要
non-sealed条件付きで使えるsealed型を継承・実装したクラスで使う
private / protected使えないトップレベルでは不可。ネストしたクラスなら別
static使えないトップレベルでは不可。static nested classなら可

ステップ3: mainと呼び出し口を確認する

実行問題では、実行される main が正しい形か

// 実行エントリポイントとして有効
public static void main(String[] args) { }
public static void main(String... args) { }

// mainという名前でも、実行エントリポイントではない
public void main(String[] args) { }       // staticがない
public static int main(String[] args) { } // 戻り値がvoidではない
static void main(String[] args) { }       // publicではない

問題では「mainっぽい」形を混ぜてきます。コードの中で呼び出される普通のメソッドとしては成立しても、 java コマンドで実行する入口として成立しない場合があります

ただし、問題文で「このメソッドを呼び出したとき」と書かれている場合は、mainの形ではなく、その呼び出し自体が可能かを見ます。設問の前提を読み落とさないでください

ステップ4: 変数宣言・var・初期化を確認する

Javaでは、 var とローカル変数の初期化が頻出です。ここはかなり機械的に切れます

ローカル変数は自動初期化されない

public class Main {
    public static void main(String[] args) {
        int x;
        System.out.println(x); // コンパイルエラー
    }
}

フィールドはデフォルト値で初期化されますが、ローカル変数は使用前に明示的な初期化が必要です。ここを混同すると落ちます

varが使える場所

場所var使用例
ローカル変数可var s = "abc";
拡張forの変数可for (var x : list)
フィールド不可private var value;
メソッド引数不可void test(var x)
戻り値型不可var test()
初期値なし不可var x;
nullのみ不可var x = null;
配列初期化子だけ不可var a = {1, 2, 3};

var は「型を書かなくてよい魔法」ではありません。コンパイラが右辺から型を一意に推論できるローカル変数宣言だけ

ステップ5: 型・代入・演算で落とす

出力を追う前に、代入できるか特に小さい整数型、浮動小数点、booleanは狙われます

整数リテラルと範囲

byte a = 127;  // OK
byte b = 128;  // コンパイルエラー
short c = 300; // OK
char d = 65;   // OK。'A'に相当

byte は -128〜127、 short は -32768〜32767、 char は 0〜65535 です。整数定数を代入する場合、範囲内なら小さい型にも代入できます

演算するとintになりやすい

byte a = 10;
byte b = 20;
byte c = a + b;      // コンパイルエラー。a + b は int
byte d = (byte)(a+b); // OK

byte 、 short 、 char は、算術演算時に基本的に int に昇格します。変数同士の演算結果をそのまま小さい型に戻すことはできません

long・float・doubleの基本

int a = 2 * 3L;   // コンパイルエラー。結果はlong
long b = 2 * 3L;  // OK
float c = 10.0;   // コンパイルエラー。10.0はdouble
float d = 10.0F;  // OK

整数リテラルは基本 int 、浮動小数点リテラルは基本 double です。 L 、 F の有無で型が変わります

booleanと数値は別物

boolean b = true;
if (b) { }
if (b == true) { }
if (1) { }       // コンパイルエラー
boolean x = 1;   // コンパイルエラー

Javaでは、 0 を false、 1 を true として扱いません。C言語系の感覚を持ち込むと間違えます

ステップ6: クラス・継承・interfaceで落とす

継承問題では、見た目の出力より先に「そもそも継承・実装として成立しているか」

コンストラクタは継承されない

class A {
    A(String s) { }
}
class B extends A { }
// Bは暗黙に super(); を呼ぼうとするが、Aに無引数コンストラクタがないためコンパイルエラー

サブクラスはスーパークラスのコンストラクタを引き継ぎません。サブクラスのコンストラクタでは、最初に super(...) または this(...) が呼ばれます。明示しない場合は super() が補われます

overrideの条件

項目条件違反例
メソッド名・引数同じシグネチャ引数型が違うならoverloadでありoverrideではない
戻り値型同じ型または共変戻り値サブクラス側でより広い型を返す
アクセス修飾子同じか、より広いpublic を protected に狭める
例外checked例外を広げない親にない広いchecked例外をthrowsする
staticstaticメソッドはoverrideではなく隠蔽インスタンスメソッドとstaticを入れ替える

interfaceの基本

  • インタフェースの抽象メソッドは暗黙に public abstract。
  • インタフェースのフィールドは暗黙に public static final。
  • default メソッドは実装を持てる。
  • private メソッドはインタフェース内の共通処理として使えるが、実装クラスから呼び出すものではない。
  • クラスが抽象クラスでないなら、未実装の抽象メソッドをすべて実装する必要がある。

ステップ7: 例外処理の構文で落とす

例外は、コンパイル時に落ちるものと実行時に落ちるものを分けます。ここを混ぜると判断が不安定になります

checked例外は処理または宣言が必要

import java.io.IOException;

class A {
    void test() throws IOException { }
    void call() {
        test(); // コンパイルエラー。IOExceptionを処理していない
    }
}

Exception 系でも、 RuntimeException のサブクラスは非チェック例外です。 IOException などのチェック例外は、 try-catch または throws が必要

catchは狭い例外から広い例外へ

try {
    // any code
} catch (Exception e) {
} catch (NullPointerException e) { // コンパイルエラー。到達不能
}

親クラスの例外を先にcatchすると、子クラスのcatchに到達できません。catchは具体的な例外から順に書きます

try-catch-finallyの並び

// OK
try { } catch (Exception e) { } finally { }
try { } finally { }

// NG
try { } finally { } catch (Exception e) { }

finally の後に catch は書けません。構文順序で落とせる問題

ステップ8: 実行時例外を確認する

コンパイルできると判断したら、次は実行時例外です。例外が発生した時点で、通常の後続処理は実行されません

例外典型例判定ポイント
NullPointerExceptions.length() where s == nullnull参照に対してインスタンスメソッドやフィールドを使う
ArrayIndexOutOfBoundsExceptionarr[3] where length is 3配列の添字は0からlength-1
StringIndexOutOfBoundsExceptions.charAt(5)Stringの範囲外アクセス
ClassCastException(B) a where 実体がBではない参照型のキャストはコンパイル可能でも実体が合わないと失敗
NumberFormatExceptionInteger.parseInt("abc")文字列を数値に変換できない
ArithmeticException10 / 0整数の0除算。浮動小数点の0除算とは挙動が違う
 
try {
    String s = null;
    System.out.println(s.length());
    System.out.println("A");
} catch (NullPointerException e) {
    System.out.println("B");
} finally {
    System.out.println("C");
}

s.length() で NullPointerException が発生します。そのため A は表示されません。対応するcatchで B 、最後にfinallyで C が表示されます

ステップ9: 出力追跡に入る

コンパイルも実行時例外も問題ないと判断してから、初めて値を追います。ここでは、評価順と副作用が中心

++/--は表にする

int a = 10;
int b = a++ + a + --a;
System.out.println(a + "," + b);
式の部分式に使われる値処理後のa理由
a++1011後置なので、元の値を使ってから増える
a1111現在値を使う
--a1010前置なので、先に減ってから使う

したがって b は 10 + 11 + 10 = 31 、最後の a は 10

短絡評価

int a = 10;
int b = 10;
if (a > 10 && ++b > 10) {
    a++;
}
System.out.println(a + b);

&& は左が false なら右を評価しません。 a > 10 は false なので ++b は実行されず、出力は 20 です。 & なら右も評価されます

String結合は左から効く

System.out.println(1 + 2 + "A" + 3 + 4); // 3A34

左から評価され、途中でStringになると以降は文字列結合です。 1 + 2 は数値加算で 3 、そこから "A" と結合して "3A" 、以降は文字列結合

出力追跡で必ず分けるもの

出力問題で混同しやすいのは、参照型、フィールド、メソッド、static

対象決まり方例
インスタンスメソッド実体のクラスで動的に決まるoverrideされたメソッドは実体側が呼ばれる
フィールド変数の宣言型で決まる同名フィールドはoverrideではなく隠蔽
staticメソッド基本的に参照型・クラス名で決まるoverrideではなく隠蔽
overloadコンパイル時の引数型で決まる実体ではなく、呼び出し時に見えている型が重要
override実行時の実体で決まるポリモーフィズムの中心
 
class A {
    String name = "A";
    String getName() { return name; }
}
class B extends A {
    String name = "B";
    String getName() { return name; }
}
A obj = new B();
System.out.println(obj.name);     // A
System.out.println(obj.getName()); // B

obj.name はフィールドなので変数の型 A で決まります。 obj.getName() はインスタンスメソッドなので実体 B のメソッドが呼ばれます

実戦テンプレート

問題を解くときは、次の順番をそのまま使ってください

  1. 選択肢を見る。 コンパイルエラー・実行時例外・出力のどれを判定する問題かを決める。
  2. ソース構造を見る。 package/import、publicクラス名、トップレベル修飾子を確認する。
  3. 宣言を見る。 main、メソッド、戻り値、引数、static、var、ローカル変数初期化を見る。
  4. 型を見る。 代入、数値昇格、キャスト、戻り値、boolean互換性を見る。
  5. クラス関係を見る。 constructor、super/this、override、abstract/interface、アクセス修飾子を見る。
  6. 例外処理を見る。 checked例外、catch順序、finallyの位置を見る。
  7. ここまで通ったら実行する。 null、範囲外、キャスト失敗、parse失敗、0除算を見る。
  8. 最後に出力を追う。 ++/--、短絡評価、String結合、参照、メソッド呼び出しを表にする。

この順番を守るだけで、無駄なトレースが減ります。逆に、この順番を守れないうちは、知識が増えても点数は安定しません

例題1: コンパイルエラーを先に切る

public class Main {
    public static void main(String[] args) {
        int a;
        if (args.length > 0) {
            a = 10;
        }
        System.out.println(a);
    }
}

この問題で args.length の値を想像してはいけません。コンパイラは、 if に入らない可能性があることを見ます。その場合、 a は未初期化のまま使われます。したがってコンパイルエラー

「実行時に引数を渡せば大丈夫」という考え方は不正確です。Javaのコンパイル時チェックは、実行時の都合ではなく、コード上で確実に初期化されるかを見ます

例題2: コンパイル後に実行時例外を切る

public class Main {
    public static void main(String[] args) {
        String[] data = {"A", "B"};
        System.out.println(data[2]);
        System.out.println("end");
    }
}

配列の型、初期化、 main の形に問題はありません。したがってコンパイルは通ります

次に実行します。配列 data の要素数は2なので、使える添字は 0 と 1 だけです。 data[2] は範囲外なので、 ArrayIndexOutOfBoundsException が発生します。 end は表示されません

例題3: 出力追跡は最後にする

public class Main {
    public static void main(String[] args) {
        int a = 1;
        int b = 2;
        System.out.println("" + a + b + (a + b));
    }
}

宣言、型、例外に問題はありません。ここで初めて出力を追います

  • "" + a で文字列結合になり、"1"。
  • 次の + b も文字列結合なので、"12"。
  • (a + b) は括弧内で先に数値加算され、3。
  • 結果は "123"。

出力は 123

この単元の到達基準

このページを読んだだけで終わらせると弱いです。次の状態になっていなければ、まだ使える知識になっていません

  • 選択肢を見て、コンパイル問題か実行時例外問題か出力問題かを分類できる。
  • 出力追跡の前に、最低限のコンパイルチェックを行える。
  • var、ローカル変数未初期化、byte/short演算、override条件を先に確認できる。
  • 例外が出た行以降の通常処理を追わない判断ができる。
  • ++/--、短絡評価、String結合、参照型の問題を表で追える。

次に深掘りするなら、得点効率が高いのは「型・変数・リテラル」または「数値・型昇格」
