Java Silverの実行時例外まとめ

実行時例外は、コンパイルは通るが実行中に止まる問題です。Java Silverでは「コンパイルエラー」「実行時例外」「正常出力」を混ぜて出されるため、例外の発生タイミングを正確に読む必要があります。

頻出の実行時例外
  • NullPointerException
  • ArrayIndexOutOfBoundsException
  • ClassCastException
  • NumberFormatException
  • ArithmeticException
  • UnsupportedOperationException

1. NullPointerException

String s = null;
System.out.println(s.length()); // 実行時例外

参照変数が null の状態でインスタンスメソッドやフィールドにアクセスすると発生します。

例外にならない例：System.out.println(s); は文字列 null を表示するだけです。

2. 配列の範囲外アクセス

int[] a = {10, 20};
System.out.println(a[2]); // 実行時例外

配列の添字は0から始まります。要素数2なら有効な添字は 0 と 1 です。

3. ClassCastException

Object obj = "Java";
Integer n = (Integer) obj; // 実行時例外

コンパイル上はキャスト可能でも、実体がその型でなければ実行時に失敗します。

if (obj instanceof Integer n) {
    System.out.println(n + 1);
}

4. NumberFormatException

int n = Integer.parseInt("10A"); // 実行時例外

文字列が整数として解釈できない場合に発生します。コマンドライン引数問題で頻出です。

5. ArithmeticException

int x = 10 / 0; // 実行時例外

整数のゼロ除算は例外です。一方、浮動小数点数では Infinity になるケースがあります。

System.out.println(10.0 / 0); // Infinity

6. UnsupportedOperationException

List<String> list = java.util.List.of("A", "B");
list.add("C"); // 実行時例外

List.of() の戻り値は変更不可です。add、remove、set などが失敗します。

7. Arrays.asListの罠

List<String> list = java.util.Arrays.asList("A", "B");
list.set(0, "X"); // OK
list.add("C");    // 実行時例外

Arrays.asList は固定長リストです。要素の置換はできますが、サイズ変更はできません。

8. StringBuilderと範囲指定

StringBuilder sb = new StringBuilder("abc");
sb.delete(1, 10); // 実行時例外

APIによって範囲外の扱いが異なります。String.substring や StringBuilder の操作は、開始・終了位置に注意します。

9. try-catchで捕まる型

try {
    int[] a = new int[1];
    System.out.println(a[3]);
} catch (NullPointerException e) {
    System.out.println("NPE");
}

この例で発生するのは ArrayIndexOutOfBoundsException です。catchの型が違うため捕まりません。

10. finallyは例外があっても実行される

try {
    throw new RuntimeException();
} finally {
    System.out.println("finally");
}

例外は消えませんが、finally は実行されます。出力問題では、例外で止まる前に何が表示されたかを確認します。

見分け方

  • 構文と型に問題がなければ、次に null、添字、キャスト、parse、変更不可コレクションを見る。
  • 例外がcatchされるかは、実際に発生する例外型とcatchの型で判断する。
  • finallyやtry-with-resourcesは、例外発生後にも出力を追加する可能性がある。

演習前チェック

この記事を演習で使える知識に変える

対象レベル：Java基礎〜Silver共通。この範囲は、基礎理解とSilverの読解をつなぐ部分です。まず「コンパイルできるか」「実行時にどこまで進むか」「出力は何か」の順で考えます。

本文で必ず押さえること

  • まずコンパイル時点で必要な処理が書かれているか見る。checked例外は特にここで落ちる。
  • try内で例外が出た行以降は実行されない。catchへ制御が移る。
  • finallyとtry-with-resourcesの実行順序を分ける。

この後の演習で見るポイント

  • catchの順番が親子関係として正しいか確認する。
  • 例外が発生する行を特定し、その後の行が実行されるかを判断する。
  • 非チェック例外でも、catchされるかされないかで終了の仕方が変わる。

ここで落ちる典型ミス

  • Exceptionを先にcatchしてから、その下に細かい例外を書く。これは到達不能になる。
  • finallyがあるから例外が消えると誤解する。finallyは原則として実行されるだけ。
  • コンパイルエラーと実行時例外を同じ「エラー」として処理する。

演習前の小課題

try内で3行目に例外が出るコードを想定し、4行目が実行されるか、catch後にどこへ進むかを言葉で説明してください。
