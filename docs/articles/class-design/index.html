クラス・インスタンス・static・コンストラクタ完全整理

Silverのクラス問題は、用語を知っているだけでは解けません。メンバ、フィールド、メソッド、コンストラクタ、static、this、superを、コンパイル可否と実行順に直結する形で整理します。

まず用語を固定する

クラス単元で最初に崩れる原因は、単語の境界が曖昧なことです。特に「メンバ」「フィールド」「メソッド」「コンストラクタ」を混ぜて読むと、static文脈・初期化順・継承問題で必ず詰まります。

class Item {
    String name;              // フィールド。メンバの一種
    static int count;         // staticフィールド。メンバの一種

    Item(String name) {       // コンストラクタ。メンバの一種
        this.name = name;
        count++;
    }

    void print() {            // メソッド。メンバの一種
        System.out.println(name);
    }
}
メンバとはクラスの中に直接定義される構成要素。Silverでは主にフィールド、メソッド、コンストラクタを指す。field / method / constructor
フィールドとはクラスまたはインスタンスが保持する値。メソッド内のローカル変数とは違い、自動初期化される。
メソッドとは呼び出されて処理を行う部品。戻り値型、名前、引数リスト、処理本体を持つ。
コンストラクタとはnewでインスタンスを作るときに実行される初期化処理。戻り値型を書かない。
用語コード上の位置初期化試験で見る点
フィールドクラス直下自動初期化ありインスタンスごとか、staticで共有か
ローカル変数メソッドやコンストラクタの中自動初期化なし使う前に確実に代入されているか
メソッドクラス直下呼び出されたとき実行static文脈、戻り値、オーバーロード
コンストラクタクラス直下new時に実行this()/super()、暗黙super、初期化順
重要: 「クラスの中にある変数は全部フィールド」ではありません。メソッド内にあるものはローカル変数です。自動初期化の有無が変わります。

この単元の判定順

クラス問題はいきなり出力を追うとミスります。次の順で機械的に見てください。

  1. 宣言が正しいか: クラス名、ファイル名、修飾子、メソッド/コンストラクタの形を見る。
  2. static文脈か: staticメソッドからインスタンスメンバを直接触っていないかを見る。
  3. インスタンス生成: newが何回実行され、どのコンストラクタが呼ばれるかを見る。
  4. 参照先: 変数がどのインスタンスを参照しているかを追う。
  5. 初期化順: フィールド初期化、初期化ブロック、コンストラクタ本体の順を見る。
  6. 出力・例外: 最後に結果を確定する。

クラス、オブジェクト、インスタンス

クラスは定義です。インスタンスは new によって実行時に作られる実体です。オブジェクトという言葉は、実体全般を指す広い言い方として使われます。Silverでは「newで作られたもの」と考えて問題ありません。

class Box {
    int value;
}

Box a = new Box(); // Boxインスタンスを1つ作り、参照をaへ入れる
Box b = new Box(); // 別のBoxインスタンスを1つ作る

a.value = 10;
b.value = 20;
System.out.println(a.value); // 10

a と b はインスタンスそのものではなく、インスタンスへの参照を持つ変数です。

参照とはオブジェクトそのものではなく、オブジェクトを見るためのリンク情報。代入するとリンクがコピーされる。
参照型変数とはオブジェクトへの参照を保持する変数。String、配列、クラス型、インタフェース型など。

参照コピーとフィールド変更

参照型の代入は、インスタンスのコピーではありません。参照のコピーです。

class Box {
    String text;
}

Box x = new Box();
Box y = new Box();
x.text = "A";
y.text = "B";

Box z = x;   // zはxと同じインスタンスを見る
x = y;       // xはyと同じインスタンスを見る
z.text = "C";

System.out.println(x.text); // B
System.out.println(y.text); // B
System.out.println(z.text); // C
行xyz意味
Box z = x;A側B側A側zにxの参照をコピー
x = y;B側B側A側xだけ参照先が変わる
z.text = "C";B側B側A側A側の中身だけ変更
読み方: 変数名を追うのではなく、「変数 → 参照先インスタンス」を追います。

フィールドとローカル変数の初期化

フィールドは自動初期化されます。ローカル変数は自動初期化されません。この差はコンパイルエラー判定に直結します。

class Sample {
    int number;        // 0
    boolean flag;      // false
    String text;       // null
    static int count;  // 0

    void test() {
        int local;
        System.out.println(number); // OK
        System.out.println(text);   // OK: null
        // System.out.println(local); // NG: 初期化されていない
    }
}
型フィールドの初期値
整数型0
浮動小数点型0.0
char''
booleanfalse
参照型null

コンストラクタの基本

コンストラクタは、インスタンス生成時に初期化を行う特殊な構文です。メソッドではありません。戻り値型を書かず、名前はクラス名と完全一致します。

class User {
    String name;

    User(String name) {     // コンストラクタ
        this.name = name;
    }

    void User(String name) { // メソッド。コンストラクタではない
        this.name = name;
    }
}
観点メソッドコンストラクタ
名前任意クラス名と同じ
戻り値型必要。void も戻り値型書かない
呼び出しobj.method()new User(...)
継承条件次第で継承される継承されない
試験の罠: void User() はコンストラクタではありません。戻り値型 void を持つ通常メソッドです。

デフォルトコンストラクタ

コンストラクタを1つも書かない場合だけ、コンパイラが引数なしコンストラクタを暗黙に追加します。1つでも自分で書いたら、引数なしコンストラクタは自動追加されません。

class A {
    // 何も書かないので A() が暗黙に追加される
}
new A(); // OK

class B {
    B(String s) {}
}
// new B(); // NG: B() は存在しない

サブクラスの暗黙 super() と組み合わさると、ここがコンパイルエラーの原因になります。

this と this()

this は現在のインスタンスを指す参照です。this() は同じクラスの別コンストラクタを呼ぶ構文です。似ていますが別物です。

class Item {
    String name;
    int price;

    Item() {
        this("unknown", 0); // 同じクラスの別コンストラクタを呼ぶ
    }

    Item(String name, int price) {
        this.name = name;   // フィールドnameを指す
        this.price = price;
    }
}
表記意味使える場所
this現在のインスタンスインスタンスメソッド、コンストラクタ、初期化ブロック
this.field現在のインスタンスのフィールド同名ローカル変数との区別に使う
this(...)同じクラスの別コンストラクタ呼び出しコンストラクタの先頭行のみ

super と super()

super はスーパークラス側のメンバを指すために使います。super() はスーパークラスのコンストラクタを呼ぶ構文です。

class Parent {
    String name = "P";
    Parent(String name) {
        this.name = name;
    }
    void print() {
        System.out.println(name);
    }
}

class Child extends Parent {
    String name = "C";
    Child() {
        super("parent"); // 親コンストラクタ呼び出し
    }
    void print() {
        super.print();   // 親メソッド呼び出し
        System.out.println(super.name); // 親フィールド
        System.out.println(this.name);  // 子フィールド
    }
}
スーパークラスとは継承元のクラス。親クラスとも呼ばれる。
サブクラスとは継承して作られたクラス。子クラスとも呼ばれる。

this() / super() の先頭行ルール

this(...) と super(...) は、コンストラクタの先頭行にしか書けません。同じコンストラクタ内で両方を書くこともできません。

class A {
    A(int n) {}
}
class B extends A {
    B() {
        this(10); // OK: 先頭行
    }
    B(int n) {
        super(n); // OK: 先頭行
    }
}

class C extends A {
    C() {
        // System.out.println("x");
        // super(10); // NG: 先頭行ではない
        super(10);
    }
}
理由: インスタンスを正しく初期化するため、親または別コンストラクタ呼び出しを最初に確定させる必要があります。

暗黙の super()

コンストラクタの先頭に this(...) も super(...) も書かない場合、コンパイラは暗黙に super() を追加します。

class Parent {
    Parent(String s) {}
}

class Child extends Parent {
    Child() {
        // 暗黙に super(); が入る
        // しかしParentにParent()がないのでNG
    }
}

この問題は「サブクラス側のコンストラクタが正しいか」だけでは判断できません。親クラスに引数なしコンストラクタが存在するかを必ず見ます。

インスタンス生成時の実行順

出力問題では、フィールド初期化、初期化ブロック、コンストラクタの順序を固定します。

  1. メモリ確保。フィールドがデフォルト値になる。
  2. スーパークラス側の初期化が先に進む。
  3. 自クラスのフィールド初期化とインスタンス初期化ブロックが、ソースに書かれた順に実行される。
  4. 自クラスのコンストラクタ本体が実行される。
class Sample {
    int a = print("field", 1);

    {
        a = print("block", 2);
    }

    Sample() {
        a = print("constructor", 3);
    }

    static int print(String label, int value) {
        System.out.println(label);
        return value;
    }
}

new Sample();
// field
// block
// constructor

staticの基本

static は「インスタンスごと」ではなく「クラス側」に属するという意味です。staticフィールドはインスタンスを何個作ってもクラスに1つだけです。

class Counter {
    static int total;
    int value;

    Counter() {
        total++;
        value++;
    }
}

Counter a = new Counter();
Counter b = new Counter();
System.out.println(Counter.total); // 2
System.out.println(a.value);       // 1
System.out.println(b.value);       // 1
対象所属個数呼び出し方
インスタンスフィールド各インスタンスインスタンスごとobj.value
staticフィールドクラスクラスに1つClassName.value
インスタンスメソッドインスタンスインスタンス経由で実行obj.method()
staticメソッドクラスクラス経由で実行ClassName.method()

staticメソッドからインスタンスメンバを直接使えない

staticメソッドはクラス側の処理です。特定のインスタンスが決まっていないため、インスタンスフィールドやインスタンスメソッドを直接使えません。

class Sample {
    int value = 10;
    static int count = 20;

    void instanceMethod() {}
    static void staticMethod() {}

    static void test() {
        System.out.println(count); // OK
        staticMethod();            // OK
        // System.out.println(value); // NG
        // instanceMethod();          // NG

        Sample s = new Sample();
        System.out.println(s.value); // OK: インスタンス経由
    }
}
mainで出る理由: main は static です。だから、mainからインスタンスメンバを使うには、まず new が必要です。

static初期化ブロックとインスタンス初期化ブロック

static初期化ブロックはクラスが初期化されるタイミングで実行されます。インスタンス初期化ブロックはインスタンス生成ごとに実行されます。

class Sample {
    static {
        System.out.println("static block");
    }
    {
        System.out.println("instance block");
    }
    Sample() {
        System.out.println("constructor");
    }
}

new Sample();
new Sample();
// static block
// instance block
// constructor
// instance block
// constructor

staticブロックは最初のクラス初期化時に1回だけです。インスタンスブロックとコンストラクタはnewのたびに動きます。

アクセス修飾子とメンバ

アクセス修飾子は「どこから見えるか」を決めます。メンバに使える主なアクセス修飾子は public、protected、指定なし、private です。

修飾子同じクラス同じパッケージ別パッケージのサブクラスその他
private可不可不可不可
指定なし可可不可不可
protected可可条件付きで可不可
public可可可可

指定なしは「デフォルトアクセス」です。default と書くわけではありません。

コード断片判定理由

this(); super();不可同じコンストラクタで両方は書けない
System.out.println(field); in static method不可インスタンスメンバを直接参照
class Child extends Parent { Child(){} }親次第暗黙 super() を呼べるか確認

読解ミニ演習

class A {
    static int x = print("A.static", 1);
    int y = print("A.field", 2);

    A() {
        print("A.constructor", 3);
    }

    static int print(String text, int value) {
        System.out.println(text);
        return value;
    }
}

class B extends A {
    static int x = print("B.static", 4);
    int y = print("B.field", 5);

    B() {
        print("B.constructor", 6);
    }

    public static void main(String[] args) {
        new B();
        new B();
    }
}

順番は、初回だけstatic初期化、その後newごとに親インスタンス初期化→親コンストラクタ→子インスタンス初期化→子コンストラクタです。

試験での結論: クラス・インスタンス問題は、単語の意味、static/instance、初期化順、参照先の4つを分ければかなり安定します。
