クラス設計・static・recordの要点

クラス問題では、フィールド、ローカル変数、コンストラクタ、 this 、 super 、 static の区別が重要です。

クラス・オブジェクト・インスタンスの入口

クラスは設計図、オブジェクトは実体、インスタンスはクラスから作られた具体的な実体です。学習上は「オブジェクト」と「インスタンス」は近い意味で使われますが、コードを読むときは クラス側にあるもの と インスタンス側にあるもの を分けることが重要です。

用語意味コード上の例
クラスフィールドやメソッドの定義を持つ設計。class Item { ... }
オブジェクト / インスタンスnew によって作られる実体。new Item()
フィールドオブジェクトやクラスが保持する状態。int price;
メソッド状態を使って処理を行う動作。void print()
 
class Item {
    String name;       // フィールド: 状態
    int price;         // フィールド: 状態

    void print() {     // メソッド: 動作
        System.out.println(name + ":" + price);
    }
}

static はクラス側に属する

static は「インスタンスごと」ではなく「クラス側」に属します。インスタンスを何個作っても、static フィールドはクラスに1つです。

class Counter {
    static int total; // クラス側に1つ
    int value;        // インスタンスごとに1つ

    Counter() {
        total++;
        value++;
    }
}
対象所属先個数注意点
インスタンスフィールド各インスタンスインスタンスごと参照先が違えば値も別。
インスタンスメソッド各インスタンスの状態を使う処理呼び出しには原則インスタンスが必要this を使える。
staticフィールドクラス側クラスに1つ全インスタンスで共有される。
staticメソッドクラス側クラスから呼べるインスタンスメンバを直接使えない。
recordの見方: record はデータ保持用クラスを短く書く仕組みです。通常クラスと同じく、型・コンストラクタ・メソッド・staticの規則からは逃げられません。

この単元の目的

この単元は、Javaで頻出の「クラスを見て、どの値がどこにあるか」を判断するための土台です。単語を覚えるだけでは足りません。問題では、インスタンスが複数作られたり、参照がコピーされたり、static とインスタンスメンバが混在したりします。

読む順番は固定してください。まずコンパイル可否、次にインスタンス生成、次に参照、最後に出力です。この順番を崩すと、出力問題で無駄に悩みます。

  • クラスは設計、インスタンスは実体。
  • 参照型変数はインスタンスそのものではなく、インスタンスへの参照を持つ。
  • static はインスタンスごとではなく、クラス側に属する。
  • record はデータ保持用クラスを短く書く仕組みだが、通常クラスと同じコンパイル規則から逃げられるわけではない。

最初に使う判定順

クラス・インスタンス問題は、いきなり出力を追わないでください。次の順番で切るとミスが減ります。

  1. 宣言が正しいか: クラス名、ファイル名、修飾子、フィールド、メソッド、コンストラクタを見る。
  2. static文脈かインスタンス文脈か: static メソッドからインスタンスメンバを直接触っていないかを見る。
  3. インスタンスが何個作られたか: new の回数だけでなく、代入で参照が上書きされていないかを見る。
  4. 変数がどのインスタンスを参照しているか: 参照コピー、null、再代入を追う。
  5. フィールドかメソッドか: フィールドは宣言型寄り、メソッドは実体型寄りの挙動が絡む場合がある。
  6. 出力・例外・GC対象を判断する: 最後に結果を出す。
「new が2回あるからインスタンスは2個」と数えるだけでは不十分です。参照が上書きされると、どこからも参照されないインスタンスが出ます。問題ではそこを狙われます。

クラスとインスタンスを分ける

クラスはプログラム上の定義です。インスタンスは new によって実行時に作られる実体です。Javaでは、参照型変数がインスタンスそのものを保持するわけではありません。保持するのは「どのインスタンスを見るか」という参照です。

class Item {
    int price;
}

Item a = new Item(); // 1個目のItemインスタンス
Item b = new Item(); // 2個目のItemインスタンス

a.price = 100;
b.price = 200;
System.out.println(a.price); // 100
System.out.println(b.price); // 200

この時点で a と b は別々のインスタンスを参照しています。フィールド price も別々です。

b = a;
System.out.println(b.price); // 100
b.price = 300;
System.out.println(a.price); // 300

b = a は、aの参照をbにコピーする処理です。インスタンスをコピーしているわけではありません。この代入後、a と b は同じインスタンスを見ます。

参照コピーの典型パターン

参照問題は、コードの横に「変数 → インスタンス」を書けば解けます。

class Box {
    String value;
}

Box x = new Box(); // インスタンスA
Box y = new Box(); // インスタンスB
x.value = "A";
y.value = "B";

Box z = x;         // zもインスタンスAを見る
x = y;             // xはインスタンスBを見る
z.value = "C";

System.out.println(x.value); // B
System.out.println(y.value); // B
System.out.println(z.value); // C
行xyzポイント
Box z = x;ABAzはxと同じAを見る
x = y;BBAxだけBへ変わる。zはAのまま
z.value = "C";BBAAの中身だけがCになる

この種の問題で「変数名」を追っているうちは弱いです。追うべきなのは、変数名ではなく参照先です。

フィールドの初期値とローカル変数の違い

フィールドには自動初期値があります。ローカル変数にはありません。この違いは、クラス問題とコンパイルエラー問題の境界です。

場所自動初期化使う前の注意
インスタンスフィールドされる数値は0、booleanはfalse、参照型はnull
staticフィールドされるインスタンスフィールドと同じ規則
ローカル変数されない使う前に確実に代入が必要
class Sample {
    int a;
    static String text;

    void test() {
        int b;
        System.out.println(a);    // OK: 0
        System.out.println(text); // OK: null
        // System.out.println(b); // NG: ローカル変数が未初期化
    }
}

「フィールドだから初期化される」「ローカル変数だから確定代入が必要」と機械的に切ってください。

インスタンス生成時の実行順

Javaでは、コンストラクタだけでなく、フィールド初期化や初期化ブロックが絡む出力問題も出ます。実行順を固定で覚えてください。

  1. メモリ上にインスタンス領域が確保され、フィールドがデフォルト値になる。
  2. スーパークラスのコンストラクタ処理が先に進む。
  3. 自クラスのフィールド初期化とインスタンス初期化ブロックが、書かれた順に実行される。
  4. 自クラスのコンストラクタ本体が実行される。
class Sample {
    int a = print("field", 1);

    {
        a = print("block", 2);
    }

    Sample() {
        a = print("constructor", 3);
    }

    static int print(String label, int value) {
        System.out.println(label);
        return value;
    }
}

new Sample();

出力順は field → block → constructor です。フィールド初期化と初期化ブロックは、ソース上の出現順で評価されます。

staticフィールドはクラス側に1つ

static フィールドは、インスタンスごとに存在する値ではありません。クラス側に1つだけ存在し、全インスタンスで共有されます。

class Counter {
    int instanceCount;
    static int classCount;

    Counter() {
        instanceCount++;
        classCount++;
    }
}

Counter a = new Counter();
Counter b = new Counter();

System.out.println(a.instanceCount); // 1
System.out.println(b.instanceCount); // 1
System.out.println(Counter.classCount); // 2

instanceCount は各インスタンスに1つずつあります。一方、classCount はCounterクラス側に1つです。

staticフィールドはインスタンス経由で a.classCount のように書ける場合があります。ただし、意味としてはクラス側の値です。問題では「インスタンスごとに別」と誤認させる選択肢が出ます。

staticメソッドから直接使えるもの

static メソッドは、特定のインスタンスに結びついていません。そのため、インスタンスフィールド、インスタンスメソッド、this を直接使えません。

class Sample {
    int value = 10;
    static int count = 20;

    void instanceMethod() {}

    static void staticMethod() {
        System.out.println(count); // OK
        // System.out.println(value); // NG
        // instanceMethod();          // NG
        // System.out.println(this);  // NG

        Sample s = new Sample();
        System.out.println(s.value); // OK
        s.instanceMethod();          // OK
    }
}

「インスタンスを作って、その参照経由で使う」なら可能です。直接使うのが不可です。

static初期化とインスタンス初期化

static初期化はクラスが使われるタイミングで一度だけ実行されます。インスタンス初期化は new のたびに実行されます。

class Sample {
    static int a = print("static field");

    static {
        print("static block");
    }

    int b = print("instance field");

    {
        print("instance block");
    }

    Sample() {
        print("constructor");
    }

    static int print(String s) {
        System.out.println(s);
        return 0;
    }
}

new Sample();
new Sample();

最初の new Sample() では、static field → static block → instance field → instance block → constructor の順です。2回目の new Sample() ではstatic側は再実行されません。

フィールド名とローカル変数名が衝突した場合

メソッドやコンストラクタ内で、フィールドと同名のローカル変数・引数を宣言すると、単に名前を書いた場合は近いスコープの変数が優先されます。フィールドを明示するには this を使います。

class Item {
    private int price;

    Item(int price) {
        price = price;       // 引数priceに引数priceを代入しているだけ
    }

    int getPrice() {
        return price;
    }
}

System.out.println(new Item(100).getPrice()); // 0

正しくは次です。

class Item {
    private int price;

    Item(int price) {
        this.price = price;
    }
}

this.price はインスタンスフィールド、price は引数です。この区別が曖昧だとコンストラクタ問題で落ちます。

final参照型変数の意味

final 変数は再代入できません。ただし、参照先オブジェクトが可変なら、オブジェクトの中身は変更できます。

final StringBuilder sb = new StringBuilder("A");
sb.append("B");       // OK: 参照先の中身を変更している
System.out.println(sb); // AB

// sb = new StringBuilder("C"); // NG: 再代入

逆に、String は不変なので、文字列操作は新しいStringを作る処理です。

final String s = "A";
// s += "B"; // NG: s = s + "B" と同じ再代入

「finalだから中身変更不可」と雑に覚えると間違えます。finalが禁止するのは、変数が持つ参照の差し替えです。

ガベージコレクション対象の判断

ガベージコレクションの対象になるのは、どこからも参照されなくなったインスタンスです。実際に回収されるタイミングは保証されません。問題で聞かれるのは「対象になるか」であり、「いつ回収されるか」ではありません。

Object a = new Object(); // インスタンスA
Object b = new Object(); // インスタンスB
Object c = a;            // cもAを見る

a = null; // aからAへの参照が外れる。ただしcがAを見ている
b = a;    // bにnullを代入。Bへの参照が外れる

// この時点でGC対象: インスタンスB
// インスタンスAはcが参照しているので対象外
操作ABGC対象
Object c = aa, cが参照bが参照なし
a = nullcが参照bが参照なし
b = acが参照参照なしB
System.gc() はGCの実行を促すだけです。必ず即回収されるとは言えません。選択肢に「必ず回収される」があれば疑ってください。

recordの基本構造

record は、データを保持するクラスを簡潔に定義する仕組みです。コンポーネントを書くと、フィールド、コンストラクタ、アクセサ、equals、hashCode、toString が自動的に用意されます。

record User(String name, int age) {}

User u = new User("Alice", 20);
System.out.println(u.name()); // Alice
System.out.println(u.age());  // 20
System.out.println(u);        // User[name=Alice, age=20] のような形式

アクセサ名は getName() ではなく name() です。この違いは頻出です。

// u.getName(); // NG
u.name();       // OK

recordの制約

recordは普通のクラスより制約があります。ここを曖昧にすると、Java 17論点で失点します。

項目判定理由
暗黙にfinal継承されないrecordをextendsするクラスは作れない
明示的に別クラスをextends不可recordは暗黙にjava.lang.Recordを継承する
interface実装可implements は使える
インスタンスフィールド追加不可recordの状態はコンポーネントで表す
staticフィールド追加可クラス側の値なので状態追加ではない
interface Named {
    String name();
}

record User(String name, int age) implements Named {
    static int count;     // OK
    // int level;         // NG: 追加のインスタンスフィールド
}

recordのコンストラクタ

recordには、コンポーネントと同じ引数を持つ標準コンストラクタがあります。また、コンパクトコンストラクタを使うと、引数リストを書かずに検証処理を追加できます。

record User(String name, int age) {
    User {
        if (age < 0) {
            throw new IllegalArgumentException();
        }
    }
}

コンパクトコンストラクタ内では、最終的にコンポーネントフィールドへ値が設定されます。明示的に this.name = name; のような代入を書く通常のコンストラクタとは感覚が違います。

record User(String name, int age) {
    public User(String name, int age) {
        if (age < 0) {
            throw new IllegalArgumentException();
        }
        this.name = name;
        this.age = age;
    }
}

明示的な標準コンストラクタを書く場合は、すべてのコンポーネントを初期化する必要があります。

recordは浅い不変性で考える

recordのコンポーネントフィールドは再代入できません。ただし、参照先が可変オブジェクトなら、その中身まで自動的に不変になるわけではありません。

import java.util.ArrayList;

record Team(ArrayList<String> members) {}

ArrayList<String> list = new ArrayList<>();
list.add("A");
Team t = new Team(list);

list.add("B");
System.out.println(t.members()); // [A, B]

Javaでは深い不変性まで問われる頻度は高くありませんが、「recordだから中のListも変更不可」とは言えません。

printlnとtoString

System.out.println(obj) は、参照型を出力するときに基本的に toString() の結果を使います。null の場合は例外ではなく null と表示されます。

Object obj = null;
System.out.println(obj); // null

// obj.toString(); // NG: NullPointerException

println(obj) と obj.toString() は同じではありません。ここは地味に落とされます。

典型ひっかけ

  • a = b をオブジェクトコピーだと思う。
  • static フィールドをインスタンスごとに別だと思う。
  • static メソッドからインスタンスフィールドを直接使えると思う。
  • フィールドとローカル変数の初期化ルールを混同する。
  • final 参照型変数の中身まで変更不可だと思う。
  • GC対象とGC実行タイミングを混同する。
  • recordのアクセサを getXxx() だと思う。
  • recordが通常クラスを extends できると思う。

class A { int x; }
A a = new A();
A b = a;
b.x = 10;

答え: 10 。aとbは同じインスタンスを参照しています

class A {

答え: コンパイルエラー。staticメソッドからインスタンスフィールド x を直接参照しています

この単元のゴール

メソッド・コンストラクタの問題は、コード量が少なくても罠が多いです。特に オーバーロード解決、戻り値、可変長引数、this/super、暗黙super、値渡し は、Javaで頻出

  1. メソッドなら: 宣言が正しいか、呼び出しに一致する候補があるか、戻り値が適切かを見る。
  2. コンストラクタなら: 戻り値型を書いていないか、this()/super() が先頭か、親コンストラクタを呼べるかを見る。
  3. 引数渡しなら: 変数そのものが変わるのか、参照先オブジェクトの中身が変わるのかを分ける。
雑に覚えると危険: 「参照型は参照渡し」という言い方は学習対策としては不正確です。Javaは常に値渡しです。参照型では参照値がコピーされます。

メソッド宣言の構成

メソッド宣言は、修飾子、戻り値型、メソッド名、引数リスト、throws句、メソッド本体で構成されます。

public static int add(int a, int b) throws Exception {
    return a + b;
}
部分例見るポイント
修飾子public staticアクセス、static/instanceの違い
戻り値型intvoid 以外は必ず値を返す必要がある
メソッド名add識別子ルールに従う
引数int a, int b型と順番がシグネチャに関係する
throwsthrows Exceptionチェック例外の処理に関係する
本体{ return a + b; }戻り値、到達不能文、例外発生を確認する

メソッドシグネチャとオーバーロード

Javaでオーバーロード判定に使うシグネチャは、基本的に メソッド名 + 引数リスト です。戻り値型だけを変えてもオーバーロードにはなりません。

class Sample {
    void test(int a) {}
    void test(String s) {}     // OK: 引数型が違う
    void test(int a, int b) {} // OK: 引数の数が違う
    // int test(int a) { return a; } // NG: 戻り値型だけ違う
}
違いオーバーロード可否理由
引数の型が違う可test(int) と test(String) は別
引数の数が違う可test(int) と test(int,int) は別
引数の順番が違う可test(int,String) と test(String,int) は別
戻り値型だけ違う不可呼び出し側だけでは区別できない
仮引数名だけ違う不可test(int x) と test(int y) は同じシグネチャ

オーバーロード解決の優先順位

複数の候補がある場合、呼び出しに最も適合するメソッドが選ばれます。Javaでは、細かい仕様を完全暗記するより、次の優先順で選択肢を切ると安定します。

優先変換例
1完全一致test(int) に 10
2拡大プリミティブ変換int → long
3ボクシングint → Integer
4可変長引数int...
 
void test(long x) { System.out.print("long"); }
void test(Integer x) { System.out.print("Integer"); }
void test(int... x) { System.out.print("varargs"); }

test(10); // long

10 はintリテラルです。long への拡大変換は、Integer へのボクシングや可変長引数より優先されます。

nullを渡すオーバーロード

null は参照型に代入できます。複数の参照型候補がある場合、より具体的な型が選ばれます。どちらが具体的か決められない場合はコンパイルエラーです。

void test(Object o) { System.out.print("Object"); }
void test(String s) { System.out.print("String"); }

test(null); // String

String は Object のサブクラスなので、より具体的な String が選ばれます。

void sample(String s) {}
void sample(StringBuilder sb) {}

// sample(null); // NG: String と StringBuilder のどちらとも決められない
判断: null 問題では、候補が親子関係か、横並びの別クラスかを見ます。

varargsのルール

可変長引数は、0個以上の引数を受け取れる仕組みです。メソッド内では配列として扱われます。

void print(String label, int... values) {
    System.out.println(values.length);
}

print("A");          // 0
print("B", 1, 2, 3); // 3
ルール例
可変長引数は最後に1つだけvoid m(String s, int... x)
最後以外には書けないvoid m(int... x, String s) はNG
2つ以上書けないvoid m(int... a, String... b) はNG
配列として渡せるm(new int[]{1,2})

戻り値の判定

void 以外のメソッドは、正常終了するすべての経路で値を返す必要があります。

int ok(boolean b) {
    if (b) {
        return 1;
    }
    return 2;
}

int ng(boolean b) {
    if (b) {
        return 1;
    }
    // NG: bがfalseの経路でreturnがない
}

return の後に通常文を書くと到達不能文としてコンパイルエラーになることがあります。

void test() {
    return;
    // System.out.println("A"); // NG: 到達不能
}

コンストラクタの基本

コンストラクタはインスタンス生成時に呼ばれる特別な処理です。クラス名と同じ名前で、戻り値型を書きません。戻り値型を書いたら、それはコンストラクタではなく普通のメソッドです。

class Sample {
    Sample() { }       // コンストラクタ
    void Sample() { }  // メソッド。コンストラクタではない
}
項目メソッドコンストラクタ
名前任意の識別子クラス名と同じ
戻り値型必要。void も可書かない
呼び出しobj.method()new Sample()
継承条件次第で継承される継承されない

デフォルトコンストラクタ

コンストラクタを1つも定義しない場合、コンパイラが引数なしコンストラクタを自動で追加します。しかし、1つでも自分でコンストラクタを定義すると、デフォルトコンストラクタは追加されません。

class A {
    // コンストラクタを何も書かないので A() が暗黙にある
}
new A(); // OK

class B {
    B(String s) {}
}
// new B(); // NG: B() は自動追加されない
頻出: 「引数ありコンストラクタを書いたから、引数なしもあるはず」は誤りです。必要なら明示的に定義します。

this() と super()

this() は同じクラスの別コンストラクタを呼び、super() は親クラスのコンストラクタを呼びます。どちらもコンストラクタの先頭行にしか書けません。同じコンストラクタ内で両方を書くこともできません。

class A {
    A(int n) {}
}
class B extends A {
    B() {
        this(10); // OK: 先頭行
    }
    B(int n) {
        super(n); // OK: 先頭行
    }
}
class C extends A {
    C() {
        System.out.println("C");
        // super(10); // NG: 先頭行ではない
    }
}

暗黙のsuper()

コンストラクタの先頭に this() も super() も書かない場合、コンパイラが super() を挿入します。そのため、親クラスに引数なしコンストラクタがないとコンパイルエラーになります。

class Parent {
    Parent(String name) {}
}
class Child extends Parent {
    Child() {
        // ここに super(); が暗黙挿入される
        // しかし Parent() が存在しないのでNG
    }
}

修正するなら、親のコンストラクタに合うように明示的に super("x") を呼びます。

class Child extends Parent {
    Child() {
        super("x");
    }
}

コンストラクタ連鎖の実行順

サブクラスのインスタンス生成では、最終的に親クラス側から初期化されます。this() で同じクラスの別コンストラクタへ進んでも、どこかで super() に到達します。

class A {
    A() { System.out.print("A"); }
}
class B extends A {
    B() {
        this(1);
        System.out.print("B");
    }
    B(int n) {
        System.out.print("C");
    }
}
new B();
順番処理出力
1B() が this(1) を呼ぶ-
2B(int) の先頭に暗黙 super()-
3A()A
4B(int) の本体C
5B() の残りB

出力は ACB です。

引数渡し: Javaは常に値渡し

プリミティブ型では値そのもののコピー、参照型では参照値のコピーが渡されます。

static void change(int x) {
    x = 20;
}
int a = 10;
change(a);
System.out.println(a); // 10

呼び出し元の a は変わりません。コピーされた x だけが変わります。

static void change(StringBuilder sb) {
    sb.append("B");
    sb = new StringBuilder("C");
}
StringBuilder s = new StringBuilder("A");
change(s);
System.out.println(s); // AB
操作呼び出し元に影響理由
sb.append("B")ある同じオブジェクトの中身を変更している
sb = new StringBuilder("C")ない仮引数の参照値コピーを差し替えただけ

staticメソッドとインスタンスメソッド

static メソッドはインスタンスなしで呼べますが、インスタンスフィールドへ直接アクセスできません。

class Sample {
    private String name = "A";
    static void print() {
        // System.out.println(name); // NG: staticからinstanceを直接参照できない
    }
}

インスタンスを作って、その参照を経由すればアクセスできます。

class Sample {
    private String name = "A";
    static void print() {
        Sample s = new Sample();
        System.out.println(s.name); // OK
    }
}

mainメソッドの形

実行エントリポイントとして使えるmainメソッドは、基本形を崩さないことが重要です。

public static void main(String[] args) {}
public static void main(String... args) {}
要素必要条件
アクセスpublic
static必要
戻り値void
名前main
引数String[] または String...

コンストラクタとアクセス修飾子

コンストラクタにもアクセス修飾子を付けられます。サブクラスから親コンストラクタを呼ぶ場合、そのコンストラクタにアクセスできる必要があります。

class A {
    private A() {}
}
class B extends A {
    // B() { super(); } が必要だが、A() はprivateなのでNG
}

protected の親コンストラクタは、サブクラスから呼べます。

class A {
    protected A() {}
}
class B extends A {
    public B() {
        super(); // OK
    }
}

アクセス修飾子の基本表

修飾子同じクラス同じパッケージ別パッケージのサブクラス別パッケージの非サブクラス
private可不可不可不可
指定なし可可不可不可
protected可可条件付きで可不可
public可可可可

指定なしは「デフォルトアクセス」または「パッケージアクセス」です。default という修飾子を書くわけではありません。default キーワードはインタフェースのデフォルトメソッドで出てきます。ここを混同しないでください。

トップレベルクラスに使える修飾子

トップレベルクラスとは、他のクラスの中に入っていないクラスです。トップレベルクラスに使えるアクセス修飾子は public または指定なしです。

public class A { }     // OK
class B { }            // OK: デフォルトアクセス

// private class C { } // NG
// protected class D { } // NG
// static class E { }  // NG: トップレベルでは不可

トップレベルクラスでは、abstract、final、sealed、non-sealed などの継承方針に関係する修飾子も出題されます。ただし、private、protected、static はトップレベルクラスには使えません。

メンバに使うアクセス修飾子

フィールド、メソッド、コンストラクタには、public、protected、指定なし、private を使えます。

class Sample {
    public int a;
    protected int b;
    int c;          // デフォルトアクセス
    private int d;

    public Sample() {}
    private void helper() {}
}

ただし、使えることと、外部からアクセスできることは別です。宣言として正しくても、呼び出し側の場所によってはアクセスエラーになります。

privateメンバは継承先から直接見えない

private メンバは同じクラス内からだけ使えます。サブクラスであっても直接アクセスできません。

class Parent {
    private int value = 10;
    private void test() {}
}

class Child extends Parent {
    void sample() {
        // System.out.println(value); // NG
        // test();                    // NG
    }
}

ただし、privateメンバが「インスタンス内に存在しない」という意味ではありません。アクセスできないだけです。publicなメソッド経由で値を使うことはできます。

class Parent {
    private int value = 10;
    public int getValue() { return value; }
}

class Child extends Parent {
    void sample() {
        System.out.println(getValue()); // OK
    }
}

privateメソッドはoverrideされない

privateメソッドはサブクラスから見えません。そのため、サブクラスで同じシグネチャのメソッドを書いてもoverrideではありません。別メソッドです。

class A {
    private void test() {
        System.out.println("A");
    }
}

class B extends A {
    void test() {
        System.out.println("B");
    }
}

@Override を付けると、overrideではないことが検出され、コンパイルエラーになります。

class B extends A {
    // @Override
    void test() { }
}

protectedの正しい理解

protected は「同じパッケージなら継承関係なしでアクセス可能」です。さらに、別パッケージでもサブクラスから条件付きでアクセスできます。

package p1;
public class A {
    protected int value;
}

package p1;
public class X {
    void test() {
        A a = new A();
        a.value = 10; // OK: 同じパッケージ
    }
}

別パッケージのサブクラスでは、単に「継承しているから何でもOK」ではありません。サブクラス自身またはサブクラス型の参照を通じたアクセスとして考える必要があります。

package p2;
public class B extends p1.A {
    void test(B b, p1.A a) {
        this.value = 1; // OK
        b.value = 2;    // OK
        // a.value = 3; // NGになりうる典型形
    }
}
protectedを「別パッケージのサブクラスなら親型参照からでも自由に触れる」と覚えるのは危険です。Javaの問題では、この雑な理解を狙われます。

コンストラクタのアクセス修飾子

コンストラクタにもアクセス修飾子を付けられます。コンストラクタのアクセス範囲は、インスタンス生成できる範囲を決めます。

class Sample {
    private Sample() {}

    static Sample create() {
        return new Sample(); // 同じクラス内なのでOK
    }
}

// new Sample(); // クラス外ならNG

また、サブクラスのコンストラクタは、スーパークラスのコンストラクタを直接または間接的に呼び出す必要があります。スーパークラスの利用可能なコンストラクタがなければ、サブクラス側でコンパイルエラーになります。

class A {
    private A() {}
}

class B extends A {
    // B() { super(); } が暗黙に入るが、A()がprivateなのでNG
}

一方、protected コンストラクタなら、サブクラスから super() で呼べます。

class A {
    protected A() {}
}

class B extends A {
    public B() {
        super(); // OK
    }
}

abstract classとabstract method

abstract クラスは、インスタンス化できないクラスです。抽象メソッドは本体を持たないメソッドです。

abstract class A {
    abstract void test(); // 本体なし

    void common() {       // 通常メソッドも持てる
        System.out.println("common");
    }
}

class B extends A {
    @Override
    void test() {
        System.out.println("B");
    }
}

抽象メソッドを持つクラスは、必ずabstractクラスでなければなりません。

// class A {
//     abstract void test(); // NG: クラスもabstractにする必要あり
// }

逆に、abstractクラスに抽象メソッドが必ず必要というわけではありません。抽象メソッドなしのabstractクラスも作れます。

abstractと矛盾する修飾子

抽象メソッドは、サブクラスで実装されることが前提です。そのため、実装やoverrideを妨げる修飾子とは相性が悪いです。

組み合わせ判定理由
abstract final class不可継承前提と継承禁止が矛盾
private abstract メソッド不可サブクラスから見えず、実装できない
final abstract メソッド不可override必須とoverride禁止が矛盾
static abstract メソッド不可staticメソッドはoverride対象ではない
abstract class A {
    // private abstract void a(); // NG
    // final abstract void b();   // NG
    // static abstract void c();  // NG
}

finalの3つの意味

final は付ける対象によって意味が変わります。

対象意味例
変数再代入不可final int x = 10;
メソッドoverride不可final void test()
クラス継承不可final class A
final class A {}
// class B extends A {} // NG

class C {
    final void test() {}
}
class D extends C {
    // void test() {} // NG
}

final変数の注意点は、参照型の場合です。再代入は不可ですが、可変オブジェクトの中身変更は別問題です。

staticはoverrideではなく隠蔽

staticメソッドはインスタンスではなくクラスに属します。そのため、サブクラスで同じシグネチャのstaticメソッドを定義してもoverrideではありません。メソッドの隠蔽です。

class A {
    static void test() {
        System.out.println("A");
    }
}

class B extends A {
    static void test() {
        System.out.println("B");
    }
}

A a = new B();
a.test(); // A: 宣言型A側のstaticメソッド

staticメソッド呼び出しは、実体型による動的ディスパッチではなく、宣言型・クラス名に寄ると考えると整理できます。

staticメソッドに @Override を付けるとコンパイルエラーです。overrideではありません。

override時のアクセス修飾子

オーバーライドでは、アクセス範囲を狭くできません。同じか、より広いアクセスにする必要があります。

親メソッド子メソッドで可能子メソッドで不可
publicpublicprotected、指定なし、private
protectedprotected、public指定なし、private
指定なし指定なし、protected、publicprivate
class A {
    protected void test() {}
}

class B extends A {
    public void test() {} // OK: 広くしている
}

class C extends A {
    // void test() {} // NG: protectedより狭い
}

privateメソッドはoverride対象ではないため、この表とは別扱いです。

インタフェースの暗黙修飾子

インタフェースでは、暗黙に付く修飾子が多いです。ここは問題でよく省略されます。

宣言暗黙の扱い注意
フィールドpublic static final定数。必ず初期化が必要
抽象メソッドpublic abstract実装クラスではpublicで実装が必要
default メソッドpublicなインスタンスメソッド本体が必要
static メソッドインタフェース名で呼ぶ実装クラスに継承された通常メソッド扱いではない
private メソッドインタフェース内の補助default/staticメソッドから使う
interface A {
    int NUM = 10;        // public static final int NUM = 10;
    void test();         // public abstract void test();

    default void call() {
        helper();
    }

    private void helper() {}
}

class B implements A {
    public void test() {} // publicが必要
}
インタフェースの抽象メソッドはpublicです。実装クラス側で指定なしにすると、アクセスを狭くすることになりコンパイルエラーです。

sealed / non-sealed / permits

sealed は、継承または実装できる相手を制限する仕組みです。permits で許可されたクラスだけが直接のサブクラスになれます。

sealed class A permits B, C {}
final class B extends A {}
non-sealed class C extends A {}

// class D extends A {} // NG: permitsに含まれていない

sealedクラスを継承した直接のサブクラスは、次のどれかで今後の継承方針を示す必要があります。

修飾子意味
finalここで継承を止める
sealedさらに継承先を限定する
non-sealed以降の継承制限を解除する
sealed class Parent permits Child {}

// class Child extends Parent {} // NG: final/sealed/non-sealedのどれかが必要

final class Child extends Parent {} // OK

non-sealed は、sealedな親を持つクラスで意味を持ちます。何でもないクラスに突然 non-sealed を付けるのは不可です。
