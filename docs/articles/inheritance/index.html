継承・interface・ポリモーフィズム完全整理

Silverの継承問題は「なんとなく親子関係を読む」だけでは解けません。宣言型、実体、override、フィールド隠蔽、コンストラクタ連鎖、interfaceの暗黙ルールを分けて判定します。

最初に用語を固定する

この単元で崩れる原因は、継承・オーバーライド・ポリモーフィズム・interfaceを同じ意味で読んでしまうことです。Javaでは、それぞれ役割が違います。

class Animal {
    String name = "animal";
    void speak() {
        System.out.println("...");
    }
}

class Dog extends Animal {
    String name = "dog";
    @Override
    void speak() {
        System.out.println("bow");
    }
}

Animal a = new Dog();
a.speak();              // bow
System.out.println(a.name); // animal

上のコードでは、メソッドとフィールドで結果が変わります。ここを曖昧にすると、出力問題で必ず落ちます。

継承とは既存クラスを土台にして、新しいクラスを作る仕組み。extends を使う。
スーパークラスとは継承元。親クラス、基底クラスとも呼ぶ。extends の右側。
サブクラスとは継承先。子クラス、派生クラスとも呼ぶ。extends の左側。
interfaceとは「この型として扱うなら、この操作ができる」という約束を定義する型。
overrideとはスーパークラスやinterfaceのメソッドを、サブクラス側で同じシグネチャで再定義すること。
ポリモーフィズムとは同じ型の変数から呼んでも、実体のクラスに応じて動く実装が変わる性質。
結論: 呼べるかどうかは宣言型で決まります。overrideされたインスタンスメソッドの実行内容は実体で決まります。フィールドはoverrideされず、宣言型側で解決されます。

継承問題の判定順

  1. コンパイル可否: 継承関係、アクセス修飾子、抽象メソッド実装、コンストラクタ呼び出しを確認する。
  2. 宣言型: 変数の型から、その場で呼べるメンバを決める。
  3. 実体: new されている実際のクラスを見る。
  4. 呼び出し対象: メソッドなのか、フィールドなのか、staticメソッドなのかを分ける。
  5. 実行結果: overrideされたメソッドなら実体側、フィールドなら宣言型側で確定する。
見るもの判定基準例
呼べるメソッド変数の宣言型Animal a なら Animal にあるメソッドだけ
overrideメソッドの実行本体実体の型new Dog() なら Dog 側
フィールド変数の宣言型a.name は Animal 側
staticメソッド基本は参照型側overrideではなく隠蔽
キャストの実行可否実体がその型で扱えるか違えば ClassCastException

extends と is-a関係

class Dog extends Animal は「DogはAnimalの一種」と読めます。これを is-a関係 と考えます。

class Animal {}
class Dog extends Animal {}

Animal a = new Dog(); // OK: DogはAnimalの一種
// Dog d = new Animal(); // NG: Animalが必ずDogとは限らない

サブクラスのインスタンスはスーパークラス型で扱えます。しかし、スーパークラスのインスタンスをサブクラス型へ自然代入することはできません。

Animal a = new Dog();
Dog d1 = (Dog) a;      // OK: 実体がDog

Animal b = new Animal();
Dog d2 = (Dog) b;      // コンパイルは通るが実行時にClassCastException
注意: キャストは「コンパイル時に型として扱わせる」ための記述です。実体が変化するわけではありません。

Object クラス

Javaのクラスは、明示的に継承を書かなければ暗黙に Object を継承します。そのため、どのクラスのインスタンスも Object 型の変数で扱えます。

class Item {}

Object o = new Item(); // OK
System.out.println(o.toString()); // Object由来のメソッドは呼べる
// o.itemOnly(); // NG: Object型にitemOnlyはない

toString、equals、hashCode などは Object 由来です。ただし、宣言型が Object なら、呼べるのは Object に見えているメソッドだけです。

コンストラクタは継承されない

継承しても、スーパークラスのコンストラクタをサブクラスのコンストラクタとして使えるわけではありません。サブクラスのインスタンス生成時には、スーパークラス側の初期化が先に必要になるため、サブクラスのコンストラクタから super(...) を呼びます。

class Parent {
    Parent(String name) {}
}

class Child extends Parent {
    Child() {
        super("java"); // 親コンストラクタを呼ぶ
    }
}

コンストラクタの先頭に this(...) も super(...) も書かない場合、コンパイラは暗黙に super() を追加します。

class Parent {
    Parent(String name) {}
}

class Child extends Parent {
    Child() {
        // 暗黙にsuper()を呼ぼうとする
        // しかしParentに引数なしコンストラクタがないのでコンパイルエラー
    }
}
ルール結果
コンストラクタは継承されないnew Child("x") ができるかはChild側のコンストラクタ定義で決まる
super(...) は先頭行のみ前に代入文や出力文を書くとコンパイルエラー
this(...) と super(...) は同時に書けないどちらも先頭行を要求するため
暗黙の super()親に引数なしコンストラクタがないと失敗

アクセス修飾子と継承

継承していることと、直接アクセスできることは別です。特に private とデフォルトアクセス、protected を雑に覚えると失敗します。

修飾子同じクラス同じパッケージ別パッケージのサブクラス別パッケージの無関係クラス
publicOKOKOKOK
protectedOKOK条件付きOKNG
デフォルトOKOKNGNG
privateOKNGNGNG
package p1;
public class Parent {
    protected int value = 10;
}

package p2;
import p1.Parent;

public class Child extends Parent {
    void test() {
        System.out.println(value); // OK: 継承した側からアクセス
    }
}
試験での見方: defaultアクセスは「サブクラスならOK」ではありません。同じパッケージかどうかで決まります。

オーバーライドの成立条件

オーバーライドは、スーパークラスのインスタンスメソッドをサブクラスで再定義する仕組みです。名前が同じだけでは不十分です。

条件内容失敗例
シグネチャメソッド名、引数の数・型・順番が同じ引数型を変えるとオーバーロード
戻り値型同じ型、または共変戻り値親が String 戻り値で子が Object はNG
アクセス修飾子同じか、より広くするpublic を protected に狭めるのはNG
例外checked例外を広げられない親が例外なしで子が throws IOException はNG
対象インスタンスメソッドstatic、private、final は通常のoverride対象にならない
class Parent {
    public Number get() { return 1; }
}

class Child extends Parent {
    @Override
    public Integer get() { return 2; } // OK: IntegerはNumberのサブクラス
}
class Parent {
    public void test() {}
}

class Child extends Parent {
    // @Override
    // protected void test() {} // NG: アクセスを狭くしている
}

オーバーライドとオーバーロード

名前が同じでも、引数が違えばオーバーロードです。実行時のポリモーフィズムで切り替わるのはオーバーライドです。

class Parent {
    void test(Object o) {
        System.out.println("Parent Object");
    }
}

class Child extends Parent {
    void test(String s) { // overrideではなくoverload
        System.out.println("Child String");
    }
}

Parent p = new Child();
p.test("x"); // Parent Object

p の宣言型は Parent なので、コンパイル時に候補になるのは Parent から見える test(Object) です。Child 側の test(String) は、Parent 型変数からは候補になりません。

フィールド隠蔽とstaticメソッド隠蔽

フィールドはオーバーライドされません。サブクラスで同名フィールドを定義すると、フィールド隠蔽になります。

class A {
    String name = "A";
    String getName() { return name; }
}

class B extends A {
    String name = "B";
    @Override
    String getName() { return name; }
}

A obj = new B();
System.out.println(obj.name);      // A
System.out.println(obj.getName()); // B

staticメソッドもインスタンスメソッドのようにはオーバーライドされません。サブクラスで同じ形のstaticメソッドを書くと隠蔽です。

class A {
    static void print() { System.out.println("A"); }
}
class B extends A {
    static void print() { System.out.println("B"); }
}

A obj = new B();
obj.print(); // A。非推奨な書き方だが、宣言型A側で解決される
試験対策: () が付いていてもstaticメソッドなら通常の動的ディスパッチではありません。メソッド名だけで判断しないでください。

抽象クラスと抽象メソッド

abstract クラスはインスタンス化できません。抽象メソッドは本体を持たず、具象サブクラスで実装が必要です。

abstract class Service {
    abstract void execute();

    void log() {
        System.out.println("log");
    }
}

class UserService extends Service {
    @Override
    void execute() {
        System.out.println("run");
    }
}

// new Service(); // NG: abstract classはインスタンス化不可
Service s = new UserService();
s.execute(); // run
状態結果
抽象メソッドを持つクラスクラス自体も abstract にする必要がある
抽象メソッドを実装しないサブクラスそのサブクラスも abstract にする必要がある
abstract final クラス矛盾するためNG。抽象は継承前提、finalは継承禁止

interfaceの基本

interface は「この型として扱うなら、この操作ができる」という約束を定義します。クラスは implements でinterfaceを実装します。

interface Printable {
    void print(); // 暗黙にpublic abstract
}

class Report implements Printable {
    @Override
    public void print() {
        System.out.println("report");
    }
}

Printable p = new Report();
p.print();

interfaceの抽象メソッドは暗黙に public abstract です。そのため、実装クラス側では public にする必要があります。

interface Task {
    void run();
}

class Job implements Task {
    // void run() {} // NG: publicより狭いアクセス
    public void run() {} // OK
}

interfaceに書けるメンバ

要素暗黙の扱い注意点
フィールドpublic static finalインスタンスごとの状態ではなく定数
抽象メソッドpublic abstract実装クラスではpublicで実装
defaultメソッド実装を持つインスタンスメソッド実装クラスでoverride可能
staticメソッドinterface自身に属する実装クラスのインスタンスから呼ばない
privateメソッドinterface内の共通処理外部や実装クラスから直接呼べない
interface Config {
    int SIZE = 10; // public static final int SIZE = 10;

    default void show() {
        helper();
    }

    private void helper() {
        System.out.println(SIZE);
    }

    static void printType() {
        System.out.println("config");
    }
}
注意: interfaceのフィールドは定数です。SIZE = 20; のような再代入はできません。

複数interfaceと衝突

クラスは1つのクラスしか extends できませんが、複数のinterfaceを implements できます。

interface A { void a(); }
interface B { void b(); }

class C implements A, B {
    public void a() {}
    public void b() {}
}

複数のinterfaceに同じdefaultメソッドがある場合、実装クラス側で明示的に解決する必要があります。

interface A {
    default void hello() { System.out.println("A"); }
}
interface B {
    default void hello() { System.out.println("B"); }
}

class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();
    }
}

A.super.hello() は、interface A のdefault実装を明示的に呼ぶ書き方です。

クラス継承とinterface実装の書き方

クラスを継承し、同時にinterfaceを実装する場合、順序は extends が先、implements が後です。

class Base {}
interface RunnableLike { void run(); }

class App extends Base implements RunnableLike {
    public void run() {}
}
// class App implements RunnableLike extends Base {} // NG: 順序が逆
関係使うキーワード複数指定
クラスがクラスを継承extends不可。1つだけ
クラスがinterfaceを実装implements可
interfaceがinterfaceを継承extends可

final と継承

final は文脈によって意味が変わります。継承単元では、クラスとメソッドの final を特に見ます。

対象意味例
finalクラス継承できないfinal class A {}
finalメソッドoverrideできないfinal void test() {}
final変数再代入できないfinal int x = 10;
final class A {}
// class B extends A {} // NG

class Parent {
    final void test() {}
}
class Child extends Parent {
    // void test() {} // NG: finalメソッドはoverride不可
}

instanceof とキャスト

instanceof は、実体が指定した型として扱えるかを調べる演算子です。

Object obj = "java";

if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}

Java 16以降では、パターンマッチングによりキャストと変数宣言をまとめられます。

Object obj = "java";

if (obj instanceof String s) {
    System.out.println(s.length());
}

ただし、null instanceof String は false です。例外にはなりません。

コード結果理由

class C implements A で抽象メソッド未実装コンパイルエラー具象クラスなら実装が必要
interfaceメソッドをデフォルトアクセスで実装コンパイルエラーpublicを弱めている
親の public メソッドを子で privateコンパイルエラーoverrideでアクセスを狭められない
親に引数なしコンストラクタなし、子で super(...) なしコンパイルエラー暗黙の super() が失敗

読解ミニ演習

問1: 出力を答える 
class A {
    String name = "A";
    String getName() { return name; }
}
class B extends A {
    String name = "B";
    String getName() { return name; }
}
public class Main {
    public static void main(String[] args) {
        A a = new B();
        System.out.println(a.name + ":" + a.getName());
    }
}

答え: A:B。フィールドは宣言型 A、overrideメソッドは実体 B で決まります。

問2: コンパイルできるか 
interface A {
    void run();
}
class B implements A {
    void run() {}
}

答え: コンパイルエラー。interfaceの run は暗黙に public なので、実装側も public void run() にする必要があります。

問3: 何が問題か 
class Parent {
    Parent(String s) {}
}
class Child extends Parent {
    Child() {}
}

答え: Child() の先頭で暗黙に super() が呼ばれますが、Parent に引数なしコンストラクタがないためコンパイルエラーです。

問4: オーバーライドかオーバーロードか 
class A {
    void test(Object o) {}
}
class B extends A {
    void test(String s) {}
}

答え: オーバーロードです。引数型が違うため、test(Object) を再定義していません。

まとめ

  • 継承は extends、interface実装は implements。
  • サブクラスのインスタンスはスーパークラス型で扱える。
  • 呼べるかどうかは宣言型で決まる。
  • overrideされたインスタンスメソッドの実行本体は実体で決まる。
  • フィールドとstaticメソッドは通常のoverrideにならない。
  • コンストラクタは継承されない。
  • interfaceの抽象メソッドは暗黙に public abstract。
  • interfaceのフィールドは暗黙に public static final。

Silverでは「継承しているから使える」と雑に読まないでください。宣言型、実体、アクセス修飾子、メンバ種別の4つで機械的に切るのが正解です。
