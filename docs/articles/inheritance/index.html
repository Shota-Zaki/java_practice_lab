継承・オーバーライド・フィールド隠蔽

継承問題は「変数の型」と「実体の型」を分けて考えます。フィールドは変数の型、オーバーライドメソッドは実体の型で決まる、という区別が重要です。

継承・interface・overrideを読むための基礎概念

継承の問題は、まず「どちらが土台で、どちらが拡張先か」を固定します。class Child extends Parent なら、Parent がスーパークラス、Child がサブクラスです。extends の左側が新しく作る側、右側が土台になる側です。

用語別名意味コード上の位置
スーパークラス親クラス / 基底クラス継承元。共通のフィールドやメソッドを持つ側。extends の右側
サブクラス子クラス / 派生クラス継承先。スーパークラスを土台に差分を追加する側。extends の左側
 
class Parent {
    String name = "parent";
    void hello() {
        System.out.println("hello " + name);
    }
}

class Child extends Parent {
    void childOnly() {
        System.out.println("child");
    }
}

Child のインスタンスは、Parent として扱える部分と、Child で追加した部分をあわせて持ちます。そのため、次のようにサブクラスのインスタンスをスーパークラス型の変数に代入できます。

Parent p = new Child();
p.hello();
// p.childOnly(); // コンパイルエラー: 宣言型ParentにはchildOnlyが見えない
重要: 実体が Child でも、変数の宣言型が Parent なら、Parent に見えているメンバしか呼べません。呼べるかどうかは宣言型で判定し、overrideされたインスタンスメソッドの実行内容は実体で判定します。

is-a関係で読む

継承は「サブクラスはスーパークラスの一種」として読める関係です。Dog extends Animal なら「Dog is an Animal」と読めます。だから Animal a = new Dog(); は自然に成立します。

class Animal { void eat() {} }
class Dog extends Animal { void bark() {} }

Animal a = new Dog(); // OK: DogはAnimalの一種
// Dog d = new Animal(); // NG: Animalが必ずDogとは限らない

継承されるもの・されないもの・直接アクセスできないもの

対象サブクラスでの扱い注意点
public / protected / 条件を満たすdefaultメンバ利用できるアクセス修飾子とパッケージ条件を確認する。
privateフィールド・privateメソッド直接アクセス不可親の内部には存在するが、子から直接触れない。
コンストラクタ継承されないsuper(...) で呼び出すだけ。子のコンストラクタとしては使えない。
同名フィールドoverrideではないフィールドは宣言型で解決される。
インスタンスメソッドoverride可能呼び出し可否は宣言型、実行本体は実体で決まる。
staticメソッドoverrideではなく隠蔽インスタンスメソッドとは解決規則が違う。

this と super の違い

this は現在のインスタンス側を指します。super はスーパークラス側のコンストラクタやメンバを明示するために使います。ただし、super(...) と super.member は役割が違います。

書き方意味使える場所
this.field現在のクラス側のフィールドを指す。インスタンスメソッド・コンストラクタ内
this(...)同じクラスの別コンストラクタを呼ぶ。コンストラクタ先頭のみ
super(...)スーパークラスのコンストラクタを呼ぶ。コンストラクタ先頭のみ
super.field / super.method()スーパークラス側のメンバを参照する。インスタンスメソッド・コンストラクタ内
 
class Parent {
    String name = "parent";
    Parent(String name) {
        this.name = name;
    }
    void print() {
        System.out.println(name);
    }
}

class Child extends Parent {
    String name = "child";

    Child() {
        super("super-name"); // 親コンストラクタ呼び出し。必ず先頭
    }

    void printBoth() {
        System.out.println(this.name);  // Child側のname
        System.out.println(super.name); // Parent側のname
        super.print();                  // Parent側のメソッド呼び出し
    }
}
落とし穴: super(...) はコンストラクタ呼び出しです。super.name や super.print() はメンバ参照です。同じ super でも、書ける場所と意味を混ぜないでください。

この単元で最初に固定する考え方

継承・interfaceの問題は、用語を知っているだけでは足りません。毎回、次の順で切ります。

  1. クラス関係を見る。extends か implements か、単一継承か複数interfaceか。
  2. 宣言型を見る。変数の型から、その場で呼べるメソッドが決まります。
  3. 実体を見る。overrideされたインスタンスメソッドは、実体のクラスで動きます。
  4. アクセス修飾子を見る。継承していることと、直接アクセスできることは別です。
  5. コンストラクタ連鎖を見る。コンストラクタは継承されず、生成時には必ず親のコンストラクタが先に動きます。

雑に「子クラスだから親の全部が使える」と考えると落ちます。Javaでは、private、default、protected、コンストラクタ、フィールド解決で崩されます。

スーパークラスとサブクラス

継承を読むときは、まず「どちらが元で、どちらが拡張先か」を固定します。class Child extends Parent なら、Parent がスーパークラス、Child がサブクラスです。

用語別名意味コード上の位置
スーパークラス親クラス / 基底クラス継承元。共通のフィールドやメソッドを持つ側extends の右側
サブクラス子クラス / 派生クラス継承先。スーパークラスを土台に差分を追加する側extends の左側
 
class Parent {          // スーパークラス
    String name = "parent";
    void hello() {
        System.out.println("hello " + name);
    }
}

class Child extends Parent { // サブクラス
    void childOnly() {
        System.out.println("child");
    }
}

Child のインスタンスは、Parent として扱える部分と、Child で追加した部分をあわせて持ちます。だから Parent p = new Child(); のように、サブクラスのインスタンスをスーパークラス型の変数に代入できます。

Parent p = new Child();
p.hello();
// p.childOnly(); // NG: 変数の宣言型ParentにはchildOnlyがない

ここで重要なのは、インスタンスが持っている機能と、変数の型から呼べる機能は別だという点です。実体が Child でも、変数の宣言型が Parent なら、Parent に見えているメンバしか呼べません。

継承は「is-a関係」として読む

サブクラスはスーパークラスの一種として扱えます。これを is-a関係 と考えます。Child extends Parent なら「Child は Parent の一種」と読める関係です。

class Animal {
    void eat() {}
}

class Dog extends Animal {
    void bark() {}
}

Animal a = new Dog(); // OK: DogはAnimalの一種
// Dog d = new Animal(); // NG: Animalが必ずDogとは限らない

この非対称性が重要です。サブクラスからスーパークラスへの代入は自然にできますが、スーパークラスからサブクラスへの代入にはキャストが必要です。そして、実体がそのサブクラスでなければ実行時に ClassCastException になります。

問題での読み方

A a = new B(); が出たら、まず B extends A または B implements A の関係があるか関係がなければコンパイルエラーです。関係がある場合でも、 a から呼べるのは宣言型 A にあるメンバだけ

継承されるもの・されないもの・直接アクセスできないもの

「サブクラスはスーパークラスを継承する」という表現を雑に覚えると危険です。問題では、何が使えて何が使えないかを分けて問われます。

対象サブクラスでどう扱うか注意点
public / protected / 条件を満たすdefaultメンバ利用できるアクセス修飾子とパッケージ条件を見る
privateフィールド・privateメソッド直接アクセス不可親の内部には存在するが、子から直接触れない
コンストラクタ継承されないsuper(...) で呼び出すだけ
同名フィールドoverrideではないフィールドは宣言型で解決される
インスタンスメソッドoverride可能呼び出し可否は宣言型、実行される本体は実体で決まる
staticメソッドoverrideではなく隠蔽インスタンスメソッドとは解決規則が違う

特に private とコンストラクタは「継承しているから使える」という雑な理解を壊すためによく出ます。ここを曖昧にしたまま問題を解くと、コンパイルエラー判定を外します。

this と super の役割

this は現在のインスタンス、super はスーパークラス側のメンバやコンストラクタを指すために使います。

class Parent {
    String name = "parent";
    Parent(String name) {
        this.name = name;
    }
    void print() {
        System.out.println(name);
    }
}

class Child extends Parent {
    String name = "child";

    Child() {
        super("super-name"); // 親コンストラクタ呼び出し。必ず先頭
    }

    void printBoth() {
        System.out.println(this.name);  // Child側のname
        System.out.println(super.name); // Parent側のname
        super.print();                  // Parent側のメソッド呼び出し
    }
}

super(...) はコンストラクタ呼び出しで、コンストラクタの先頭にしか書けません。super.name や super.print() はスーパークラス側のメンバ参照です。同じ super でも使い方を混ぜないでください。

extends：クラス継承の基本

extends は、既存クラスを拡張してサブクラスを作る構文です。左側のクラスがサブクラス、右側のクラスがスーパークラスです。サブクラスのインスタンスは、スーパークラス部分とサブクラス部分を持ちます。

class Parent {
    String name = "parent";
    void hello() {
        System.out.println("hello " + name);
    }
}

class Child extends Parent {
    void childOnly() {
        System.out.println(name); // Parentから継承されたフィールドを使える
    }
}

ここで重要なのは、継承されたメンバがあるから使えるという点です。もし Parent に name がなく、Child にも name がなければ、name は未定義なのでコンパイルエラーです。

コンストラクタは継承されない

継承で引き継がれない代表がコンストラクタです。サブクラスのインスタンス生成時には、サブクラスのコンストラクタが呼ばれ、その先頭でスーパークラスのコンストラクタが呼ばれます。

class A {
    A(String value) {
        System.out.println("A:" + value);
    }
}

class B extends A {
    B() {
        super("b");
        System.out.println("B");
    }
}

new B() の実行順は A:b → B です。B が A(String) を継承して new B("x") できるわけではありません。

ひっかけ：親に引数なしコンストラクタがない場合

子クラスのコンストラクタで super(...) を書かないと、コンパイラは暗黙に super() を入れます。親に A() がなければコンパイルエラー

privateメンバは「存在する」が「直接使えない」

privateフィールドやprivateメソッドは、サブクラスから直接アクセスできません。ただし、親クラスの内部からは当然使えます。

class A {
    private String value = "A";
    public String getValue() {
        return value;
    }
}

class B extends A {
    void print() {
        // System.out.println(value); // NG: privateなので直接アクセス不可
        System.out.println(getValue()); // OK: publicメソッド経由
    }
}

問題では「継承しているからprivateも使える」と思わせる選択肢が出ます。これは切ってください。privateは同じクラス内だけです。

implements：interfaceの実装

implements は、interfaceで定義された使い方をクラスに実装させる構文です。クラスは1つのクラスしか継承できませんが、複数のinterfaceを実装できます。

interface Printable {
    void print(); // public abstract扱い
}

interface Savable {
    void save();
}

class Report implements Printable, Savable {
    public void print() {
        System.out.println("print");
    }
    public void save() {
        System.out.println("save");
    }
}

interfaceの抽象メソッドは暗黙的に public abstract です。そのため、実装クラス側のメソッドも public にしなければいけません。アクセスを狭くするとコンパイルエラーです。

抽象クラスとinterfaceの違い

観点抽象クラスinterface
継承方法extendsimplements
複数指定不可。クラス継承は1つだけ可。複数実装できる
インスタンス化不可不可
フィールド通常フィールドを持てる暗黙に public static final
コンストラクタ持てる持てない
用途共通実装・共通状態を持たせる外部からの扱い方を定義する

抽象クラスは「未完成のクラス」です。interfaceは「扱い方・契約」です。この区別を曖昧にすると、コンストラクタやフィールドの問題で間違えます。

abstractメソッドを実装しない場合

抽象メソッドを継承した具象クラスは、そのメソッドを実装しなければコンパイルエラーです。実装しないなら、そのクラス自身も abstract にする必要があります。

abstract class A {
    abstract void test();
}

abstract class B extends A {
    // testを実装しないならBもabstract
}

class C extends B {
    void test() {
        System.out.println("C");
    }
}

overrideの成立条件

overrideは「同じ名前のメソッドを書いた」だけでは成立しません。次の条件を同時に満たす必要があります。

条件正しい形誤りやすい形
シグネチャメソッド名と引数リストが同じ引数型が違うとoverload
戻り値型同じ、または戻り値が参照型ならサブタイプ親より広い戻り値型は不可
アクセス同じか、より広いpublic を protected に狭めるのは不可
チェック例外同じか、より狭いより広いチェック例外を追加不可
staticstatic同士なら隠蔽staticとインスタンスの相互変更は不可
 
class A {
    protected Number test() throws java.io.IOException {
        return 1;
    }
}

class B extends A {
    @Override
    public Integer test() throws java.io.FileNotFoundException {
        return 1;
    }
}

上の例はOKです。アクセスは protected から public に広がり、戻り値は Number から Integer に具体化され、チェック例外も狭くなっています。

overloadとoverrideを混同しない

引数が違えばoverloadです。override問題で、選択肢の引数型が少し変わっているだけなら、それはoverrideではありません。

class A {
    void test(int x) {}
}

class B extends A {
    void test(long x) {} // overrideではなくoverload
}

@Override を付けた場合、overrideになっていなければコンパイルエラーです。問題では @Override の有無が大きなヒントです。

interface型で受ける意味

interface型の変数で受けると、そのinterfaceに宣言されたメソッドだけを呼べます。実体クラスに独自メソッドがあっても、宣言型に存在しなければ呼べません。

interface A {
    void test();
}

class B implements A {
    public void test() {}
    void onlyB() {}
}

A a = new B();
a.test();
// a.onlyB(); // NG: A型にはonlyBがない

これはポリモーフィズムの基本です。「実体にあるから呼べる」ではありません。呼べるかは宣言型、実行される中身は実体です。

フィールドはoverrideされない

メソッドはoverrideされますが、フィールドはoverrideされません。同名フィールドを子クラスに定義すると、隠蔽に近い形になります。

class A {
    String name = "A";
    String getName() { return name; }
}

class B extends A {
    String name = "B";
    String getName() { return name; }
}

A obj = new B();
System.out.println(obj.name);      // A: フィールドは宣言型
System.out.println(obj.getName()); // B: メソッドは実体

Javaの出力問題では、ここが頻出です。フィールドアクセスとメソッド呼び出しを同じルールで追うと誤答します。

メソッド解決の判断表

式まず見るもの実際の解決
obj.field変数の宣言型宣言型のフィールド
obj.method()宣言型にそのメソッドがあるかoverrideされていれば実体のメソッド
ClassName.staticMethod()指定したクラスstaticメソッド。overrideではなく隠蔽
((B)obj).method()キャスト後の型実行時にキャスト可能なら呼べる

ミニ演習

Q1. class B extends A のとき、AのコンストラクタはBに継承される？

されません。Bのコンストラクタ内から super(...) でAのコンストラクタを呼ぶだけ

Q2. interfaceの抽象メソッドを実装するとき、アクセス修飾子を省略できる？

できません。interface側は暗黙に public なので、実装側も public が必要

Q3. A a = new B(); で、Bだけにあるメソッドを呼べる？

宣言型Aにそのメソッドがなければ呼べません。必要なら ((B)a).method() のようにキャストします。ただし実体がBでなければ実行時例外

interfaceの暗黙修飾子

interfaceに書いたメンバは、クラスと同じ感覚で読むと間違えます。抽象メソッドとフィールドには暗黙の修飾子があります。

書いた形実質的な意味注意点
void test();public abstract void test();実装側は public 必須
int MAX = 10;public static final int MAX = 10;代入し直せない
default void test(){}実装付きインスタンスメソッドpublic 扱い。private/protected不可
static void test(){}interfaceに属するstaticメソッド実装クラスへ継承されない
private void helper(){}interface内部の補助メソッド外部から呼べない。実装クラスからも呼べない
 
interface A {
    int VALUE = 10;
    void run();
}

class B implements A {
    public void run() {          // publicが必要
        // VALUE = 20;           // NG: static final
        System.out.println(VALUE);
    }
}

defaultメソッドの基本

defaultメソッドは、interfaceに実装を持たせる仕組みです。実装クラスがoverrideしなければ、そのまま使えます。

interface A {
    default void test() {
        System.out.println("A");
    }
}

class B implements A { }

new B().test(); // A

ただし、defaultメソッドは「interfaceの通常メソッド」ではありません。default を付けずに本体を書くとコンパイルエラーです。

interface A {
    // void test() { } // NG: default, static, privateのいずれでもない実装付きメソッド
}

defaultメソッドの衝突

複数のinterfaceから同じシグネチャのdefaultメソッドを受け継ぐと、クラス側でどちらを使うか決めなければいけません。

interface A {
    default void test() { System.out.println("A"); }
}
interface B {
    default void test() { System.out.println("B"); }
}

class C implements A, B {
    public void test() {
        A.super.test();
    }
}

A.super.test() は、interfaceのdefault実装を明示的に呼ぶ構文です。super.test() ではありません。

クラスの具象メソッドがdefaultより優先される

スーパークラスに具象メソッドがあり、interfaceにも同じシグネチャのdefaultメソッドがある場合、クラス側が優先されます。

class A {
    public void test() {
        System.out.println("class");
    }
}

interface B {
    default void test() {
        System.out.println("interface");
    }
}

class C extends A implements B { }

new C().test(); // class

優先順位は、ざっくり「クラスが勝つ」です。defaultメソッドは、クラス階層に実装がないときの補助と考えると判断しやすくなります。

interfaceのstaticメソッドは継承されない

interfaceのstaticメソッドは、そのinterface名で呼び出します。実装クラス名やインスタンスから呼ぶものではありません。

interface A {
    static void test() {
        System.out.println("A");
    }
}

class B implements A { }

A.test();
// B.test();       // NG
// new B().test(); // NG

クラスのstaticメソッドとは感覚が違います。interface staticは「継承されない」と覚えてください。

interfaceのprivateメソッド

privateメソッドは、defaultメソッドやstaticメソッド内の共通処理を切り出すためのものです。実装クラスから呼ぶためのものではありません。

interface A {
    default void test() {
        log();
    }

    private void log() {
        System.out.println("log");
    }
}

class B implements A {
    void sample() {
        // log(); // NG: interfaceのprivateメソッドは呼べない
    }
}

interface同士の継承

interfaceはinterfaceを extends で継承します。クラスのように implements は使いません。また、interfaceは複数のinterfaceを継承できます。

interface A {
    void a();
}

interface B {
    void b();
}

interface C extends A, B {
    void c();
}

class D implements C {
    public void a() {}
    public void b() {}
    public void c() {}
}

D は C だけでなく、A と B の抽象メソッドも実装する必要があります。

抽象クラスはinterfaceメソッドを未実装のままにできる

interfaceを実装したクラスが抽象クラスなら、すべての抽象メソッドを実装しなくても構いません。ただし、最終的な具象クラスでは実装が必要です。

interface A {
    void test();
}

abstract class B implements A {
    // testを実装しなくてもOK。B自身がabstractだから
}

class C extends B {
    public void test() {
        System.out.println("C");
    }
}

protectedの別パッケージアクセス

protected は「同じパッケージならアクセス可」「別パッケージでも継承関係があれば条件付きでアクセス可」です。別パッケージのサブクラスから、親型の別インスタンス経由で自由に触れるわけではありません。

// package p1;
public class A {
    protected void test() {}
}

// package p2;
class B extends A {
    void ok() {
        test();       // OK: 自分自身の継承メンバとして呼ぶ
        this.test();  // OK
    }

    void ng(A a) {
        // a.test();  // NGになり得る: 別パッケージで親型参照経由
    }
}

protectedは「サブクラスなら何でもOK」ではありません。この誤解はかなり危険です。

sealedクラス・sealed interface

sealed は、継承・実装できるクラスを制限する仕組みです。許可する相手は permits で指定します。

sealed interface Shape permits Circle, Rectangle { }

final class Circle implements Shape { }
non-sealed class Rectangle implements Shape { }

sealedを継承・実装した側は、次のいずれかを明示する必要があります。

修飾子意味使う場面
finalここで継承を止めるこれ以上派生させない
sealedさらに許可先を制限する階層を引き続き管理する
non-sealed制限を解除するここから先は通常の継承に戻す
ひっかけ：permitsに書かれていないクラス

許可されていないクラスがsealedクラスを継承・実装するとコンパイルエラー

sealedの典型コンパイルエラー

sealed class A permits B { }

// class C extends A { } // NG: permitsにCがない

// class B extends A { } // NG: final/sealed/non-sealedのいずれかが必要

final class B extends A { } // OK

問題では、許可リストだけでなく、許可されたサブクラス側の修飾子も見ます。片方だけ見ても足りません。

メソッドとフィールドの解決を分ける

継承/interfaceの詳細問題では、出力追跡と組み合わされます。特に、同名フィールドとoverrideメソッドの混在に注意します。

class A {
    String value = "A";
    String getValue() { return value; }
}

class B extends A {
    String value = "B";
    String getValue() { return value; }
}

A a = new B();
System.out.println(a.value);      // A
System.out.println(a.getValue()); // B

フィールドは宣言型、インスタンスメソッドは実体。ここは機械的に分けてください。

defaultメソッドとObjectメソッド

interfaceのdefaultメソッドでは、Object の equals、hashCode、toString などをdefault実装として提供することはできません。すべてのクラスがObjectを継承するため、クラス側のルールと衝突するからです。

interface A {
    // default String toString() { return "A"; } // NG
}

これは細かいですが、interface問題で出されると落としやすい論点です。

総合判断フロー

  1. 宣言を読む：class / abstract class / interface / sealed interface を分類する。
  2. 継承関係を読む：extends / implements / permits を線でつなぐ。
  3. 未実装メソッドを見る：具象クラスが抽象メソッドを残していないか確認する。
  4. メソッド修飾子を見る：public不足、static/instance不一致、戻り値型、例外を確認する。
  5. 衝突を見る：複数default、クラス具象メソッド、同名フィールドを確認する。
  6. 実行を追う：コンパイルできる場合だけ、宣言型と実体を分けて出力を追う。
