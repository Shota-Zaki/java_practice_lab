Java Silverのコンストラクタで落ちるパターン

コンストラクタ問題は、呼び出し順、this() / super()、フィールド初期化、親コンストラクタの有無を確認します。

1. 親から先に実行される

class A { A() { System.out.print("A"); } }
class B extends A { B() { System.out.print("B"); } }
new B(); // AB

子オブジェクトを作るときでも、先に親部分が初期化されます。

2. super()は暗黙に入る

class A { A() { System.out.print("A"); } }
class B extends A { B() { System.out.print("B"); } }

B() の先頭には暗黙に super() が入ります。

3. 親に引数なしコンストラクタがない

class A { A(int x) {} }
class B extends A { B() {} } // コンパイルエラー

暗黙の super() が呼べないためです。super(1) のように明示します。

4. this()とsuper()は同時に書けない

B() {
    this(1);
    super(); // コンパイルエラー
}

どちらも先頭文でなければならないため、同じコンストラクタ内で両方は書けません。

5. フィールド初期化とコンストラクタ

class Sample {
    int x = 10;
    Sample() { x = 20; }
}

フィールド初期化の後にコンストラクタ本体が実行されます。最終的な x は 20 です。

確認ポイント

  • 親に引数なしコンストラクタがあるか。
  • this() / super() が先頭か。
  • 出力順は親から子。
  • フィールド初期化よりコンストラクタ本体が後。

演習前チェック

この記事を演習で使える知識に変える

対象レベル：Java11 / Java17 Silver共通。Silver共通範囲では、バージョン固有構文に逃げず、型、参照、制御構文、クラス、継承、例外の基本ルールだけで判定します。Java17固有の記事はJava17 Silver側に分けています。

本文で必ず押さえること

  • インスタンスに属するものと、クラスに属するstaticなものを分ける。
  • コンストラクタは戻り値型を書かない。メソッドと見た目が似ていても別物。
  • オーバーロードはコンパイル時に引数型で決まる。実行時のオーバーライドとは分ける。

この後の演習で見るポイント

  • 呼び出し元の変数型、引数型、アクセス修飾子を順に確認する。
  • thisとsuperが何を指しているかを明確にする。
  • コンストラクタチェーンでは、最初にどのコンストラクタへ制御が移るかを見る。

ここで落ちる典型ミス

  • staticメソッドからインスタンスフィールドへ直接触れると思い込む。
  • privateメンバが継承先から自由に使えると考える。
  • 同名メソッドを見てすぐオーバーライドと判断する。引数が違えばオーバーロード。

演習前の小課題

同じ名前のメソッドが複数あるコードでは、先に「候補メソッド一覧」を作り、引数型でどれが選ばれるかを判定してください。
