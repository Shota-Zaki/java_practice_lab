参照の同一性と同値比較

参照比較は最も間違えやすい分野の一つです。 == は「同じものを指しているか」、 equals() は「同じ内容とみなすか」を確認するためのものです。ただし、 equals() の挙動はクラスによって変わります。

この単元で固定する判断

== と equals は、Javaで最も誤答が出やすい比較論点です。最初に結論だけ覚えるのではなく、次の順番で必ず判定します。

  1. 比較している対象はプリミティブ型か、参照型か。 プリミティブ型なら値比較、参照型なら原則として参照比較です。
  2. 片方がプリミティブ、片方がラッパー型ではないか。 この場合はアンボクシングが起きる可能性があり、null なら実行時例外です。
  3. equals を呼び出している左辺が null ではないか。 null.equals(...) はコンパイルエラーではなく NullPointerException です。
  4. そのクラスが equals をオーバーライドしているか。 String や record は内容比較ですが、自作クラスや配列、StringBuilder は要注意です。

雑に「== は参照、equals は中身」と覚えると落ちます。equals もメソッドなので、クラスごとの実装に従います。

== の基本: プリミティブ型は値、参照型は参照

比較対象== の意味問題での見方
プリミティブ型同士値が等しいか型変換後の値を見る
参照型同士同じインスタンスを参照しているかnew、代入、文字列プールを見る
参照型と null参照先がないかobj == null は安全
ラッパー型とプリミティブ型アンボクシング後に値比較ラッパーが null なら実行時例外
 
int a = 10;
long b = 10L;
System.out.println(a == b); // true: 数値として比較される

String x = new String("A");
String y = new String("A");
System.out.println(x == y); // false: 別インスタンス

String z = x;
System.out.println(x == z); // true: 参照のコピー

String x = y; のような代入は、文字列の中身をコピーしているわけではありません。参照をコピーしています。したがって x == y は同じ参照かどうかの判定になります。

数値比較では型昇格を先に見る

プリミティブ型同士の == では、必要に応じて数値昇格が起きます。byte、short、char は演算や比較の場面で int に昇格することがあります。

char c = 'A';     // 文字コードは65
int n = 65;
System.out.println(c == n); // true

byte b1 = 10;
byte b2 = 10;
System.out.println(b1 == b2); // true

boolean は数値型ではありません。true == 1 のような比較はできません。ほかの言語経験がある人ほどここで雑に解釈しがちです。

boolean flag = true;
// System.out.println(flag == 1); // コンパイルエラー

equals は「内容比較」ではなく「メソッド呼び出し」

equals は演算子ではありません。メソッドです。したがって、次の2点を必ず確認します。

  • 左辺が null なら NullPointerException
  • どのクラスの equals が呼ばれるかで結果が変わる
String s = null;
// System.out.println(s.equals("A")); // NullPointerException
System.out.println("A".equals(s));    // false

問題では「コンパイルエラー」ではなく「実行時例外」になる点が狙われます。equals の呼び出し自体は文法上正しくても、実行時に左辺が null なら落ちます。

Object.equals とオーバーライド

Object クラスの equals は、基本的に同じオブジェクトかどうかを判定します。つまり、自作クラスで equals をオーバーライドしていない場合、フィールドの値が同じでも false になります。

class Item {
    int id;
    Item(int id) { this.id = id; }
}

Item a = new Item(1);
Item b = new Item(1);
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // false: Objectのequals

「フィールドの値が同じだから equals は true」と考えるのは危険です。クラス側が内容比較を実装しているかを見る必要があります。

String の比較: 文字列プールと new String

String は内容比較の equals を持っています。一方、== はあくまで参照比較です。文字列リテラルは文字列プール上で共有されるため、同じリテラル同士なら == が true になることがあります。

String a = "java";
String b = "java";
String c = new String("java");

System.out.println(a == b);      // true: 同じリテラル
System.out.println(a == c);      // false: cはnewで別インスタンス
System.out.println(a.equals(c)); // true: 文字列内容は同じ

new String("java") は、明示的に新しい String インスタンスを作ります。中身が同じでも、参照先はリテラルとは別です。

コンパイル時定数の文字列結合

文字列結合では、すべてがコンパイル時定数ならコンパイル時に結合され、文字列プール上の同じリテラルとして扱われます。一方、変数を含む結合は実行時に新しい文字列が作られると考えます。

String a = "Ja" + "va";
String b = "Java";
System.out.println(a == b); // true: コンパイル時に"Java"へ結合

String x = "Ja";
String y = x + "va";
System.out.println(y == b);      // false: 実行時結合
System.out.println(y.equals(b)); // true

final でコンパイル時定数になる場合は、リテラル結合と同じように扱われることがあります。ここは「その変数がコンパイル時定数か」を見る論点です。

final String x = "Ja";
String y = x + "va";
String z = "Java";
System.out.println(y == z); // true: xはコンパイル時定数

intern() の見方

intern() は、文字列プール上の同じ内容の String への参照を返します。new String で作ったインスタンスそのものが返ると決めつけないでください。

String a = new String("sample");
String b = a;
String c = a.intern();
String d = "sample";

System.out.println(a == b); // true: 参照コピー
System.out.println(a == c); // false: aはヒープ上のnew String
System.out.println(c == d); // true: どちらも文字列プール側

このタイプの問題は、各変数が「ヒープ上の new インスタンス」なのか「文字列プール上のインスタンス」なのかを表で追うのが最も確実です。

StringBuilder の equals は罠

StringBuilder は可変オブジェクトですが、String のような内容比較の equals を持つと雑に覚えると危険です。通常の試験問題では、StringBuilder の equals は参照比較として扱います。

StringBuilder a = new StringBuilder("A");
StringBuilder b = new StringBuilder("A");
StringBuilder c = a;

System.out.println(a == b);      // false
System.out.println(a.equals(b)); // false
System.out.println(a == c);      // true
System.out.println(a.equals(c)); // true

StringBuilder の中身を文字列として比較したいなら、toString() して String.equals を使う発想になります。

配列の比較

配列はオブジェクトです。== は参照比較です。また、配列の equals も、要素の内容比較として読んではいけません。

int[] a = {1, 2};
int[] b = {1, 2};
int[] c = a;

System.out.println(a == b);      // false
System.out.println(a.equals(b)); // false
System.out.println(a == c);      // true

配列の中身比較には java.util.Arrays.equals がありますが、Javaの選択肢では「配列の equals が要素比較になる」と誤認させる形が出やすいです。

ラッパークラス比較: オートボクシングと null

Integer などのラッパークラスは参照型です。ただし、プリミティブ型と比較するとアンボクシングが起き、値比較になります。

Integer a = 10;
int b = 10;
System.out.println(a == b); // true: aがintへアンボクシング

Integer c = null;
// System.out.println(c == b); // NullPointerException

ラッパー同士の == は参照比較です。小さい整数でキャッシュにより true になるケースはありますが、実務的にも学習対策としても、値比較には equals を使うと整理してください。

Integer x = 1000;
Integer y = 1000;
System.out.println(x == y);      // falseと考えるのが基本
System.out.println(x.equals(y)); // true

record の比較

record は、コンポーネントに基づく equals、hashCode、toString が自動生成されます。== は通常どおり参照比較です。

record Point(int x, int y) {}

Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);
System.out.println(p1 == p2);      // false
System.out.println(p1.equals(p2)); // true

recordは「データを持つクラス」なので、equals が中身比較になる代表例です。ただし、== まで中身比較になるわけではありません。

ミニ演習

Q1. 次の出力は？
String a="ab"; String b="a"+"b"; System.out.println(a==b);

true 。リテラル同士の結合はコンパイル時に "ab" になります

Q2. 次の出力は？
String a="a"; String b=a+"b"; System.out.println(b=="ab");

false 。 a が通常の変数なので、実行時結合として考えます

Q3. 次のコードは？
Integer i=null; System.out.println(i==0);

実行時例外 。 i を int にアンボクシングしようとして NullPointerException

Q4. 次の出力は？
StringBuilder a=new StringBuilder("x"); StringBuilder b=a; a.append("y"); System.out.println(b);

xy 。 a と b は同じ可変オブジェクトを参照しています
