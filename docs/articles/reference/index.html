Java Silverで間違えやすい == と equals の違い

== と equals() の違いは、Java Silverで最重要です。値が同じに見えても、参照先が違えば == は false になります。逆に、equals() はクラスがどう実装しているかで結果が変わります。

1. 参照変数とオブジェクトは別物

参照型変数は、オブジェクトそのものを持っているわけではありません。オブジェクトの場所を指す情報、つまり参照を持っています。まずここを間違えると、比較問題はほぼ崩れます。

Sample s1 = new Sample(10); // 新しいSampleオブジェクトAを作り、s1が参照する
Sample s2 = s1;             // s2はs1と同じオブジェクトAを参照する
s1 = new Sample(10);        // 新しいSampleオブジェクトBを作り、s1だけ参照先を変える
System.out.println(s1 == s2); // false
行状態重要点
new Sample(10)ヒープに新しいインスタンスが作られるnumが同じでも、newすれば別インスタンス。
Sample s2 = s1参照のコピーオブジェクトをコピーしていない。同じものを見る変数が2つになるだけ。
s1 = new Sample(10)s1の参照先だけ変更s2は古いオブジェクトAを見続ける。
s1 == s2参照先が同じか確認中身のnumは見ない。参照が違うのでfalse。
危険な理解：「中の値が同じなら同じオブジェクト」。これは誤りです。newを2回使ったら、基本的に別オブジェクトです。

2. == は何を比較するか

対象== の意味例
基本データ型値そのものを比較10 == 10 は true
参照型同じインスタンスを参照しているか比較new Sample(10) == new Sample(10) は false
null参照先がないか比較obj == null はよく使う
int a = 10;
int b = 10;
System.out.println(a == b); // true: 基本データ型は値比較

Sample x = new Sample(10);
Sample y = new Sample(10);
System.out.println(x == y); // false: 参照型は同一性比較

3. equals() はクラス次第

equals() は「中身が同じか」を見るメソッドだと思われがちですが、それは半分だけ正しいです。正確には、そのクラスが equals() をどう実装しているかで決まります。

クラスequals() の傾向注意
自作クラスで未オーバーライドObject.equals() のまま。実質 ==フィールド値が同じでも false
String文字列の内容を比較new String("A").equals("A") は true
StringBuilder基本的に同一性比較内容が同じでも別インスタンスなら false
配列equals() は基本的に同一性比較中身比較は Arrays.equals()
recordコンポーネント値で比較する equals() が自動生成recordは値比較に近い感覚で扱える

4. 自作クラスで値比較したい場合

自作クラスで「num が同じなら同じ」と判断したいなら、equals() と hashCode() を実装します。

import java.util.Objects;

public class Sample {
    private final int num;

    public Sample(int num) {
        this.num = num;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (!(obj instanceof Sample other)) return false;
        return this.num == other.num;
    }

    @Override
    public int hashCode() {
        return Objects.hash(num);
    }
}

この実装をすると、new Sample(10).equals(new Sample(10)) は true になります。ただし、== は相変わらず false です。ここを混同してはいけません。

5. Stringで特に注意すること

String a = "Java";
String b = "Java";
String c = new String("Java");

System.out.println(a == b);      // true  文字列リテラルはプールされる
System.out.println(a == c);      // false newで別インスタンス
System.out.println(a.equals(c)); // true  文字内容は同じ

String は文字列プールがあるため、== がたまたま true になる場面があります。しかし、試験では「だから文字列比較は == でよい」と考えた瞬間に落とされます。文字列の内容比較は equals() です。

6. 配列の比較

int[] a = {1, 2, 3};
int[] b = {1, 2, 3};

System.out.println(a == b);              // false
System.out.println(a.equals(b));         // false
System.out.println(Arrays.equals(a, b)); // true

配列の equals() は、中身比較ではありません。一次元配列の中身比較は Arrays.equals()、多次元配列なら Arrays.deepEquals() を使います。

7. メソッド引数は参照のコピー

メソッドにオブジェクトを渡すと、参照そのものがコピーされます。メソッド内で参照先オブジェクトのフィールドを書き換えると呼び出し元にも影響します。しかし、引数変数に別オブジェクトを再代入しても、呼び出し元の変数は変わりません。

static void changeValue(Sample s) {
    s.num = 20;          // 参照先の中身を変更するので外側にも影響する
}

static void reassign(Sample s) {
    s = new Sample(30);  // 引数変数の参照先を変えるだけ。外側の変数は変わらない
}

8. 判定チェックリスト

  1. 比較対象は基本型か参照型か。 基本型なら値比較、参照型なら参照比較。
  2. new が何回出てくるか。 new が複数回なら別インスタンスの可能性が高い。
  3. 代入は参照コピーか。 s2 = s1 は同じオブジェクトを見るだけ。
  4. equals() がオーバーライドされているか。 自作クラス未実装なら実質 ==。
  5. String、record、配列は特別扱い。 Stringは内容比較、recordは自動equals、配列はArrays系。
