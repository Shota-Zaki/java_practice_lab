頻出メソッドリスト 詳細版

メソッド問題は、戻り値、元オブジェクトを変更するか、例外条件、添字の範囲

メソッド呼び出しの基本部品

メソッド問題では、呼び出し側の値と、受け取る側の変数を混同しないことが重要です。特にオーバーロード、戻り値、値渡し、コンストラクタで差が出ます。

用語意味例
呼び出し先実行されるメソッド。print("A") の print
実引数呼び出し時に渡す値。print("A") の "A"
仮引数メソッド側で受け取る変数。void print(String s) の s
戻り値メソッドが呼び出し元に返す値。return s.length();
シグネチャメソッド名と引数リスト。print(String)
 
static int lengthOf(String text) { // textは仮引数
    return text.length();          // intを戻す
}

int n = lengthOf("abc");          // "abc"は実引数

オーバーロード判定では、戻り値だけでは区別できません。見るのはメソッド名と引数リストです。

コンストラクタはメソッドではない

コンストラクタは、インスタンス生成時に初期化を行う特別な構文です。名前はクラス名と同じで、戻り値型を書きません。戻り値型を書いた瞬間、それはコンストラクタではなく通常メソッドになります。

観点メソッドコンストラクタ
名前任意クラス名と同じ
戻り値型必要。void も戻り値型。書かない
呼び出し通常のメソッド呼び出しnew 時、または this(...) / super(...)
継承条件を満たせば継承・override可能継承されない
 
class Sample {
    Sample() { }      // コンストラクタ
    void Sample() { } // メソッド。コンストラクタではない
}
落とし穴: void Sample() はコンストラクタに見えますが、戻り値型 void を持つ通常メソッドです。

この単元のゴール

メソッド・コンストラクタの問題は、コード量が少なくても罠が多いです。特に オーバーロード解決、戻り値、可変長引数、this/super、暗黙super、値渡し は、Javaで頻出

  1. メソッドなら: 宣言が正しいか、呼び出しに一致する候補があるか、戻り値が適切かを見る。
  2. コンストラクタなら: 戻り値型を書いていないか、this()/super() が先頭か、親コンストラクタを呼べるかを見る。
  3. 引数渡しなら: 変数そのものが変わるのか、参照先オブジェクトの中身が変わるのかを分ける。
雑に覚えると危険: 「参照型は参照渡し」という言い方は学習対策としては不正確です。Javaは常に値渡しです。参照型では参照値がコピーされます。

メソッド宣言の構成

メソッド宣言は、修飾子、戻り値型、メソッド名、引数リスト、throws句、メソッド本体で構成されます。

public static int add(int a, int b) throws Exception {
    return a + b;
}
部分例見るポイント
修飾子public staticアクセス、static/instanceの違い
戻り値型intvoid 以外は必ず値を返す必要がある
メソッド名add識別子ルールに従う
引数int a, int b型と順番がシグネチャに関係する
throwsthrows Exceptionチェック例外の処理に関係する
本体{ return a + b; }戻り値、到達不能文、例外発生を確認する

メソッドシグネチャとオーバーロード

Javaでオーバーロード判定に使うシグネチャは、基本的に メソッド名 + 引数リスト です。戻り値型だけを変えてもオーバーロードにはなりません。

class Sample {
    void test(int a) {}
    void test(String s) {}     // OK: 引数型が違う
    void test(int a, int b) {} // OK: 引数の数が違う
    // int test(int a) { return a; } // NG: 戻り値型だけ違う
}
違いオーバーロード可否理由
引数の型が違う可test(int) と test(String) は別
引数の数が違う可test(int) と test(int,int) は別
引数の順番が違う可test(int,String) と test(String,int) は別
戻り値型だけ違う不可呼び出し側だけでは区別できない
仮引数名だけ違う不可test(int x) と test(int y) は同じシグネチャ

オーバーロード解決の優先順位

複数の候補がある場合、呼び出しに最も適合するメソッドが選ばれます。Javaでは、細かい仕様を完全暗記するより、次の優先順で選択肢を切ると安定します。

優先変換例
1完全一致test(int) に 10
2拡大プリミティブ変換int → long
3ボクシングint → Integer
4可変長引数int...
 
void test(long x) { System.out.print("long"); }
void test(Integer x) { System.out.print("Integer"); }
void test(int... x) { System.out.print("varargs"); }

test(10); // long

10 はintリテラルです。long への拡大変換は、Integer へのボクシングや可変長引数より優先されます。

nullを渡すオーバーロード

null は参照型に代入できます。複数の参照型候補がある場合、より具体的な型が選ばれます。どちらが具体的か決められない場合はコンパイルエラーです。

void test(Object o) { System.out.print("Object"); }
void test(String s) { System.out.print("String"); }

test(null); // String

String は Object のサブクラスなので、より具体的な String が選ばれます。

void sample(String s) {}
void sample(StringBuilder sb) {}

// sample(null); // NG: String と StringBuilder のどちらとも決められない
判断: null 問題では、候補が親子関係か、横並びの別クラスかを見ます。

varargsのルール

可変長引数は、0個以上の引数を受け取れる仕組みです。メソッド内では配列として扱われます。

void print(String label, int... values) {
    System.out.println(values.length);
}

print("A");          // 0
print("B", 1, 2, 3); // 3
ルール例
可変長引数は最後に1つだけvoid m(String s, int... x)
最後以外には書けないvoid m(int... x, String s) はNG
2つ以上書けないvoid m(int... a, String... b) はNG
配列として渡せるm(new int[]{1,2})

戻り値の判定

void 以外のメソッドは、正常終了するすべての経路で値を返す必要があります。

int ok(boolean b) {
    if (b) {
        return 1;
    }
    return 2;
}

int ng(boolean b) {
    if (b) {
        return 1;
    }
    // NG: bがfalseの経路でreturnがない
}

return の後に通常文を書くと到達不能文としてコンパイルエラーになることがあります。

void test() {
    return;
    // System.out.println("A"); // NG: 到達不能
}

コンストラクタの基本

コンストラクタはインスタンス生成時に呼ばれる特別な処理です。クラス名と同じ名前で、戻り値型を書きません。戻り値型を書いたら、それはコンストラクタではなく普通のメソッドです。

class Sample {
    Sample() { }       // コンストラクタ
    void Sample() { }  // メソッド。コンストラクタではない
}
項目メソッドコンストラクタ
名前任意の識別子クラス名と同じ
戻り値型必要。void も可書かない
呼び出しobj.method()new Sample()
継承条件次第で継承される継承されない

デフォルトコンストラクタ

コンストラクタを1つも定義しない場合、コンパイラが引数なしコンストラクタを自動で追加します。しかし、1つでも自分でコンストラクタを定義すると、デフォルトコンストラクタは追加されません。

class A {
    // コンストラクタを何も書かないので A() が暗黙にある
}
new A(); // OK

class B {
    B(String s) {}
}
// new B(); // NG: B() は自動追加されない
頻出: 「引数ありコンストラクタを書いたから、引数なしもあるはず」は誤りです。必要なら明示的に定義します。

this() と super()

this() は同じクラスの別コンストラクタを呼び、super() は親クラスのコンストラクタを呼びます。どちらもコンストラクタの先頭行にしか書けません。同じコンストラクタ内で両方を書くこともできません。

class A {
    A(int n) {}
}
class B extends A {
    B() {
        this(10); // OK: 先頭行
    }
    B(int n) {
        super(n); // OK: 先頭行
    }
}
class C extends A {
    C() {
        System.out.println("C");
        // super(10); // NG: 先頭行ではない
    }
}

暗黙のsuper()

コンストラクタの先頭に this() も super() も書かない場合、コンパイラが super() を挿入します。そのため、親クラスに引数なしコンストラクタがないとコンパイルエラーになります。

class Parent {
    Parent(String name) {}
}
class Child extends Parent {
    Child() {
        // ここに super(); が暗黙挿入される
        // しかし Parent() が存在しないのでNG
    }
}

修正するなら、親のコンストラクタに合うように明示的に super("x") を呼びます。

class Child extends Parent {
    Child() {
        super("x");
    }
}

コンストラクタ連鎖の実行順

サブクラスのインスタンス生成では、最終的に親クラス側から初期化されます。this() で同じクラスの別コンストラクタへ進んでも、どこかで super() に到達します。

class A {
    A() { System.out.print("A"); }
}
class B extends A {
    B() {
        this(1);
        System.out.print("B");
    }
    B(int n) {
        System.out.print("C");
    }
}
new B();
順番処理出力
1B() が this(1) を呼ぶ-
2B(int) の先頭に暗黙 super()-
3A()A
4B(int) の本体C
5B() の残りB

出力は ACB です。

引数渡し: Javaは常に値渡し

プリミティブ型では値そのもののコピー、参照型では参照値のコピーが渡されます。

static void change(int x) {
    x = 20;
}
int a = 10;
change(a);
System.out.println(a); // 10

呼び出し元の a は変わりません。コピーされた x だけが変わります。

static void change(StringBuilder sb) {
    sb.append("B");
    sb = new StringBuilder("C");
}
StringBuilder s = new StringBuilder("A");
change(s);
System.out.println(s); // AB
操作呼び出し元に影響理由
sb.append("B")ある同じオブジェクトの中身を変更している
sb = new StringBuilder("C")ない仮引数の参照値コピーを差し替えただけ

staticメソッドとインスタンスメソッド

static メソッドはインスタンスなしで呼べますが、インスタンスフィールドへ直接アクセスできません。

class Sample {
    private String name = "A";
    static void print() {
        // System.out.println(name); // NG: staticからinstanceを直接参照できない
    }
}

インスタンスを作って、その参照を経由すればアクセスできます。

class Sample {
    private String name = "A";
    static void print() {
        Sample s = new Sample();
        System.out.println(s.name); // OK
    }
}

mainメソッドの形

実行エントリポイントとして使えるmainメソッドは、基本形を崩さないことが重要です。

public static void main(String[] args) {}
public static void main(String... args) {}
要素必要条件
アクセスpublic
static必要
戻り値void
名前main
引数String[] または String...

コンストラクタとアクセス修飾子

コンストラクタにもアクセス修飾子を付けられます。サブクラスから親コンストラクタを呼ぶ場合、そのコンストラクタにアクセスできる必要があります。

class A {
    private A() {}
}
class B extends A {
    // B() { super(); } が必要だが、A() はprivateなのでNG
}

protected の親コンストラクタは、サブクラスから呼べます。

class A {
    protected A() {}
}
class B extends A {
    public B() {
        super(); // OK
    }
}

典型ひっかけ

戻り値型付きコンストラクタ

void Sample() はコンストラクタではなくメソッドです。 new Sample() 時には呼ばれません

戻り値だけ違うオーバーロード

同じ引数リストで戻り値だけ違う同名メソッドは定義できません

暗黙super()

親に無引数コンストラクタがないのに、子で明示的な super(...) を書かないとコンパイルエラー

参照型を参照渡しだと思う

仮引数を別インスタンスに差し替えても、呼び出し元の変数は差し替わりません

varargsを通常引数より優先する

可変長引数は原則として最後の候補です。完全一致や拡大変換があればそちらを優先します

staticからinstanceへ直接アクセス

main やstaticメソッドからインスタンスフィールドを直接読めません

ミニ演習

Q1. class A { void A(){} } にはコンストラクタがある？

void A() はメソッドです。明示的なコンストラクタはありません。そのため、引数なしデフォルトコンストラクタが追加されます

Q2. void test(int) と int test(int) は同時に定義できる？

できません。戻り値型だけの違いではオーバーロードになりません

Q3. 親に A(String) しかないとき、子の B(){} はコンパイルできる？

できません。 B() の先頭に暗黙の super() が入りますが、親に A() がないため

Q4. test(10) に対して test(long) と test(Integer) がある場合は？

通常は test(long) が選ばれます。 int から long への拡大変換が、ボクシングより優先されるため

関連: 出力追跡 / クラス・static・record / 継承・interface / ポリモーフィズム

この単元で固定する判断順

  1. 対象が何かを見る。 String、StringBuilder、配列、ArrayList、List、ラッパークラスでは同じメソッド名でも意味が変わります。
  2. 戻り値を確認する。 返された値を使っているか、戻り値を捨てているかで結果が変わります。
  3. 元のオブジェクトが変わるかを見る。 String は不変、StringBuilder と ArrayList は基本的に可変です。
  4. インデックス範囲を見る。 0 始まり、終端は「含む」のか「含まない」のかを必ず分けます。
  5. null・空・範囲外の例外を見る。 コンパイルできても、実行時に NullPointerException や範囲外例外で止まることがあります。
雑に暗記すると危険です。
問題では「このメソッドを知っているか」より、「戻り値を受け取らないと変わらない」「元が変わっているから後続行に影響する」という形で落としてきます。

String：不変オブジェクトとして見る

String のメソッドは、多くの場合「新しい結果」を返します。元の文字列は変わりません。ここを間違えると、出力問題でほぼ確実に落ちます。

String s = "abc";
s.concat("de");
System.out.println(s);     // abc

s = s.concat("de");
System.out.println(s);     // abcde
メソッド戻り値元のString問題での注意
length()int変化なし配列は length フィールド。() を付けない。
charAt(int index)char変化なし0 以上 length() - 1 以下。範囲外は例外。
substring(int begin)String変化なしbegin から末尾まで。戻り値を使わないと意味がない。
substring(int begin, int end)String変化なしbegin 以上 end 未満。end の文字は含まない。
indexOf(String s)int変化なし見つからない場合は -1。例外ではない。
replace(char old, char new)String変化なしすべて置換した新しい文字列を返す。
startsWith(String)boolean変化なし大小文字は区別される。
endsWith(String)boolean変化なし末尾一致。部分一致とは違う。
equals(Object)boolean変化なし左辺が null なら呼び出し時に例外。
equalsIgnoreCase(String)boolean変化なし左辺 null は例外。右辺 null は false。
toLowerCase() / toUpperCase()String変化なし戻り値を代入しないと元は変わらない。
trim()String変化なし前後の空白を取り除いた結果を返す。中間の空白は消えない。
concat(String)String変化なし+ と似るが、null 引数は例外になり得る。

Stringの境界値：substringとcharAt

境界値は、問題作成者が最も使いやすいひっかけです。substring の第2引数は「取り出す最後の位置」ではなく、「取り出しを終える位置」です。

String s = "abcd";
System.out.println(s.substring(1, 3)); // bc
System.out.println(s.substring(4));    // 空文字
System.out.println(s.charAt(3));       // d
System.out.println(s.charAt(4));       // 実行時例外
式結果理由
s.substring(0, 4)"abcd"end は length() まで許される。
s.substring(4)空文字末尾から末尾までを取り出す扱い。
s.substring(2, 2)空文字開始位置と終了位置が同じ。
s.substring(3, 2)実行時例外開始位置が終了位置より後。
s.charAt(4)実行時例外charAt は length() と同じ位置を参照できない。

StringBuilder：可変オブジェクトとして見る

StringBuilder は中身を書き換えます。戻り値も多くの場合、自分自身への参照です。つまり、同じインスタンスを複数変数で参照していると、片方の操作がもう片方の見え方にも影響します。

StringBuilder a = new StringBuilder("A");
StringBuilder b = a;
a.append("B");
System.out.println(b); // AB
メソッド戻り値元のStringBuilder問題での注意
append(x)StringBuilder変わる末尾追加。戻り値は基本的に同じインスタンス。
insert(index, x)StringBuilder変わる指定位置の前に挿入。範囲外に注意。
delete(begin, end)StringBuilder変わるbegin 以上 end 未満を削除。
deleteCharAt(index)StringBuilder変わる1文字削除。範囲外は例外。
replace(begin, end, str)StringBuilder変わる範囲を別文字列で置換。
reverse()StringBuilder変わる戻り値を受けなくても反転済み。
length()int変化なし現在の文字数。
charAt(index)char変化なし範囲外は例外。
toString()String変化なしString と比較したいなら文字列化してから。
equals(Object)boolean変化なし内容比較ではない。基本は参照比較。
StringBuilder.equals() は頻出の罠。
new StringBuilder("a").equals(new StringBuilder("a")) は、内容が同じでも通常 false です。内容比較したいなら toString().equals(...) を使う必要があります。

配列：lengthはフィールド、固定長、初期値あり

配列はオブジェクトですが、length はメソッドではなくフィールドです。length() と書いたらコンパイルエラーです。

対象要素数取得注意
配列array.lengthフィールド。() なし。
Stringstr.length()メソッド。() あり。
ArrayListlist.size()メソッド。length ではない。
 
int[] nums = new int[3];
System.out.println(nums.length); // 3
System.out.println(nums[0]);     // 0
nums[3] = 10;                    // 実行時例外

ArrayList / List：戻り値と副作用を分ける

ArrayList は可変です。add、set、remove の戻り値がそれぞれ違うため、選択肢で狙われます。

メソッド戻り値副作用注意
add(E e)boolean末尾に追加通常 true。戻り値は追加要素ではない。
add(int index, E e)void指定位置に挿入指定位置以降が後ろへずれる。
set(int index, E e)置換前の要素指定位置を置換追加ではない。サイズは変わらない。
get(int index)指定位置の要素なし範囲外は例外。
remove(int index)削除した要素指定位置を削除後続要素のインデックスが詰まる。
remove(Object o)boolean一致要素を削除最初に一致した要素のみ削除。
contains(Object o)booleanなしequals で判定。
size()intなし配列の length と混同しない。
clear()void全要素削除空になるがリスト変数自体は null ではない。
isEmpty()booleanなしサイズ0なら true。

removeのオーバーロード：intかObjectか

remove は、引数が int ならインデックス削除、Object なら要素削除です。整数リストでは特に危険です。

var list = new java.util.ArrayList<Integer>();
list.add(10);
list.add(20);
list.add(30);

list.remove(1);                  // インデックス1を削除。20が消える
list.remove(Integer.valueOf(30)); // 要素30を削除
ここは暗記ではなくオーバーロード解決です。
remove(1) の 1 は int リテラルなので、remove(int index) が優先されます。要素の 1 を消したい場合は Integer.valueOf(1) のように Object として渡す必要があります。

List.of と Arrays.asList

どちらも短くリストを作れますが、性質が違います。変更できるか、元配列と連動するかを見ます。

作り方サイズ変更要素変更注意
new ArrayList<>()できるできる通常の可変リスト。
List.of("A", "B")できないできない変更系操作は実行時例外になり得る。
Arrays.asList(array)できないできる固定サイズ。元配列と連動する点に注意。
 
String[] array = {"A", "B"};
var list = java.util.Arrays.asList(array);
list.set(0, "X");
System.out.println(array[0]); // X
// list.add("C");            // 実行時例外

ラッパークラス・変換メソッド

ラッパークラスでは、文字列を数値に変換するメソッド、数値を文字列にするメソッド、オートボクシングとの関係が問われます。

メソッド/式戻り値注意
Integer.parseInt("10")intプリミティブを返す。変換不能なら NumberFormatException。
Integer.valueOf("10")Integerラッパーを返す。
String.valueOf(x)Stringnull にも比較的強い。
Integer i = 10;Integerオートボクシング。
int n = i;intアンボクシング。i が null なら実行時例外。

equals / compareTo / contains の見方

比較系メソッドは「何を基準に比較しているか」を見る必要があります。

対象メソッド判定基準注意
Stringequals文字列の内容== は参照比較。
StringcompareTo辞書順等しければ0。大小は正負で判定。
StringBuilderequals基本は参照内容比較ではない。
ArrayListcontainsequals要素側の equals 実装に依存。
配列equals基本は参照内容比較ではない。問題では参照比較として扱う。

問題での最終チェック

  • String の戻り値を捨てていないか。
  • StringBuilder の変更が別変数にも見えていないか。
  • substring の第2引数を「含む」と誤解していないか。
  • 配列の length、文字列の length()、Listの size() を混同していないか。
  • remove(int) と remove(Object) を区別しているか。
  • List.of や Arrays.asList を通常の ArrayList と同じに扱っていないか。
  • 左辺が null のメソッド呼び出しになっていないか。
