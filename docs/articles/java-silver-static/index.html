staticで間違えるポイント

static文脈、インスタンスメンバー、フィールド隠蔽を整理します。

最初に固定する読み方

結論を急がない。 まずコンパイルできるか、次に実行時に止まるか、最後に出力を追います。出力だけを先に追うと、型・アクセス・例外で簡単に崩れます。
  1. 宣言を見る。 型、修飾子、継承関係、メソッドシグネチャを先に確定します。
  2. 代入できるかを見る。 右辺の型・値・参照先が左辺に入るかを判断します。
  3. 呼び出せるかを見る。 変数の宣言型から、その場で見えるメンバだけを呼べます。
  4. 実体で動くものを分ける。 オーバーライドされたインスタンスメソッドは実体、フィールドやstaticは別規則です。
  5. 副作用を表にする。 代入、インクリメント、可変オブジェクト、例外で状態が変わる箇所を書き出します。

クラス・オブジェクト・インスタンスの入口

クラスは設計図、オブジェクトは実体、インスタンスはクラスから作られた具体的な実体です。学習上は「オブジェクト」と「インスタンス」は近い意味で使われますが、コードを読むときは クラス側にあるもの と インスタンス側にあるもの を分けることが重要です。

用語意味コード上の例
クラスフィールドやメソッドの定義を持つ設計。class Item { ... }
オブジェクト / インスタンスnew によって作られる実体。new Item()
フィールドオブジェクトやクラスが保持する状態。int price;
メソッド状態を使って処理を行う動作。void print()
 
class Item {
    String name;       // フィールド: 状態
    int price;         // フィールド: 状態

    void print() {     // メソッド: 動作
        System.out.println(name + ":" + price);
    }
}

static はクラス側に属する

static は「インスタンスごと」ではなく「クラス側」に属します。インスタンスを何個作っても、static フィールドはクラスに1つです。

class Counter {
    static int total; // クラス側に1つ
    int value;        // インスタンスごとに1つ

    Counter() {
        total++;
        value++;
    }
}
対象所属先個数注意点
インスタンスフィールド各インスタンスインスタンスごと参照先が違えば値も別。
インスタンスメソッド各インスタンスの状態を使う処理呼び出しには原則インスタンスが必要this を使える。
staticフィールドクラス側クラスに1つ全インスタンスで共有される。
staticメソッドクラス側クラスから呼べるインスタンスメンバを直接使えない。
recordの見方: record はデータ保持用クラスを短く書く仕組みです。通常クラスと同じく、型・コンストラクタ・メソッド・staticの規則からは逃げられません。

この単元の目的

修飾子の問題は、知識問題に見えて実際は「どこに書けるか」「誰から見えるか」「継承・overrideと矛盾しないか」を切る問題です。暗記だけだと、protected、private、abstract、final、sealed の組み合わせで崩れます。

この単元では、アクセス制御と修飾子の組み合わせを、コンパイル可否の判断に直結する形で整理します。

最初に使う判定順

  1. 宣言場所を見る: トップレベルクラス、メンバクラス、フィールド、メソッド、コンストラクタ、インタフェースのどれか。
  2. その修飾子をそこに書けるかを見る: 例: トップレベルに private や static は不可。
  3. アクセス範囲を見る: 同じクラス、同じパッケージ、別パッケージのサブクラス、その他で切る。
  4. 継承・overrideと矛盾しないかを見る: private、final、static、abstract の関係を確認する。
  5. sealed系の継承方針を見る: sealed の子は final / sealed / non-sealed のいずれかが必要。
修飾子問題は、意味を先に考えるより「その場所に書けるか」を先に見る方が速いです。書けない修飾子なら、その時点でコンパイルエラーです。

アクセス修飾子の基本表

修飾子同じクラス同じパッケージ別パッケージのサブクラス別パッケージの非サブクラス
private可不可不可不可
指定なし可可不可不可
protected可可条件付きで可不可
public可可可可

指定なしは「デフォルトアクセス」または「パッケージアクセス」です。default という修飾子を書くわけではありません。default キーワードはインタフェースのデフォルトメソッドで出てきます。ここを混同しないでください。

トップレベルクラスに使える修飾子

トップレベルクラスとは、他のクラスの中に入っていないクラスです。トップレベルクラスに使えるアクセス修飾子は public または指定なしです。

public class A { }     // OK
class B { }            // OK: デフォルトアクセス

// private class C { } // NG
// protected class D { } // NG
// static class E { }  // NG: トップレベルでは不可

トップレベルクラスでは、abstract、final、sealed、non-sealed などの継承方針に関係する修飾子も出題されます。ただし、private、protected、static はトップレベルクラスには使えません。

メンバに使うアクセス修飾子

フィールド、メソッド、コンストラクタには、public、protected、指定なし、private を使えます。

class Sample {
    public int a;
    protected int b;
    int c;          // デフォルトアクセス
    private int d;

    public Sample() {}
    private void helper() {}
}

ただし、使えることと、外部からアクセスできることは別です。宣言として正しくても、呼び出し側の場所によってはアクセスエラーになります。

privateメンバは継承先から直接見えない

private メンバは同じクラス内からだけ使えます。サブクラスであっても直接アクセスできません。

class Parent {
    private int value = 10;
    private void test() {}
}

class Child extends Parent {
    void sample() {
        // System.out.println(value); // NG
        // test();                    // NG
    }
}

ただし、privateメンバが「インスタンス内に存在しない」という意味ではありません。アクセスできないだけです。publicなメソッド経由で値を使うことはできます。

class Parent {
    private int value = 10;
    public int getValue() { return value; }
}

class Child extends Parent {
    void sample() {
        System.out.println(getValue()); // OK
    }
}

privateメソッドはoverrideされない

privateメソッドはサブクラスから見えません。そのため、サブクラスで同じシグネチャのメソッドを書いてもoverrideではありません。別メソッドです。

class A {
    private void test() {
        System.out.println("A");
    }
}

class B extends A {
    void test() {
        System.out.println("B");
    }
}

@Override を付けると、overrideではないことが検出され、コンパイルエラーになります。

class B extends A {
    // @Override
    void test() { }
}

protectedの正しい理解

protected は「同じパッケージなら継承関係なしでアクセス可能」です。さらに、別パッケージでもサブクラスから条件付きでアクセスできます。

package p1;
public class A {
    protected int value;
}

package p1;
public class X {
    void test() {
        A a = new A();
        a.value = 10; // OK: 同じパッケージ
    }
}

別パッケージのサブクラスでは、単に「継承しているから何でもOK」ではありません。サブクラス自身またはサブクラス型の参照を通じたアクセスとして考える必要があります。

package p2;
public class B extends p1.A {
    void test(B b, p1.A a) {
        this.value = 1; // OK
        b.value = 2;    // OK
        // a.value = 3; // NGになりうる典型形
    }
}
protectedを「別パッケージのサブクラスなら親型参照からでも自由に触れる」と覚えるのは危険です。Javaの問題では、この雑な理解を狙われます。

コンストラクタのアクセス修飾子

コンストラクタにもアクセス修飾子を付けられます。コンストラクタのアクセス範囲は、インスタンス生成できる範囲を決めます。

class Sample {
    private Sample() {}

    static Sample create() {
        return new Sample(); // 同じクラス内なのでOK
    }
}

// new Sample(); // クラス外ならNG

また、サブクラスのコンストラクタは、スーパークラスのコンストラクタを直接または間接的に呼び出す必要があります。スーパークラスの利用可能なコンストラクタがなければ、サブクラス側でコンパイルエラーになります。

class A {
    private A() {}
}

class B extends A {
    // B() { super(); } が暗黙に入るが、A()がprivateなのでNG
}

一方、protected コンストラクタなら、サブクラスから super() で呼べます。

class A {
    protected A() {}
}

class B extends A {
    public B() {
        super(); // OK
    }
}

abstract classとabstract method

abstract クラスは、インスタンス化できないクラスです。抽象メソッドは本体を持たないメソッドです。

abstract class A {
    abstract void test(); // 本体なし

    void common() {       // 通常メソッドも持てる
        System.out.println("common");
    }
}

class B extends A {
    @Override
    void test() {
        System.out.println("B");
    }
}

抽象メソッドを持つクラスは、必ずabstractクラスでなければなりません。

// class A {
//     abstract void test(); // NG: クラスもabstractにする必要あり
// }

逆に、abstractクラスに抽象メソッドが必ず必要というわけではありません。抽象メソッドなしのabstractクラスも作れます。

abstractと矛盾する修飾子

抽象メソッドは、サブクラスで実装されることが前提です。そのため、実装やoverrideを妨げる修飾子とは相性が悪いです。

組み合わせ判定理由
abstract final class不可継承前提と継承禁止が矛盾
private abstract メソッド不可サブクラスから見えず、実装できない
final abstract メソッド不可override必須とoverride禁止が矛盾
static abstract メソッド不可staticメソッドはoverride対象ではない
abstract class A {
    // private abstract void a(); // NG
    // final abstract void b();   // NG
    // static abstract void c();  // NG
}

finalの3つの意味

final は付ける対象によって意味が変わります。

対象意味例
変数再代入不可final int x = 10;
メソッドoverride不可final void test()
クラス継承不可final class A
final class A {}
// class B extends A {} // NG

class C {
    final void test() {}
}
class D extends C {
    // void test() {} // NG
}

final変数の注意点は、参照型の場合です。再代入は不可ですが、可変オブジェクトの中身変更は別問題です。

staticはoverrideではなく隠蔽

staticメソッドはインスタンスではなくクラスに属します。そのため、サブクラスで同じシグネチャのstaticメソッドを定義してもoverrideではありません。メソッドの隠蔽です。

class A {
    static void test() {
        System.out.println("A");
    }
}

class B extends A {
    static void test() {
        System.out.println("B");
    }
}

A a = new B();
a.test(); // A: 宣言型A側のstaticメソッド

staticメソッド呼び出しは、実体型による動的ディスパッチではなく、宣言型・クラス名に寄ると考えると整理できます。

staticメソッドに @Override を付けるとコンパイルエラーです。overrideではありません。

override時のアクセス修飾子

オーバーライドでは、アクセス範囲を狭くできません。同じか、より広いアクセスにする必要があります。

親メソッド子メソッドで可能子メソッドで不可
publicpublicprotected、指定なし、private
protectedprotected、public指定なし、private
指定なし指定なし、protected、publicprivate
class A {
    protected void test() {}
}

class B extends A {
    public void test() {} // OK: 広くしている
}

class C extends A {
    // void test() {} // NG: protectedより狭い
}

privateメソッドはoverride対象ではないため、この表とは別扱いです。

インタフェースの暗黙修飾子

インタフェースでは、暗黙に付く修飾子が多いです。ここは問題でよく省略されます。

宣言暗黙の扱い注意
フィールドpublic static final定数。必ず初期化が必要
抽象メソッドpublic abstract実装クラスではpublicで実装が必要
default メソッドpublicなインスタンスメソッド本体が必要
static メソッドインタフェース名で呼ぶ実装クラスに継承された通常メソッド扱いではない
private メソッドインタフェース内の補助default/staticメソッドから使う
interface A {
    int NUM = 10;        // public static final int NUM = 10;
    void test();         // public abstract void test();

    default void call() {
        helper();
    }

    private void helper() {}
}

class B implements A {
    public void test() {} // publicが必要
}
インタフェースの抽象メソッドはpublicです。実装クラス側で指定なしにすると、アクセスを狭くすることになりコンパイルエラーです。

sealed / non-sealed / permits

sealed は、継承または実装できる相手を制限する仕組みです。permits で許可されたクラスだけが直接のサブクラスになれます。

sealed class A permits B, C {}
final class B extends A {}
non-sealed class C extends A {}

// class D extends A {} // NG: permitsに含まれていない

sealedクラスを継承した直接のサブクラスは、次のどれかで今後の継承方針を示す必要があります。

修飾子意味
finalここで継承を止める
sealedさらに継承先を限定する
non-sealed以降の継承制限を解除する
sealed class Parent permits Child {}

// class Child extends Parent {} // NG: final/sealed/non-sealedのどれかが必要

final class Child extends Parent {} // OK

non-sealed は、sealedな親を持つクラスで意味を持ちます。何でもないクラスに突然 non-sealed を付けるのは不可です。

修飾子の組み合わせ早見表

コード断片判定理由
private class A不可トップレベルクラスにprivate不可
static class Aトップレベルなら不可staticなトップレベルクラスはない
abstract final class A不可継承前提と継承禁止が矛盾
private abstract void test();不可サブクラスで実装できない
final void test()可override不可の通常メソッド
abstract void test() {}不可abstractメソッドは本体を持てない
interface A { int x = 1; }可xはpublic static final
class B implements A { void test(){} }不可になりうるAのtestがpublicなら、実装もpublicが必要

典型ひっかけ

  • protected を「同じパッケージのサブクラスだけ」と覚えてしまう。
  • デフォルトアクセスを default と書けると思う。
  • トップレベルクラスに private、protected、static を付ける。
  • privateメソッドをoverrideできると思う。
  • overrideでアクセス範囲を狭くしてしまう。
  • abstract と final を同時に使えると思う。
  • staticメソッドをoverrideだと思う。
  • インタフェースのメソッド実装で public を忘れる。
  • sealedクラスの子に final / sealed / non-sealed を付け忘れる。

ミニ演習

Q1. 次はコンパイルできる？ 
private class A {}

答え: トップレベルクラスならコンパイルエラー。トップレベルクラスに private は使えません。

Q2. 次はoverride？ 
class A {
    private void test() {}
}
class B extends A {
    void test() {}
}

答え: overrideではありません。privateメソッドはサブクラスから見えません。

Q3. 次はなぜエラー？ 
interface A {
    void test();
}
class B implements A {
    void test() {}
}

答え: インタフェースの test は暗黙にpublicです。実装側で指定なしにするとアクセスを狭くするため、public void test() が必要です。

Q4. 次のsealed継承は正しい？ 
sealed class A permits B {}
class B extends A {}

答え: コンパイルエラー。sealedクラスの直接の子は final、sealed、non-sealed のいずれかで修飾する必要があります。

この単元で最初に固定する考え方

継承・interfaceの問題は、用語を知っているだけでは足りません。毎回、次の順で切ります。

  1. クラス関係を見る。extends か implements か、単一継承か複数interfaceか。
  2. 宣言型を見る。変数の型から、その場で呼べるメソッドが決まります。
  3. 実体を見る。overrideされたインスタンスメソッドは、実体のクラスで動きます。
  4. アクセス修飾子を見る。継承していることと、直接アクセスできることは別です。
  5. コンストラクタ連鎖を見る。コンストラクタは継承されず、生成時には必ず親のコンストラクタが先に動きます。

雑に「子クラスだから親の全部が使える」と考えると落ちます。Javaでは、private、default、protected、コンストラクタ、フィールド解決で崩されます。

スーパークラスとサブクラス

継承を読むときは、まず「どちらが元で、どちらが拡張先か」を固定します。class Child extends Parent なら、Parent がスーパークラス、Child がサブクラスです。

用語別名意味コード上の位置
スーパークラス親クラス / 基底クラス継承元。共通のフィールドやメソッドを持つ側extends の右側
サブクラス子クラス / 派生クラス継承先。スーパークラスを土台に差分を追加する側extends の左側
 
class Parent {          // スーパークラス
    String name = "parent";
    void hello() {
        System.out.println("hello " + name);
    }
}

class Child extends Parent { // サブクラス
    void childOnly() {
        System.out.println("child");
    }
}

Child のインスタンスは、Parent として扱える部分と、Child で追加した部分をあわせて持ちます。だから Parent p = new Child(); のように、サブクラスのインスタンスをスーパークラス型の変数に代入できます。

Parent p = new Child();
p.hello();
// p.childOnly(); // NG: 変数の宣言型ParentにはchildOnlyがない

ここで重要なのは、インスタンスが持っている機能と、変数の型から呼べる機能は別だという点です。実体が Child でも、変数の宣言型が Parent なら、Parent に見えているメンバしか呼べません。

継承は「is-a関係」として読む

サブクラスはスーパークラスの一種として扱えます。これを is-a関係 と考えます。Child extends Parent なら「Child は Parent の一種」と読める関係です。

class Animal {
    void eat() {}
}

class Dog extends Animal {
    void bark() {}
}

Animal a = new Dog(); // OK: DogはAnimalの一種
// Dog d = new Animal(); // NG: Animalが必ずDogとは限らない

この非対称性が重要です。サブクラスからスーパークラスへの代入は自然にできますが、スーパークラスからサブクラスへの代入にはキャストが必要です。そして、実体がそのサブクラスでなければ実行時に ClassCastException になります。

問題での読み方

A a = new B(); が出たら、まず B extends A または B implements A の関係があるかを確認します。関係がなければコンパイルエラーです。関係がある場合でも、a から呼べるのは宣言型 A にあるメンバだけです。

継承されるもの・されないもの・直接アクセスできないもの

「サブクラスはスーパークラスを継承する」という表現を雑に覚えると危険です。問題では、何が使えて何が使えないかを分けて問われます。

対象サブクラスでどう扱うか注意点
public / protected / 条件を満たすdefaultメンバ利用できるアクセス修飾子とパッケージ条件を見る
privateフィールド・privateメソッド直接アクセス不可親の内部には存在するが、子から直接触れない
コンストラクタ継承されないsuper(...) で呼び出すだけ
同名フィールドoverrideではないフィールドは宣言型で解決される
インスタンスメソッドoverride可能呼び出し可否は宣言型、実行される本体は実体で決まる
staticメソッドoverrideではなく隠蔽インスタンスメソッドとは解決規則が違う

特に private とコンストラクタは「継承しているから使える」という雑な理解を壊すためによく出ます。ここを曖昧にしたまま問題を解くと、コンパイルエラー判定を外します。

this と super の役割

this は現在のインスタンス、super はスーパークラス側のメンバやコンストラクタを指すために使います。

class Parent {
    String name = "parent";
    Parent(String name) {
        this.name = name;
    }
    void print() {
        System.out.println(name);
    }
}

class Child extends Parent {
    String name = "child";

    Child() {
        super("super-name"); // 親コンストラクタ呼び出し。必ず先頭
    }

    void printBoth() {
        System.out.println(this.name);  // Child側のname
        System.out.println(super.name); // Parent側のname
        super.print();                  // Parent側のメソッド呼び出し
    }
}

super(...) はコンストラクタ呼び出しで、コンストラクタの先頭にしか書けません。super.name や super.print() はスーパークラス側のメンバ参照です。同じ super でも使い方を混ぜないでください。

extends：クラス継承の基本

extends は、既存クラスを拡張してサブクラスを作る構文です。左側のクラスがサブクラス、右側のクラスがスーパークラスです。サブクラスのインスタンスは、スーパークラス部分とサブクラス部分を持ちます。

class Parent {
    String name = "parent";
    void hello() {
        System.out.println("hello " + name);
    }
}

class Child extends Parent {
    void childOnly() {
        System.out.println(name); // Parentから継承されたフィールドを使える
    }
}

ここで重要なのは、継承されたメンバがあるから使えるという点です。もし Parent に name がなく、Child にも name がなければ、name は未定義なのでコンパイルエラーです。

コンストラクタは継承されない

継承で引き継がれない代表がコンストラクタです。サブクラスのインスタンス生成時には、サブクラスのコンストラクタが呼ばれ、その先頭でスーパークラスのコンストラクタが呼ばれます。

class A {
    A(String value) {
        System.out.println("A:" + value);
    }
}

class B extends A {
    B() {
        super("b");
        System.out.println("B");
    }
}

new B() の実行順は A:b → B です。B が A(String) を継承して new B("x") できるわけではありません。

ひっかけ：親に引数なしコンストラクタがない場合

子クラスのコンストラクタで super(...) を書かないと、コンパイラは暗黙に super() を入れます。親に A() がなければコンパイルエラーです。

privateメンバは「存在する」が「直接使えない」

privateフィールドやprivateメソッドは、サブクラスから直接アクセスできません。ただし、親クラスの内部からは当然使えます。

class A {
    private String value = "A";
    public String getValue() {
        return value;
    }
}

class B extends A {
    void print() {
        // System.out.println(value); // NG: privateなので直接アクセス不可
        System.out.println(getValue()); // OK: publicメソッド経由
    }
}

問題では「継承しているからprivateも使える」と思わせる選択肢が出ます。これは切ってください。privateは同じクラス内だけです。

implements：interfaceの実装

implements は、interfaceで定義された使い方をクラスに実装させる構文です。クラスは1つのクラスしか継承できませんが、複数のinterfaceを実装できます。

interface Printable {
    void print(); // public abstract扱い
}

interface Savable {
    void save();
}

class Report implements Printable, Savable {
    public void print() {
        System.out.println("print");
    }
    public void save() {
        System.out.println("save");
    }
}

interfaceの抽象メソッドは暗黙的に public abstract です。そのため、実装クラス側のメソッドも public にしなければいけません。アクセスを狭くするとコンパイルエラーです。

抽象クラスとinterfaceの違い

観点抽象クラスinterface
継承方法extendsimplements
複数指定不可。クラス継承は1つだけ可。複数実装できる
インスタンス化不可不可

コンストラクタ持てる持てない

抽象クラスは「未完成のクラス」です。interfaceは「扱い方・契約」です。この区別を曖昧にすると、コンストラクタやフィールドの問題で間違えます。

abstractメソッドを実装しない場合

抽象メソッドを継承した具象クラスは、そのメソッドを実装しなければコンパイルエラーです。実装しないなら、そのクラス自身も abstract にする必要があります。

abstract class A {
    abstract void test();
}

abstract class B extends A {
    // testを実装しないならBもabstract
}

class C extends B {
    void test() {
        System.out.println("C");
    }
}

overrideの成立条件

overrideは「同じ名前のメソッドを書いた」だけでは成立しません。次の条件を同時に満たす必要があります。

条件正しい形誤りやすい形
シグネチャメソッド名と引数リストが同じ引数型が違うとoverload
戻り値型同じ、または戻り値が参照型ならサブタイプ親より広い戻り値型は不可
アクセス同じか、より広いpublic を protected に狭めるのは不可
チェック例外同じか、より狭いより広いチェック例外を追加不可
staticstatic同士なら隠蔽staticとインスタンスの相互変更は不可
 
class A {
    protected Number test() throws java.io.IOException {
        return 1;
    }
}

class B extends A {
    @Override
    public Integer test() throws java.io.FileNotFoundException {
        return 1;
    }
}

上の例はOKです。アクセスは protected から public に広がり、戻り値は Number から Integer に具体化され、チェック例外も狭くなっています。

overloadとoverrideを混同しない

引数が違えばoverloadです。override問題で、選択肢の引数型が少し変わっているだけなら、それはoverrideではありません。

class A {
    void test(int x) {}
}

class B extends A {
    void test(long x) {} // overrideではなくoverload
}

@Override を付けた場合、overrideになっていなければコンパイルエラーです。問題では @Override の有無が大きなヒントです。

interface型で受ける意味

interface型の変数で受けると、そのinterfaceに宣言されたメソッドだけを呼べます。実体クラスに独自メソッドがあっても、宣言型に存在しなければ呼べません。

interface A {
    void test();
}

class B implements A {
    public void test() {}
    void onlyB() {}
}

A a = new B();
a.test();
// a.onlyB(); // NG: A型にはonlyBがない

これはポリモーフィズムの基本です。「実体にあるから呼べる」ではありません。呼べるかは宣言型、実行される中身は実体です。

フィールドはoverrideされない

メソッドはoverrideされますが、フィールドはoverrideされません。同名フィールドを子クラスに定義すると、隠蔽に近い形になります。

class A {
    String name = "A";
    String getName() { return name; }
}

class B extends A {
    String name = "B";
    String getName() { return name; }
}

A obj = new B();
System.out.println(obj.name);      // A: フィールドは宣言型
System.out.println(obj.getName()); // B: メソッドは実体

Javaの出力問題では、ここが頻出です。フィールドアクセスとメソッド呼び出しを同じルールで追うと誤答します。

式まず見るもの実際の解決

obj.method()宣言型にそのメソッドがあるかoverrideされていれば実体のメソッド
