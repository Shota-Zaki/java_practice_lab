Java基礎 Lesson 1

コンピュータと機械語

Javaを書く前に、コンピュータが実際には何を実行しているのかを掴む。

このレッスンのゴール

暗記ではなく、コードを読んだときに「なぜそうなるか」を説明できる状態にする。

機械語を難しく考えすぎない

機械語は、足し算する、値を比較する、メモリから値を読む、条件によって次に進む場所を変える、という単純な命令の集合だと考えればよい。Javaのif、for、メソッド呼び出しも、最終的にはこうした単純な処理の組み合わせとして実行される。

人間が見るもの機械側で起きること学習上の意味
int a = 10;値を入れる領域を用意し、10を保持する変数は「名前付きの場所」として考える
if (a > 5)値を比較し、条件により進む先を変える分岐は「条件判定＋ジャンプ」と見る
method()別の処理単位へ移動し、終わったら戻るメソッドは処理のまとまり

Javaの場合の実行イメージ

人間が書くソースコード

public class Main {
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}

このコードは、いきなりCPUで実行されるのではない。まずjavac Main.javaでMain.classのようなバイトコードへ変換される。その後、java MainでJVMがクラスを読み込み、mainメソッドから処理を開始する。

  1. .javaを書く。
  2. javacが文法や型を検査し、.classを作る。
  3. javaコマンドがJVMを起動する。
  4. JVMがmainを探して実行する。

悪い理解と修正

誤解なぜ違うか正しい見方
CPUがJavaの文法を直接読むCPUはpublicやclassをJava文法として理解しないJavaコードはJVMが実行できる形へ変換される
コンパイルできれば必ず正しく動くコンパイルは主に文法・型の検査であり、実行中の値までは保証しない実行時例外は実行して初めて起きる
エラーは全部同じコンパイルエラーと実行時例外は発生タイミングが違うまず「いつ失敗したか」を見る

手を動かすミニ課題

次のコードについて、CPUが直接読んでいるもの、JVMが実行するもの、人間が書くものを分けて説明してみる。

int x = 3;
int y = x + 2;
System.out.println(y);
  • 人間が書くのはこのソースコード。
  • javacがバイトコードに変換する。
  • 実行時にxとyの値がメモリ上で扱われる。

丁寧な補講：理解の芯

Javaを学び始めた段階で一番危ないのは、「自分が書いた文字列をコンピュータがそのまま理解している」と考えること。実際には、人間向けのソースコードを、実行環境が扱える形へ段階的に変換している。

機械語は怖いものではなく、CPUにとっての最小単位の命令だと考えればよい。Javaのifやforも、最終的には値を読む、比べる、移動する、結果を書き戻す、という単純な処理へ分解される。

このレッスンの狙いは、機械語を詳しく覚えることではない。「ソースコード」「コンパイル後の形」「実行中の動き」を分けて考える癖を作ること。ここができると、コンパイルエラーと実行時例外の区別がかなり楽になる。

ソースコードは人間向け

publicやclassはJavaの文法であって、CPUの命令そのものではない。

実行には変換が必要

javacで検査・変換され、JVMが実行できるバイトコードになる。

値は実行時に動く

変数の中身、条件分岐、繰り返し回数は、実行中の状態として追う。

初心者が混同しやすい境界線

混同しやすいもの違い見分け方
ソースコード / 機械語ソースコードは人間が読む文章。機械語はCPUが実行する命令。.javaを直接CPUが読むわけではない。
コンパイル時 / 実行時コンパイル時は文法・型の検査。実行時は実際の値を使って処理が進む。「いつ失敗したか」を先に見る。
文法上正しい / 意図通り動く文法上正しくても、ロジックが間違っていれば意図通りにはならない。出力結果や変数の変化を追う。

コードを読むときの手順

  1. まず.javaに書かれた人間向けコードだと見る。
  2. 次に、javacで弾かれる文法・型の問題がないかを見る。
  3. コンパイル後に初めて、実行時の値の変化を追う。
  4. 出力結果を考えるときは、CPUではなくJVMがバイトコードを実行していると捉える。

確認用コード

int a = 3;
int b = a + 2;
System.out.println(b);
確認する点このコードでの判断結論
文法int変数に整数を代入しているコンパイル可能
実行時の値aは3、bは5出力は5
失敗の種類文法エラーでも例外でもない正常終了

よくある誤解を潰す

「コードを書いた＝実行された」は違う

保存しただけでは何も実行されない。コンパイルと実行の操作が別に必要。

コンパイル成功を過信しない

コンパイル成功は「Javaとして読める」という意味であり、「期待通りの結果」を保証しない。

確認問題：答えまで言語化する

次の説明は正しいか。「JavaのSystem.out.printlnはCPUが直接理解して画面へ表示している」。

考え方と答えを見る

誤り。CPUがJava文法を直接理解しているわけではない。

人間が書いたJavaコードはバイトコードへ変換され、JVMが実行する。最終的にはOSやCPUの機能を使うが、Java文法をCPUがそのまま読んでいるわけではない。

理解の流れを整える補強

このLessonは、Java文法そのものよりも前に、プログラムは最終的にコンピュータが実行できる形へ変換されるという前提を作るページです。ここを飛ばすと、後で「書いたコード」「コンパイルされたコード」「実行中の値」が全部同じものに見えてしまいます。

段階見るものここで起きること
人間が書くMain.javaJavaの文法で処理を書く。CPUが直接読むものではない。
変換するMain.classjavacが文法・型を検査し、バイトコードを作る。
実行するJVM上の処理JVMがバイトコードを読み、必要に応じてOSやCPUの機能を使う。

重要なのは、機械語やCPUの細部を暗記することではありません。Javaコードを読むときに「今考えているのはコンパイル前か、実行中か」を分けることです。この分離ができると、エラー原因の切り分けが速くなります。

int x = 10;
int y = x + 5;
System.out.println(y);

この3行を見るときは、まずJava文法として成立するかを確認し、次に実行時にxとyがどう変化するかを追います。いきなり「画面に何が出るか」だけを考えると、途中の判断を飛ばす癖がつきます。

ここで切るべき誤解

「Javaを覚える＝命令を丸暗記する」ではありません。実際には、書いたコードがどの段階で検査され、どの段階で値が動くのかを分けて読めることが基礎力です。

理解チェック

  • コンピュータと機械語を、用語だけでなく自分の言葉で説明できる。
  • コードを見たとき、コンパイル前に決まることと実行中に決まることを分けられる。
  • 悪い例がなぜ失敗するのかを、文法・型・参照・実行順序のどれかで説明できる。
