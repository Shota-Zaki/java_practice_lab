Java基礎 Lesson 20

コードを読む順番

Javaコードを上から眺めるのではなく、決まった手順で読む。

このレッスンのゴール

暗記ではなく、コードを読んだときに「なぜそうなるか」を説明できる状態にする。

初出用語：評価順・制御フロー・スコープ

コードは上から眺めるだけでは読めません。どの順に評価され、どの範囲で名前が有効かを追います。

評価順とは式やメソッド呼び出しが処理される順番。
制御フローとはif、for、return、例外などで処理が進む道筋。
スコープとは変数や名前が有効な範囲。
ライフタイムとは変数やオブジェクトが使える期間。
副作用とは値を返すだけでなく、変数変更や出力など状態を変えること。

このLessonで先に区別する言葉

似た言葉を同じ意味で読まないこと。ここを分けると、後のコード読解で迷いにくくなります。

用語切り分け
評価順 / 制御フロー評価順は式の中で何が先に処理されるか。制御フローはifやloopでどこへ進むか。
スコープ / ライフタイムスコープは名前を使える範囲。ライフタイムは値やオブジェクトが存在しうる期間。
宣言型 / 実行時の実体コンパイル時に見える型と、実行時にnewされている実体を分ける。

基礎用語一覧で確認する

読解の基本手順

  1. publicクラス名・ファイル名・mainの形を見る。
  2. 宣言されている変数の型を見る。
  3. コンパイルエラーになる箇所がないか確認する。
  4. newで作られるインスタンスや配列を図にする。
  5. 値が変わる行だけを追う。
  6. メソッド呼び出しでどこへ移動するか確認する。
  7. 最後に出力順を確認する。

小さい例で読む

class Item {
    int price;
}

public class Main {
    public static void main(String[] args) {
        Item a = new Item();
        Item b = a;
        b.price = 100;
        System.out.println(a.price);
    }
}

読み方は、aが新しいItemを指す、b = aで同じItemを指す、b.priceを書き換える、同じ実体をaから読んで100が出る、という順。

値の変化だけを表にする

行abオブジェクトのprice
Item a = new Item();Item#1-0
Item b = a;Item#1Item#10
b.price = 100;Item#1Item#1100
println(a.price)Item#1Item#1100を読む

読解でやってはいけないこと

  • いきなり出力選択肢を見る。
  • コンパイルエラー確認を飛ばす。
  • 宣言型と実体型を混同する。
  • Stringや配列の参照を図にせず頭だけで追う。
  • ++や+=の副作用を読み飛ばす。

手を動かすミニ課題

次のコードを、コンパイル可否→型→参照→値→出力の順で読む。

StringBuilder sb1 = new StringBuilder("A");
StringBuilder sb2 = sb1;
sb2.append("B");
System.out.println(sb1);

丁寧な補講：理解の芯

Javaコードは上から眺めるだけでは読めない。読む順番を固定する。まずクラス構造、次に実行入口、次に変数の型、最後に実行時の値の変化を追う。

特に試験形式や短いサンプルでは、見た目の順番と実行順序が一致しないことが多い。メソッド定義は上にあっても、呼ばれなければ実行されない。コンストラクタはnewのタイミングで動く。

コード読解の目的は「なんとなく分かる」ではなく、「この行の時点でこの変数はこの値」と説明できる状態にすること。

構造確認

クラス、フィールド、メソッド、コンストラクタを分ける。

入口確認

mainまたは呼び出し元から実行順序を追う。

状態追跡

変数と参照がどう変わるかを表にする。

初心者が混同しやすい境界線

混同しやすいもの違い見分け方
定義順 / 実行順上に書いてあるから先に実行されるとは限らない。mainから呼び出しを追う。
型 / 実体変数の型とnewされた型は違う場合がある。A a = new B()を分解する。
宣言 / 代入宣言だけでは値が入らない場合がある。初期化済みか見る。

コードを読むときの手順

  1. トップレベルクラスとファイル名を見る。
  2. フィールド、コンストラクタ、メソッドを色分けするつもりで分ける。
  3. mainから実行順に番号を振る。
  4. 各変数について、型・値・参照先を表にする。
  5. 分岐・ループ・例外で実行されない行を消す。

確認用コード

class Item {
    int price;
    Item(int price) { this.price = price; }
    void change(int price) { this.price = price; }
}
Item item = new Item(100);
item.change(200);
System.out.println(item.price);
確認する点このコードでの判断結論
new Item(100)コンストラクタ実行price=100
change(200)フィールドを200に更新price=200
出力item.price200

よくある誤解を潰す

上から全部実行される、は違う

メソッド定義やクラス定義は、そこに書いてあるだけで実行されるわけではない。

変数名だけで判断する

同じ名前でもローカル変数とフィールドがある。thisの有無を見る。

確認問題：答えまで言語化する

コードを読むとき、出力を考える前に確認すべき最初の3点を答える。

考え方と答えを見る

1つ目は、コンパイルできる構造か。ファイル名、クラス名、main、型の整合性を見る。

2つ目は、実行入口がどこか。通常はmainから追う。

3つ目は、変数の型と初期値。特にローカル変数、フィールド、参照型を分ける。

理解の流れを整える補強

コード読解で失敗する最大の原因は、上から眺めて雰囲気で答えることです。Javaコードは、必ず順序を決めて読みます。まずコンパイル可否、次に実行開始位置、次に値の変化、最後に出力や例外です。

順番確認内容飛ばすと起きる失敗
1クラス・メソッド・型が成立しているかコンパイル不可なのに出力を考える
2mainからどこへ進むか呼ばれないメソッドの処理を読んでしまう
3変数・参照・条件式の変化古い値のまま計算する
4出力、return、例外、finally途中で制御が移る箇所を見落とす
int x = 1;
int y = x++ + ++x;
System.out.println(x + "," + y);

このようなコードを見たとき、いきなり足し算しないでください。x++で「式には元の値を使い、その後増える」、++xで「先に増えて、その値を使う」と順番に処理します。表を書けばミスは減ります。

ここで切るべき誤解

読解力はセンスではありません。順番を固定して、値の変化を表に落とすだけでかなり改善します。雑に暗算する癖は捨ててください。

理解チェック

  • コードを読む順番を、用語だけでなく自分の言葉で説明できる。
  • コードを見たとき、コンパイル前に決まることと実行中に決まることを分けられる。
  • 悪い例がなぜ失敗するのかを、文法・型・参照・実行順序のどれかで説明できる。

このLessonのつまずきチェック

用語を覚えただけでは不十分です。実際に間違えやすい読み方を先に潰します。

コード読解で順番を間違える点

つまずき正しい切り分け
上から全部読もうとする試験ではまずコンパイル可否、次に実行時例外、最後に出力を判定する。
選択肢から逆算しすぎる選択肢を見る前に、コンパイル不可の行・例外発生箇所・出力候補を切る。
状態変化を書かない変数、参照、配列、フィールド値は表にしないと高確率で読み違える。

説明できる状態の目安

  • コンパイル可否→実行可否→出力の順で説明できる
  • 変数表を作るべき問題を判断できる
  • 参照の変化を図で説明できる
試験対策としての使い方

この3点を自分の言葉で説明できない場合、問題演習で似た形を見たときに運任せになります。戻ってコードを1つ実行してください。

正しいコード / 失敗コード / 修正版

このLessonの要点を、成功例と失敗例で切り分けます。試験では「なぜ失敗するか」を説明できる状態が必要です。

正しいコード：コード読解の順番

int a = 1;
int b = ++a + a;
System.out.println(b);

失敗コード

出力だけを先に予想する

修正版

1. コンパイル可否
2. 実行時例外
3. 評価順
4. 出力
判定ポイント

Java問題は出力から考えず、コンパイル可否を最初に判定します。

理解確認クイズ

配列を別変数へ代入すると、配列本体はコピーされますか。

答えと切り分けを見る

されません。参照がコピーされるだけなので、同じ配列を指します。
