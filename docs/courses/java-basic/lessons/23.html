Java基礎 Lesson 23

メモリ・スタック・ヒープ

ローカル変数、参照、newされたオブジェクトの位置関係をイメージする。

このレッスンのゴール

暗記ではなく、コードを読んだときに「なぜそうなるか」を説明できる状態にする。

先に結論

ローカル変数やメソッド呼び出しの管理はスタック、newされたオブジェクトはヒープに置かれる、と考えると参照の動きが理解しやすい。厳密なJVM内部を覚えるより、まず「変数」と「オブジェクト」を分ける。

スタックとヒープの役割

領域何を置くと考えるか学習上の使い道
スタックメソッド呼び出し、ローカル変数など今どのメソッドを実行中か、ローカル変数が何を持つかを見る
ヒープnewされたオブジェクトや配列参照型変数が何を指しているかを見る

参照型の図解を言葉で書く

Item a = new Item();
Item b = a;

このとき、ヒープ上にItemオブジェクトが1つ作られる。スタック上のaとbは、その同じオブジェクトを指す参照を持つ。

b = new Item();

ここでbだけが新しいItemを指すようになる。aが指す先は変わらない。

プリミティブ型は値のコピー

int a = 10;
int b = a;
b = 20;
System.out.println(a); // 10

intでは値がコピーされる。bを変えてもaは変わらない。参照型の「同じオブジェクトを見ている」動きとは違う。

ガベージコレクションの入口

どこからも参照されなくなったオブジェクトは、将来的にガベージコレクションの対象になる。いつ回収されるかをプログラマが正確に決めるわけではない。

Object a = new Object();
a = null; // さきほどのObjectは参照されなくなる

手を動かすミニ課題

次のコードで、最後に参照されていないオブジェクトがどれかを考える。

Object a = new Object(); // Object#1
Object b = new Object(); // Object#2
Object c = a;
a = null;
b = c;

丁寧な補講：理解の芯

スタックとヒープは、Javaの値と参照を理解するためのイメージとして使う。厳密なJVM実装を覚える必要はない。基礎では、ローカル変数はメソッド呼び出し中の作業領域、newされたオブジェクトはヒープ側、と捉えればよい。

参照型変数には、ヒープ上のオブジェクトへたどるための参照が入る。だから、参照をコピーすると同じオブジェクトを複数の変数から扱える。

どこからも参照されなくなったオブジェクトは、将来的にガベージコレクションの対象になる。ただし、いつ回収されるかをプログラム側が正確に決めるものではない。

スタック

メソッド呼び出し中のローカル変数などを扱う領域として理解する。

ヒープ

newされたオブジェクトが置かれる領域として理解する。

GC

参照されなくなったオブジェクトを回収する仕組み。

初心者が混同しやすい境界線

混同しやすいもの違い見分け方
変数 / オブジェクト参照型変数は参照を持つ。オブジェクト本体は別にある。newされたものと変数を分ける。
参照あり / 参照なしどこかの変数などから到達できるかどうか。代入やnullで参照が外れたか見る。
GC対象 / 即回収対象になっても即座に回収されるとは限らない。「対象」と「実行タイミング」を分ける。

コードを読むときの手順

  1. newが何回実行されたか数える。
  2. 各変数がどのオブジェクトを参照しているか図にする。
  3. 代入で参照がコピーされたか、上書きされたかを見る。
  4. null代入で参照が外れたか確認する。
  5. どこからも参照されないオブジェクトをGC対象と判断する。

確認用コード

Object a = new Object();
Object b = new Object();
Object c = a;
a = null;
b = null;
確認する点このコードでの判断結論
1つ目のObjectcがまだ参照しているGC対象ではない
2つ目のObjectbがnullになり参照なしGC対象
結論対象は2つ目だけただし回収タイミングは未定

よくある誤解を潰す

nullにすると必ずすぐメモリ解放、は違う

GC対象になるだけ。実際の回収タイミングはJVMが決める。

参照をコピーするとオブジェクトも増える、は違う

増えるのは参照を持つ変数。オブジェクト本体は増えない。

確認問題：答えまで言語化する

次のコードで、最後にGC対象になる可能性があるのはどちらのオブジェクトか。

確認問題

Object x = new Object(); // 1つ目
Object y = new Object(); // 2つ目
x = y;
y = null;
考え方と答えを見る

1つ目のオブジェクトがGC対象になる。

x = yでxは2つ目のオブジェクトを参照するようになる。この時点で1つ目への参照がなくなる。y = null後もxが2つ目を参照しているため、2つ目は対象ではない。

理解の流れを整える補強

メモリの話は、細かい内部実装を覚えるためではなく、変数とオブジェクトの位置関係を見失わないために使います。特に参照型では、変数の中にオブジェクトそのものが入るわけではありません。

領域置かれるもののイメージ読解での使い方
スタックメソッド呼び出しごとのローカル変数今どのメソッドの変数を見ているかを追う
ヒープnewされたオブジェクト複数の変数が同じオブジェクトを参照する可能性を見る
参照オブジェクトへのリンク情報代入で参照がコピーされることを確認する
int[] a = {1, 2, 3};
int[] b = a;
b[0] = 9;
System.out.println(a[0]); // 9

b = aで配列が複製されたわけではありません。aとbが同じ配列オブジェクトを参照するようになっただけです。だからb[0]を書き換えると、a[0]を見ても同じ変更が見えます。

ここで切るべき誤解

参照型の代入を「オブジェクトのコピー」と考えるのは危険です。コピーされるのは参照であり、参照先が同じなら変更の影響も共有されます。

理解チェック

  • メモリ・スタック・ヒープを、用語だけでなく自分の言葉で説明できる。
  • コードを見たとき、コンパイル前に決まることと実行中に決まることを分けられる。
  • 悪い例がなぜ失敗するのかを、文法・型・参照・実行順序のどれかで説明できる。
